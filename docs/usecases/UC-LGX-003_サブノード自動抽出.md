# UC-LGX-003: サブノード自動抽出

## 概要

legixy が graph.toml 読み込み時に、Markdown ドキュメントの h2/h3 見出しからサブノードを自動生成する。

## アクター

- システム（graph.toml 読み込み時に自動実行）

## 事前条件

- graph.toml にドキュメントノードが定義されている
- 対象ドキュメントが Markdown 形式（.md）である

## 基本フロー

1. システムが graph.toml を読み込む
2. 各ドキュメントノード（.md ファイル）に対して:
   a. ファイル内容を読み込む
   b. `##`（h2）および `###`（h3）の ATX 見出しを抽出する
   c. 各見出しテキストを正規化する（トリム + 連続空白統合）
   d. 見出し階層パスを構築する（h3 は直上の h2 を含む）
   e. ハッシュ対象文字列 `親ID|見出し1|見出し2` を構築する
   f. SHA-256 の先頭 16 文字（hex）でサブノード ID を生成する
   g. サブノードをグラフに追加する（SubnodeKind::AutoGenerated）
   h. 親 → サブノードの ParentChild エッジを自動生成する
3. graph.toml で明示定義されたサブノード（`#s:` 接頭辞）もマージする
4. 全サブノードの ID 一意性を担保する（SUBNODE-INV-3）: 明示サブノードとの衝突は生成スキップ（明示優先）、自動生成同士の同一 heading_path は同一 ID へ縮退する。生成段階ではエラー・Warning を発しない。違反の検出・可視化（SubnodeIdCollision Warning）は check が担う（SPEC-LGX-002.REQ.12 / SPEC-LGX-004.REQ.14）

## 代替フロー

- 2a. ドキュメントファイルが存在しない場合、そのノードのサブノード抽出はスキップする
- 2b. h2/h3 見出しが存在しない場合、サブノードは生成されない
- 3a. 明示サブノードと自動生成サブノードの ID が衝突する場合、明示側を優先する

## 事後条件

- 同じドキュメント内容に対して常に同じサブノード ID が生成される（SUBNODE-INV-5）
- 自動生成 ID（`#hex16`）と明示 ID（`#s:name`）は形式的に区別可能（SUBNODE-INV-6）
- 全サブノードが親ドキュメントノードを持つ（SUBNODE-INV-1）
- サブノードの path が親と一致する（SUBNODE-INV-2）

## 関連不変条件

- SUBNODE-INV-1〜6（全て）

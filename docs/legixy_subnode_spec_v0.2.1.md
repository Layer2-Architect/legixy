## 1. 本文書の位置づけ

本文書は legixy(有向グラフ主体のトレーサビリティエンジン)における機能拡張仕様書である。特に、有向グラフのノード粒度をドキュメント単位からサブノード単位へと細分化することで、LLM が消費するトークン量を削減する機能を定義する。

本文書は LEGIXY-SPEC-001 (legixy_foundational_spec.md) の補完文書であり、Phase 1 開発における設計判断の基礎となる。本文書の内容は、将来的に LEGIXY-SPEC-001 の §2(概要)、§4(エンジン機能)、§10(不変条件)に統合される可能性がある。

本文書のスコープは legixy 単体の仕様である。

### 1.1 読者

本文書の想定読者は以下である。

- legixy 開発者(仕様作成者自身)
- AI エージェント(開発プロセスにおける実装担当)

AI エージェントが本文書を参照しながら legixy の実装を進めることを想定している。

### 1.2 スコープ

本文書がカバーする範囲は以下である。

- サブノード化の基本概念とデータ構造
- compile_context の粒度制御機能
- サブノード ID の生成方式(ハッシュ方式)
- 明示 ID の仕組み
- Contextual Retrieval の実装(デフォルト無効)
- 既存不変条件への影響評価
- Phase 1 における導入スコープ

本文書がカバーしない範囲は以下である。

- マトリクス主体から有向グラフ主体への移行全般(別文書で扱う)
- legixy 単体のスコープ外の運用要件

## 2. 背景と目的

### 2.1 現状の課題

現世代(マトリクス主体)の legixy では、`compile_context` が返す上流成果物はドキュメント単位である。CLAUDE.md における運用ルールでも「`compile_context` が返す Upstream Artifacts セクションのファイルは全て読むこと」と規定されている。

この運用は機能的には正しいが、以下の課題を生む。

**課題 1: トークン消費の肥大化**

成果物ドキュメントの一部分のみが作業に関連する場合でも、ドキュメント全体が Claude Code のコンテキストに取り込まれる。成果物が成長するほど、この問題は深刻化する。

**課題 2: LLM トークン消費コストの増大**

AI 駆動開発では、コンテキストとして LLM に供給するドキュメントのトークン量が増えるほど、API 利用コストと応答遅延が増大する。`compile_context` が返すコンテキストの肥大化は直接的なコスト要因となる。

**課題 3: ドリフト検出粒度の粗さ**

ドキュメント全体に対する embedding と比較した場合、ドキュメント内の特定セクションだけが意味的にドリフトしている状況を検出しにくい。

### 2.2 本拡張の目的

本拡張は以下の目的を達成する。

**目的 1: トークン消費削減**

Claude Code が `compile_context` 経由で取得するコンテキストを、作業に関連するサブノード単位に絞り込む。これにより、同じ作業を遂行するのに必要なトークン量を削減する。

**目的 2: ドリフト検出精度の向上**

サブノード単位での embedding と比較により、ドキュメント内の部分的なドリフトを検出可能にする。

**目的 3: トークン効率の向上**

「AI 駆動開発におけるトークン削減」を legixy の付加的な効果として組み込む。これはドリフト対策という主目的を強化する副次効果として位置づけられる。

### 2.3 本拡張の非目的

以下は本拡張の目的ではない。

- サブノード化の強制(ドキュメント単位のノードとの共存を維持)
- 成果物の自動分割(サブノードは論理的な単位であり、物理ファイル分割を伴わない)
- Markdown 以外のドキュメント形式への対応(Phase 1 では Markdown のみを対象)

## 3. サブノード化の基本概念

### 3.1 サブノードの定義

サブノードとは、一つのドキュメント成果物(DD、UC、TS 等)の内部に存在する、意味的にまとまった一区画を指す有向グラフのノードである。

既存のドキュメント単位のノードと、サブノードは、同じグラフ構造内で共存する。

### 3.2 ノードの種類

有向グラフには以下の2種類のノードが存在する。

| 種類 | 単位 | ID 表記例 | 説明 |
|------|------|----------|------|
| ドキュメントノード | ドキュメント全体 | `DD-SP-003` | 既存のノード概念 |
| サブノード(自動生成) | ドキュメント内の一区画 | `DD-SP-003#a3f7b2c4e91dfa08` | ハッシュ方式で自動生成 |
| サブノード(明示定義) | ドキュメント内の一区画 | `DD-SP-003#s:cross-section-logic` | graph.toml で明示的に記述 |

サブノードは必ず一つのドキュメントノードに属する。

### 3.3 サブノードの生成方法

サブノードの生成方法は以下の二つである。

**方法 A: 見出しからの自動生成**

Markdown ドキュメント内の見出しを解析し、各見出しに対応するサブノードを自動生成する。graph.toml に明示的な記述は不要である。

自動生成されるサブノード ID の形式は `{親ID}#{ハッシュ}` であり、ハッシュは「親ドキュメント ID + 見出し階層パス」から SHA-256 で生成された値の先頭 16 文字(16 進数表記)である。詳細は Section 4.5 に記述する。

**方法 B: graph.toml での明示的定義**

graph.toml にサブノードを明示的に記述する。自動生成では表現できない論理的な区画や、複数の見出しをまたがるサブノードを定義する場合に使用する。

明示的に定義されるサブノードの ID は、自動生成される ID と区別するため `s:` 接頭辞を使用する。形式は `{親ID}#s:{任意文字列}`。使用は例外的な場合(自動生成で表現できない構造的必要がある場合)に限定する。詳細は Section 4.5 に記述する。

両者は共存可能である。同じドキュメントで自動生成されたサブノードと明示的に定義されたサブノードが並立しうる。

### 3.4 サブノード間のエッジ

サブノードも他のノード(ドキュメントノードまたは別のサブノード)へのエッジを持てる。これにより、以下のような細粒度のトレーサビリティが表現できる。

- `SRC-SP-003.rs` → `DD-SP-003#a3f7b2c4e91dfa08`: このコードは DD のある特定セクションに対応する
- `DD-SP-003#e91dfa08d2c5f1b8` → `UC-SP-001#b7c4f3a8d2e91bf0`: この設計は UC の特定フローに対応する

### 3.5 サブノード化の任意性

本拡張において、サブノード化はあくまで**任意の機能**である。以下の運用が全て許容される。

- ドキュメントノードのみ使用(従来の運用を維持)
- 一部のドキュメントのみサブノード化
- 全てのドキュメントをサブノード化

この任意性により、既存の graph.toml は変更なしにlegixy で動作する(後方互換性)。

### 3.6 見出し抽出の対象レベル

自動生成サブノードの対象となる見出しレベルは、Phase 1 では h2 と h3 に固定する。H4 以上は対象としない。

H4 以上の見出しが必要になる場合は、サブノード化ではなく、ドキュメント自体の分割で対応することが健全である。これは本拡張の動機(AI コーディングエージェントのコンテキストウィンドウに収めるための粒度制御)に立ち返れば、ドキュメント自体が肥大化している徴候であり、有向グラフの構造表現(独立ドキュメントとそのエッジ)のほうが意味的に明確な関係表現となるためである。

将来的なドキュメントごとの対象レベル指定は、graph.toml のドキュメントノード定義にオプショナルフィールドを予約することで設計余地を確保する(Phase 1 ではこのフィールドは使用せず、常に h2 と h3 を対象とする)。

## 4. データ構造の変更

### 4.1 graph.toml のスキーマ拡張

現状の graph.toml スキーマに、サブノード表現のための拡張を加える。

**基本形(既存と同じ)**

```toml
[[nodes]]
id = "DD-SP-003"
type = "DD"
path = "docs/detailed-design/DD-SP-003.md"
```

**自動生成サブノード(graph.toml に明示記述なし)**

Markdown ドキュメント内の h2 および h3 見出しから、legixy が自動でサブノードを生成する。graph.toml に記述する必要はない。

自動生成サブノードの ID 例:
```
DD-SP-003#a3f7b2c4e91dfa08
```

**明示サブノード定義**

```toml
[[nodes]]
id = "DD-SP-003#s:cross-section-logic"
type = "DD-section"
parent = "DD-SP-003"
path = "docs/detailed-design/DD-SP-003.md"
anchor = "## 状態遷移"
```

**フィールドの意味**

| フィールド | 必須 | 説明 |
|-----------|------|------|
| `id` | 必須 | ノード ID。明示サブノードの場合 `{親ID}#s:{任意文字列}` 形式 |
| `type` | 必須 | ノードタイプ。明示サブノードは `{親type}-section` |
| `parent` | 明示サブノードのみ必須 | 親ドキュメントノードの ID |
| `path` | 必須 | ファイルパス。サブノードは親と同じパス |
| `anchor` | 明示サブノードのみ推奨 | ドキュメント内の位置を特定する文字列(見出しテキスト) |

**ドキュメントノードの拡張フィールド(将来用、Phase 1 では未使用)**

```toml
[[nodes]]
id = "DD-SP-003"
type = "DD"
path = "docs/detailed-design/DD-SP-003.md"
# 以下は将来の拡張用フィールド。Phase 1 では値を記述しても無視される
# subnode_heading_levels = [2, 3]  # 対象とする見出しレベル
```

### 4.2 エッジの記述

エッジの記述方法は既存と同じである。`from` と `to` がサブノード ID(自動生成または明示)を指すことができる点のみが拡張である。

```toml
[[edges]]
from = "SRC-SP-003"
to = "DD-SP-003#a3f7b2c4e91dfa08"
kind = "chain"
```

### 4.3 engine.db のスキーマ変更

`.legixy/engine.db` は以下のテーブル変更を受ける。

**nodes テーブル**

既存のカラムに加えて、以下を追加する。

| カラム | 型 | 説明 |
|--------|-----|------|
| `parent_id` | TEXT NULL | 親ノード ID(サブノードの場合のみ) |
| `anchor` | TEXT NULL | ドキュメント内アンカー |
| `is_subnode` | INTEGER | 0=ドキュメントノード, 1=自動生成サブノード, 2=明示サブノード |

**embeddings テーブル**

既存の構造に加えて、Contextual Retrieval のための拡張を加える。

| カラム | 型 | 説明 |
|--------|-----|------|
| `context` | TEXT NULL | Contextual Retrieval で生成された文脈情報。Phase 1 デフォルトでは NULL |
| `context_hash` | TEXT NULL | context の内容ハッシュ |

ノード ID がサブノード ID でも格納可能とする。

**scores テーブル**

既存の構造のまま、エッジの両端がサブノード ID でも格納可能とする。

### 4.4 後方互換性

既存の graph.toml ファイル(サブノード定義を含まない)は、legixy でも変更なしに動作する。legixy は、サブノード定義が存在しない場合、従来通りドキュメントノードのみを扱う。

engine.db のマイグレーションは、既存の engine.db をlegixy が初回読み込み時に自動実行する。

### 4.5 サブノード ID の生成規則

#### 4.5.1 自動生成 ID

自動生成サブノードの ID は以下の規則で生成される。

**形式**: `{親ID}#{ハッシュ}`

**ハッシュの生成**:

1. ハッシュ対象文字列を以下の通り構築する
   - 親ドキュメント ID + 区切り文字 + 見出し階層パス
   - 例: `DD-SP-003|認証機能|状態遷移`
2. 見出しテキストは以下の正規化を適用する
   - 前後の空白を削除
   - 連続する空白を 1 つに統合
3. ハッシュ対象文字列を UTF-8 エンコードし、SHA-256 でハッシュ化する
4. ハッシュ値を 16 進数表記し、先頭 16 文字を ID のハッシュ部分として使用する

**見出し階層パス**:

自動抽出対象の見出し(h2 および h3)について、上位見出しから対象見出しまでをパイプ区切りで連結したもの。h3 の場合は直上の h2 を含む。

**例**:

```markdown
## 認証機能
### 状態遷移
```

このとき、h3 見出し「状態遷移」から生成されるハッシュ対象文字列は `DD-SP-003|認証機能|状態遷移` となる。

**連鎖変化**:

見出し階層の上位のリネームは、配下のサブノード ID を連鎖的に変化させる正しい挙動として扱う。

#### 4.5.2 明示 ID

明示サブノードの ID は以下の規則で記述される。

**形式**: `{親ID}#s:{任意文字列}`

**文字列部分の制約**:

- 英数字(大文字小文字)、ハイフン、アンダースコアのみ
- 英数字で始まり、英数字で終わる
- 1 文字以上 63 文字以下

**使用指針**:

明示 ID は自動生成で表現できない構造的必要がある場合(複数の見出しをまたがるサブノード、見出しに対応しない論理区画など)に限定して使用する。

#### 4.5.3 自動生成 ID と明示 ID の区別

両者は接頭辞の有無により形式的に区別される。

- 自動生成: `#{16 進数 16 文字}`(0-9、a-f のみ)
- 明示: `#s:{任意文字列}`

物理的に衝突しない設計である。

### 4.6 見出しテキストの正規化

自動生成サブノード ID のハッシュ対象に使用する見出しテキストは、以下の正規化を適用する。

- 前後の空白の削除(トリム)
- 連続する空白の 1 つへの統合

これ以上の正規化(Markdown 装飾の除去、NFC/NFKC 正規化、大文字小文字変換など)は Phase 1 では実施しない。AI エージェントがドキュメント作成の主体であることを前提とし、複雑な表記揺れは発生しないものとする。

## 5. 機能の変更と追加

### 5.1 compile_context の拡張

`compile_context` に粒度制御のためのオプション引数を追加する。新しい CLI ツールは追加しない(MCP-INV-1 維持のため)。

**Phase 1 で実装する引数**:

```bash
# 従来の呼び出し(引数追加後も動作)
legixy compile_context src/auth.rs

# 粒度を指定した呼び出し
legixy compile_context --granularity subnode src/auth.rs
```

**引数の意味**

| 引数 | 値 | デフォルト | 説明 |
|------|-----|-----------|------|
| `--granularity` | `document`, `subnode` | `document` | 返却の粒度 |

**デフォルト値**

デフォルトの `--granularity` は `document` である。これは後方互換性を最優先した判断である。

**Phase 1 で実装しない引数**

以下の引数は Phase 2 以降で実装する。

- `--outline-only`: 見出し構造のみ返却
- `--sections`: 特定のサブノードのみ取得
- `--depth`: 上流何階層まで辿るか

### 5.2 返却データ構造の拡張

`compile_context` の返却構造に、サブノード情報を含める。

**`--granularity document` の返却構造**

従来通り、Upstream Artifacts、Layer Guidelines、Additional Guidelines を含む。ファイル単位で返却される。

**`--granularity subnode` の返却構造**

Upstream Artifacts はサブノード単位で返却される。各エントリは以下を含む。

- サブノード ID(自動生成または明示)
- 親ドキュメント ID
- アンカー(見出しテキストまたは anchor フィールド値)
- 本文(該当セクションのみ)
- ドリフトスコア(エッジごと)

### 5.3 見出し自動抽出ロジック

Markdown ドキュメントから見出しを抽出し、暗黙的なサブノードとして扱うロジックを実装する。

**抽出ルール**

- 対象とする見出しレベル: `##`(h2)および `###`(h3)
- サブノード ID の生成: Section 4.5.1 の規則に従う
- 見出しテキストの正規化: Section 4.6 の規則に従う

**例**

```markdown
# DD-SP-003 認証機能の詳細設計

## 認証機能

### 状態遷移

...

### エラーハンドリング

...
```

このドキュメントから、h3 見出し「状態遷移」と「エラーハンドリング」について、それぞれハッシュ対象文字列 `DD-SP-003|認証機能|状態遷移` および `DD-SP-003|認証機能|エラーハンドリング` から SHA-256 短縮 16 文字のハッシュが生成される。

### 5.4 semantic_check のサブノード対応

`semantic_check` は、エッジ単位でのドリフトスコア計算を行う。サブノードが関与するエッジについても、同様にドリフトスコアを計算する。

ドリフトスコアの計算対象となるエッジは、graph.toml または自動抽出から導出される全てのエッジである。

### 5.5 investigate と impact のサブノード対応

`investigate`(逆方向探索)および `impact`(順方向探索)は、サブノードを含むグラフで動作する。探索はサブノードを経由して進む。

探索結果にサブノードが含まれる場合、呼び出し側は従来のドキュメントノードと同様に扱える形式で返却を受ける。

### 5.6 check のサブノード対応

`check --formal`(第1層検証)は以下を検証する。

- サブノード ID の形式が正しいこと
- サブノード ID がグラフ内で一意であること
- サブノードの `parent` フィールドが既存のドキュメントノードを指していること
- サブノードの `path` が親ドキュメントと一致していること
- サブノードを含むグラフに循環がないこと(DAG 制約)
- 明示 ID が `s:` 接頭辞を持ち、文字列部分が規定の文字制約を満たすこと

### 5.7 embedding 生成ロジックの抽象化

embedding 生成を「入力テキスト → embedding ベクトル」というインターフェースで抽象化する。前処理(Contextual Retrieval の context 注入)を差し替え可能な構造とする。

**抽象化の内容**

- embedding 生成関数を、前処理ステップとベクトル生成ステップに分離する
- 前処理ステップは、入力テキストを変換する関数として差し替え可能にする
- デフォルトの前処理は「何もしない」(パススルー)

この抽象化により、Phase 1 で Contextual Retrieval を実装しつつデフォルトで無効化することが可能になる。

### 5.8 Contextual Retrieval の実装

Contextual Retrieval 機能を実装する。ただし Phase 1 では**デフォルトで無効**とする。

**機能概要**

サブノードの embedding を計算する際、そのサブノードが親ドキュメント全体の中でどのように位置づけられるかを示す文脈情報(context)を LLM で生成し、embedding 対象テキストの前に付加する。

**実装範囲(Phase 1)**

- Claude API を呼び出して context を生成する関数
- context のキャッシュ機構(engine.db の embeddings テーブル拡張)
- ハッシュ計算対象の更新(context 込みでハッシュする経路)
- フラグによる有効化/無効化(graph.toml または設定ファイルで制御)

**Phase 1 で作り込まない範囲**

- prompt caching の最適化
- エラー処理の作り込み(リトライ、フォールバック)
- context 生成失敗時のグレースフル・デグラデーション

これらは Phase 2 以降、運用を有効化するタイミングで作り込む。

**デフォルト挙動**

フラグが無効(デフォルト)の場合、通常の embedding 生成ロジックで動作する。Contextual Retrieval 関連のコードは呼び出されない。

## 6. MCP-INV-1 との整合

### 6.1 MCP-INV-1 の内容

現行の LEGIXY-SPEC-001 §10.4 にて、以下の不変条件が定義されている。

> MCP-INV-1: Agent Surface 限定 - MCP は compile_context, observe, get_compile_audit の3ツールのみ

本拡張はこの不変条件を**維持する**。

### 6.2 新機能の提供方式

サブノード化に関連する新機能は、以下の方針で提供される。

**新 MCP ツールを追加しない**

`outline_only` や `expand_section` といった候補機能があったが、これらは独立した MCP ツールとして追加しない。将来 Phase 2 以降で実装される場合も、`compile_context` の引数オプションとして統合する。

**既存ツールの引数拡張として実装する**

これらの機能は、`compile_context` の引数オプションとして実装される。

| 機能 | 実装方式 | Phase |
|------|---------|-------|
| 粒度指定 | `compile_context --granularity <level>` | Phase 1 |
| 見出し構造のみ返却 | `compile_context --outline-only` | Phase 2 以降 |
| 特定サブノード取得 | `compile_context --sections <ids>` | Phase 2 以降 |

この方式により、MCP サーバーが提供するツールは従来通り 3 ツールに維持される。

### 6.3 MCP サーバー側の対応

MCP サーバー(TypeScriptMCP)は、`compile_context` の呼び出し時に新しい引数オプションを Claude Code から受け取り、Rust CLI に渡す。MCP サーバー側の実装変更は、引数の転送ロジックの拡張のみである。

### 6.4 MCP-INV-2 との整合

MCP-INV-2(忠実な転送: Rust CLI 出力のフィルタリング・省略なし)も維持される。サブノード単位の返却も、Rust CLI の出力をそのまま MCP サーバーが転送する。

## 7. 不変条件への影響と追加

### 7.1 既存不変条件への影響

LEGIXY-SPEC-001 §10 で定義された既存の不変条件について、サブノード化の影響を評価する。

**CTX-INV-1: 決定論保証**

影響: あり。サブノード単位でも、同じ入力に対して同じ結果を返すことを保証する必要がある。

対応: `compile_context` の全ての引数に対して決定論性を検証する。Contextual Retrieval が有効な場合、LLM 出力の決定論性を context のキャッシュで担保する。

**CTX-INV-2: グラフ整合性**

影響: あり。サブノードとその親ドキュメントノードの関係、サブノード間のエッジの整合性を検証する必要がある。

対応: サブノード特有の整合性検証ルールを追加する(Section 7.2 参照)。

**CTX-INV-3: カスタムエッジ独立性**

影響: 小。既存のカスタムエッジの仕組みは維持される。サブノードを指すカスタムエッジも同様に扱える。

対応: 特になし。

**CTX-INV-4: DAG 制約**

影響: あり。サブノードを含むグラフもサイクルを含まないことを保証する必要がある。

対応: `check --formal` でサブノード含めたサイクル検出を実施する。

**FB-INV-1 ~ FB-INV-5: フィードバックループ関連**

影響: 小。サブノード起点の Observation、サブノード対象の Proposal も生成可能とするが、基本的な仕組みは維持される。

対応: related_ids にサブノード ID が含まれうる点を考慮した実装とする。

**SCORE-INV-1, SCORE-INV-2: スコア管理**

影響: あり。サブノード単位のハッシュ管理とスコアのひもづけが必要。

対応: engine.db の scores テーブルで、サブノード ID をキーに含める。Contextual Retrieval が有効な場合、context を含めたハッシュでキャッシュ有効性を判定する。

**MCP-INV-1 ~ MCP-INV-4: MCP サーバー関連**

影響: 小。Section 6 で整理した通り、MCP-INV-1 は維持される。その他の不変条件も影響を受けない。

対応: 特になし。

**STATE-INV-1, STATE-INV-2: 状態管理**

影響: なし。サブノード化は legixy 内の話であり、engine.db はキャッシュ扱いのままである。Linear の状態管理や Git 管理には影響しない。

対応: 特になし。

### 7.2 追加すべき不変条件

サブノード化に伴い、以下の不変条件を追加する。

**SUBNODE-INV-1: サブノードの親子整合性**

サブノードは必ず存在するドキュメントノードを親として持つ。親が存在しないサブノードは不正とする。

**SUBNODE-INV-2: サブノードのパス整合性**

サブノードの `path` フィールドは、親ドキュメントノードの `path` と一致する。

**SUBNODE-INV-3: サブノード ID の一意性**

グラフ内で同じ ID を持つサブノードが存在しないことを `check --formal` で検証する。

**SUBNODE-INV-4: サブノードの DAG 制約**

サブノードを含めた有向グラフ全体にサイクルが存在しない。これは既存の CTX-INV-4 のサブノードへの拡張である。

**SUBNODE-INV-5: 自動生成サブノードの決定論性**

サブノード ID は、親ドキュメント ID と見出し階層パスのハッシュにより決定論的に生成される。これらのハッシュ対象が変化しない限り、サブノード ID は変化しない。見出し階層の上位のリネームは、配下のサブノード ID を連鎖的に変化させる正しい挙動として扱う。

**SUBNODE-INV-6: 明示 ID と自動生成 ID の形式的区別**

明示 ID(`s:` 接頭辞を持つ)と自動生成 ID(16 進数のハッシュ)は、形式的に区別可能であり、両者の間で衝突は発生しない。

### 7.3 設計健全性の原則

本拡張の議論で言及された「コア機能は BTS 非依存」の原則は、legixy 全体に関わるため、親文書 LEGIXY-SPEC-001 §9(設計原則)の範囲とし、本文書では記述しない。

## 8. 段階的導入の方針

### 8.1 Phase 1 のスコープ

legixy の Phase 1 におけるサブノード化の導入スコープを以下とする。

**Phase 1 必須スコープ(legixy の実装範囲)**

1. graph.toml のスキーマ拡張(サブノード記述、明示 ID の受け入れ)
2. 後方互換性の維持(既存 graph.toml の動作保証)
3. 見出し自動抽出の基本実装(h2 および h3 対応、対象レベル指定のオプショナルフィールド予約)
4. ハッシュ方式のサブノード ID 生成(親ID + 見出し階層パスの SHA-256 短縮 16 文字)
5. 見出しテキストの正規化(トリム + 連続空白統合)
6. 明示 ID の仕組み(`s:` 接頭辞方式、graph.toml でのみ定義可能)
7. compile_context の `--granularity` 引数対応
8. `check --formal` の拡張(サブノード ID 一意性、形式検証、DAG 制約)
9. SUBNODE-INV-1〜6 の実装
10. 既存の全不変条件(CTX-INV-1 ~ 4、MCP-INV-1 ~ 4 等)の維持
11. embedding 生成ロジックの抽象化(Contextual Retrieval を差し込める設計)
12. Contextual Retrieval の実装(デフォルト無効、基本機能のみ)
13. context キャッシュ機構(engine.db の embeddings テーブル拡張)

**Phase 2 以降に振り分ける項目**

- compile_context の `--outline-only` 引数対応
- compile_context の `--sections` 引数対応
- compile_context の `--depth` 引数対応
- Contextual Retrieval の運用開始(デフォルト有効化)
- Contextual Retrieval の prompt caching 最適化
- Contextual Retrieval のエラー処理作り込み(リトライ、フォールバック)
- ドキュメントごとの見出しレベルカスタマイズ
- 連鎖変化対応ツール(上位見出しリネーム時のエッジ自動書き換え等)
- 複数 BTS 対応のための抽象化
- Markdown 以外のドキュメント形式対応

### 8.2 設計余地の確保

Phase 1 で実装しない項目についても、将来の拡張を阻害しない設計判断を優先する。

**具体的には以下を Phase 1 の設計時に確保する。**

- graph.toml のノード ID フォーマットが `#` と `s:` を含む表記を許容
- engine.db のスキーマにサブノード関連カラム(parent_id、anchor、is_subnode、context、context_hash)を追加
- compile_context の返却構造が、サブノード情報を追加できる余地を持つ
- graph.toml のドキュメントノード定義に、対象見出しレベル指定のオプショナルフィールドを予約
- embedding 生成ロジックを抽象化し、前処理の差し替えが可能な構造とする

これらが確保されていれば、Phase 2 以降で追加実装できる。

### 8.3 ドッグフィーディングでの検証

開発環境での実運用(ドッグフーディング)で、サブノード化機能の効果を検証する。検証項目は以下。

- compile_context のトークン消費量の削減効果
- サブノード単位のドリフト検出の実用性
- 見出し自動抽出の精度
- `--granularity subnode` の使用状況
- Contextual Retrieval 有効化時の効果(必要に応じて検証)

検証結果に基づき、Phase 2 以降の拡張方針(特に Contextual Retrieval の運用開始判断)を決定する。

## 9. 制約と前提

### 9.1 前提条件

本拡張の実装には以下の前提が必要である。

- legixy が有向グラフ主体の実装を完了している(Phase 1 の基礎部分)
- graph.toml が Source of Truth となっている
- engine.db のスキーマ変更が可能な段階にある
- AI エージェントがドキュメント作成・編集の主体であること(前提条件)

### 9.2 Phase 1 での制約

Phase 1 時点では以下の制約を受け入れる。

- 対応するドキュメント形式は Markdown のみ
- 自動生成される見出しレベルは h2、h3 のみ
- サブノード ID の生成は決定論的なルールに従う(手動調整不可、明示 ID を除く)
- Phase 1 では、前世代 legixy からのマイグレーションパスは、ドキュメントノードのみの状態での移行を対象とする(サブノード情報のマイグレーションは不要)
- Contextual Retrieval はデフォルト無効。運用開始は Phase 2 以降で判断

### 9.3 既知の限界

- 見出し自動抽出は、Markdown の見出し構造に依存する。見出しが適切に使われていないドキュメントでは、サブノード化の恩恵が限定的になる
- サブノードの粒度が細かすぎると、エッジ数が急増し、管理コストが上がる可能性がある
- 明示的サブノード定義を多用すると、graph.toml の記述量が大幅に増える
- サブノード単位の embedding 計算は、ドキュメント単位より多くの計算資源を必要とする
- ハッシュ衝突の理論上の可能性は残る(16 文字で実用上ゼロ、`check --formal` で検出される)
- 上位見出しのリネームは配下のサブノード ID を連鎖的に変化させる(仕様として受容、連鎖変化対応ツールは Phase 2 以降)

## 10. 設計判断の確定事項

本 Section は、v0.1.0 の時点で「未解決の設計判断」として挙げられた論点の確定結果を記録する。議論の詳細経緯は記録しないが、確定結果のみを本文書に反映する。

### 10.1 デフォルト粒度

`compile_context --granularity` のデフォルトは `document` とする。

根拠: 後方互換性の維持、デフォルト挙動の変更による情報取りこぼしリスクの回避。

### 10.2 見出し自動抽出の対象レベル

h2 および h3 を対象とする。h4 以上は Phase 1 では対象としない。

graph.toml のドキュメントノード定義に、将来のカスタマイズのためのオプショナルフィールドを予約する。Phase 1 ではこのフィールドは使用されない。

根拠: 業界標準(Docusaurus、Python-Markdown、VS Code 等)との整合、本拡張の動機(AI コーディングエージェントのコンテキストウィンドウに収める粒度制御)から、H4 が必要な場合はサブノード化ではなくドキュメント分割で対応することが健全である。

### 10.3 サブノード ID の生成方式

**方式**: ハッシュ方式を採用する。

**ハッシュ対象**: 親ドキュメント ID + 見出し階層パス。

**アルゴリズムと長さ**: SHA-256 の先頭 16 文字(16 進数表記)。

**正規化**: 見出しテキストのトリムと連続空白統合のみ。過剰な正規化は行わない。

**明示定義との併用**: `s:` 接頭辞を持つ明示 ID と、接頭辞なしの自動生成 ID が形式的に区別される。明示 ID は graph.toml でのみ定義可能(Markdown 内の `{#id}` 記法は非対応)。文字制約は英数字・ハイフン・アンダースコア、1〜63 文字、英数字で始終。

**不変条件**: SUBNODE-INV-3(ID 一意性)と SUBNODE-INV-5(決定論性)を Section 7.2 の通り規定。SUBNODE-INV-6(明示 ID と自動生成 ID の形式的区別)を追加。

### 10.4 コア機能の BTS 非依存原則

legixy のコア機能は特定の BTS に依存しない。本原則は親文書 LEGIXY-SPEC-001 §9(設計原則)の範囲とし、本文書には記述しない。

### 10.5 Phase 1 スコープ

Section 8.1 の通り確定。任意スコープは廃止し、全項目を必須または Phase 2 以降に明示的に振り分けた。`--outline-only` と `--sections` は Phase 2 以降に振り分け、代わりに Contextual Retrieval の実装(デフォルト無効、方針 3)を Phase 1 必須スコープに追加した。

### 10.6 補償機構:Contextual Retrieval の採用

経路 2(エッジベース embedding の構造的限界)への補償機構として、RAG 業界の解法を調査し、Contextual Retrieval(Anthropic, 2024)の採用を決定した。Phase 1 で実装するが、デフォルトは無効とする。運用開始の判断は Phase 2 以降で行う。

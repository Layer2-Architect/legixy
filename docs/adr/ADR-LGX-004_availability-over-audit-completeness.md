# ADR-LGX-004: 監査ログ書込失敗時は可用性を優先する（MCP-INV-4 のベストエフォート化）

**ステータス**: accepted
**起票日**: 2026-06-10
**承認日**: 2026-06-10
**承認者**: 開発者（提案一括承認 2026-06-10）
**対象**: SPEC-LGX-003 §3 REQ.19、SPEC-LGX-007 §3 REQ.06、§4 MCP-INV-4

## 1. 文脈（Context）

- 背景: GAP-LGX-041（context_log 書込失敗時の本処理成否）、GAP-LGX-139（context_log 記録失敗時の本体影響）。compile_context の本処理成功後に監査記録のみが失敗した場合（ロック競合超過・ディスク満杯・読取専用 DB 等）、本処理を失敗扱いにするか成功扱いにするかが未確定だった。MCP-INV-4「監査ログ完全性」との緊張関係がある。
- 制約: 凍結終了コード規約（exit 1 = 受理済み引数の意味的不正・実行時失敗）。Agent（Claude Code）のコンテキスト合成は開発フローのクリティカルパス上にある。

## 2. 検討した選択肢（Options）

### 選択肢 A: 可用性優先（採用）

- 本処理結果と context_log 書込を別トランザクション化。本処理成功・記録失敗時は stdout 返却 + exit 0、stderr に Warning。MCP-INV-4 は「DB 利用可能時に限る」ベストエフォートに格下げ（欠落検出性は stderr Warning で確保）。

### 選択肢 B: 監査完全性優先

- 記録できなければ本処理も exit 1。
- 欠点: ディスク満杯等の周辺障害で Agent の読取系操作が全停止する。記録失敗は本処理の意味的不正ではないため凍結 exit-code 規約とも不整合。

## 3. 判断（Decision）

選択肢 A を採用する。

理由:

- compile_context は読取系であり、監査は事後検証手段であって本処理の前提条件ではない。可用性 > 監査完全性。
- なお GAP-LGX-126（DB 破損時 exit 1、ADR-LGX-005）とは意図的に方針が分岐する: 本件は Agent 読取経路の可用性、126 は再生成不能なユーザ生成データの保護であり、対象と保護目的が異なる。

## 4. 結果（Consequences）

### 期待される効果
- 周辺障害時も Agent のコンテキスト合成が継続し、開発フローが止まらない。

### 受け入れる代償
- context_log に欠落が生じ得る（「全呼出記録」は厳密には成立しない）。

### 残存リスク
- 欠落の見逃し → stderr Warning による検出可能性を SPEC 要件として固定。恒常的な書込失敗は運用で検知する。

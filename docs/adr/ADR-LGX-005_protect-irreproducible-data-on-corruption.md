# ADR-LGX-005: engine.db 破損時は自動再生成せず明示失敗する（再生成不能データの保護）

**ステータス**: accepted
**起票日**: 2026-06-10
**承認日**: 2026-06-10
**承認者**: 開発者（提案一括承認 2026-06-10）
**対象**: SPEC-LGX-007 §3 REQ.09、§4 FB-INV-4、STATE-INV-1

## 1. 文脈（Context）

- 背景: GAP-LGX-126（DB 破損時の挙動）。STATE-INV-1 は engine.db を「再生成可能キャッシュ」として扱うが、observations / proposals テーブルには人間・Agent の判断記録という再生成不能なユーザ生成データが含まれる。破損検出時に自動再生成（作り直し）すると証跡が無言で消失する。
- 制約: DB「不在」時の安全動作（FB-INV-4 — graph.toml のみで継続）は既定義であり、「破損」と「不在」の区別が必要。

## 2. 検討した選択肢（Options）

### 選択肢 A: 破損は exit 1 で明示失敗（採用）

- 不在（初期状態として正常）と破損(壊れた状態)を区別し、破損時は自動再生成せず exit 1。復旧（バックアップからの復元・手動再初期化）は人間判断に委ねる。

### 選択肢 B: 破損時も自動再生成

- 利点: 可用性が高い。
- 欠点: observation/proposal の証跡が無言で消える。FB-INV-2/3（承認原子性・承認前不変性）の前提となる履歴を破壊する。

## 3. 判断（Decision）

選択肢 A を採用する。

理由:

- embedding は再生成可能だが observation/proposal は再生成不能であり、後者の保護を優先する（STATE-INV-1 の「キャッシュ」扱いの明示的例外）。
- ADR-LGX-004（可用性優先）との分岐は意図的: 124 系は読取の継続性、本件は書込済みデータの保全という別問題。

## 4. 結果（Consequences）

### 期待される効果
- フィードバックループの証跡（誰が何を承認したか）が破損時にも無言消失しない。

### 受け入れる代償
- 破損時に手動介入が必要（自動回復しない）。

### 残存リスク
- 破損の検出契機・診断メッセージの粒度は DD で確定 — 検出漏れは TC 層の破損 fixture テストで担保。

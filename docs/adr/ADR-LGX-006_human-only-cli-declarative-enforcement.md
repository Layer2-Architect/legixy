# ADR-LGX-006: 「人間のみ CLI 実行」は二層構成（MCP 非露出 + 運用規律）に留める

**ステータス**: accepted
**起票日**: 2026-06-10
**承認日**: 2026-06-10
**承認者**: 開発者（提案一括承認 2026-06-10）
**対象**: SPEC-LGX-007 §3 REQ.05、§4 MCP-INV-1、NFR-LGX-001.SEC.08

## 1. 文脈（Context）

- 背景: GAP-LGX-140。approve/reject/analyze/feedback の「人間のみが実行する」要求に対し、Agent が Bash 経由で CLI を直接 spawn する経路には技術的強制が存在しない。強制手段をどこまで要求するかが未確定だった。
- 制約: NFR-LGX-001.SEC.08 は単独開発者前提を宣言している。CLI プロセスから呼出元が人間か Agent かを確実に判別する汎用手段は存在しない。

## 2. 検討した選択肢（Options）

### 選択肢 A: 二層構成 + リスク受容（採用）

1. 技術的境界: 承認系操作を MCP ツールとして非露出（MCP-INV-1）— Agent の正規経路からは構造的に実行不能
2. 運用規律: CLAUDE.md ルール 5 による宣言的強制
- Bash 直接 spawn は単独開発者前提（SEC.08）下のリスク受容として SPEC に明示。改ざん耐性ガードは要件としない。

### 選択肢 B: 技術的強制を追加（実行ユーザ判別・トークン・サンドボックス）

- 欠点: 確実な判別手段がなく迂回可能（セキュリティシアター化）。単独開発者前提の脅威モデルに対し過剰で、SEC.08 と不整合。

## 3. 判断（Decision）

選択肢 A を採用する。

理由:

- 脅威モデル（単独開発者・自己統制環境）に対し、構造的境界（MCP 非露出）+ 宣言的規律で十分。検証不能な「強制」を SPEC に書くことこそが偽の保証になる。
- リスク受容を SPEC に明文化することで、将来マルチユーザ化する際の再判断ポイント（SEC.08 改訂）が明確になる。

## 4. 結果（Consequences）

### 期待される効果
- MCP-INV-1 の保証範囲が正確に定義され、過大宣言が解消される。

### 受け入れる代償
- Agent の規律違反（Bash 直接実行）は技術的には阻止されない。

### 残存リスク
- マルチユーザ環境への展開時は本 ADR を supersede し技術的強制を再検討する（トリガ: SEC.08 改訂）。

## Q1: 矛盾 — REQ.07 の根拠記述と v3 実装事実の不一致（`[exit N]` フォーマットの再決定）

**質問**: SPEC-LGX-009 v0.5.0 REQ.07 は「v3 実測: exitCode を内部（RustCliError）に保持しながら、応答は一律 `isError: true` + stderr 本文のみで**区別不能**だった」を根拠に、`[exit N]` 新フォーマットを【v3 差分】として規定しています。しかしレビューによる実物照合の結果、**v3 の全 3 MCP ツールは既に exit code をエラーメッセージに含めています**:

```ts
// compile-context.ts:98, observe.ts:96, get-compile-audit.ts:60（全て同形式）
text: `Rust CLI failed (exit ${err.exitCode}): ${err.message}`,
```

つまり「Agent が exit 1 / exit 2 を判別できる」という REQ.07 の目的は v3 で既に成立しており、`[exit N]` の採用は虚偽の根拠に基づく不要な v3 乖離（ts-mcp 実装変更の強制）です。エラーメッセージのフォーマットをどう確定しますか?

**SPEC 上の該当箇所**: SPEC-LGX-009 §3 REQ.07（v0.5.0、SPP-LGX-009 差分 2 由来）

**選択肢**:

- [x] 選択肢 A: v3 実在形式 `Rust CLI failed (exit N): <stderr 本文>` を正準化する（実装変更不要、v3 差分なし）
- [ ] 選択肢 B: `[exit N]` 形式を意図的な v3 差分として維持し、根拠記述のみ訂正する（ts-mcp 実装変更が発生）
- [ ] その他: 下記回答参照

**回答**:

**選択肢 A を採用**（2026-06-07 開発者決定。レビュー報告に対する承認として記録。QSET-LGX-009 Q2 の回答も同日、承認済み訂正として同内容に修正済み）。

- v3 実在形式 `Rust CLI failed (exit N): <stderr 本文>` を SPEC-009 REQ.07 に明記して正準化する。【v3 差分】ラベルは撤回し「v3 既存挙動の正準化」とする。
- プロセス起動不能・シグナル等で数値 exit code が無い場合は v3 同様 `exit -1` として同形式で報告する（`ts-mcp/src/engine.ts:67` の `exitCode = -1` フォールバック）。
- 根拠: (1) 実装変更ゼロで REQ.07 の目的（Agent の検証 Error / 呼び出しミス判別）が達成済み。(2) `[exit N]` 採用は MCP 応答本文（凍結対象 (a)〜(f) 外）の範囲内で互換違反ではないが、誤った根拠で v3 乖離を固定する点が品質偏向防止（00-philosophy §1）に反する。(3) MCP-INV-2（忠実転送）とも整合。

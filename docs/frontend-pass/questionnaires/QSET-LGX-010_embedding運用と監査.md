**回答**:

（2026-06-08 開発者決定・AI 起草）

**Q1-a: 選択肢 A を採用**（v3 正準化＝非永続）。

- 整合性: 空ストア create は「結果が空」のケースであり REQ.01 の exit 0 分類と整合。
- 堅牢性: 空ストアの事実は WARNING（stderr）+ `--json {node_count:0, warning}` で既に**可視化済み**であり、本 SPEC の判定基準「黙殺されていた情報のみ可視化」に該当する沈黙は存在しない。B（メタ行永続化）の唯一の優位はダングリング snapshot_id の解消だが、その情報は create 時 WARNING で既に届いており優位は減殺される。
- 実現性: A はコスト 0。B は行複製方式とは別のメタデータテーブルを要し（DD 未凍結のスキーマを制約）、かつ空ベースラインは drift 対象として退化しており価値が希薄。
- 申し送り: 後続 delete の 0 件 WARNING との混乱を避けるため、create 時の `warning` 文言に「ストアが空のため永続化されません」を明示する旨を DD へ（REQ.02 文言は不変）。

**Q1-b: 選択肢 B を採用**（`snap-` プレフィクス＋一意性のみ SPEC、以降は DD 委任）。

- 整合性: LGX-COMPAT-001 が凍結するのは delete target が snapshot_id を受理する点のみで、内部形式は外部契約外。crate 名・parser 詳細を DD に置く本プロジェクトの高度規律と同型。
- 堅牢性: snapshot_id は consumer から見て不透明トークンであるべき。形式（epoch ミリ秒＋乱数）を SPEC 凍結すると consumer が内部構造（時刻抽出等）に依存する隠れ契約を招く。B は「一意性要求のみ SPEC」とすることでこれを防ぐ。
- 実現性: B は DD に生成方式変更の自由（将来 UUID 化等）を SPEC 改訂なしで残す。REQ.06 が既に「snapshot_id は時刻＋乱数由来・create は決定性対象外」と述べており、SPEC に残すべき要求は**一意性**である。

**Q1-c: 選択肢 A を採用**（SPEC は「決定論的に 1 件へ解決する」要求のみ、具体規則は DD）。

- 整合性: 決定性（CTX-INV-1）が SPEC レベルの関心、具体的タイブレーカーは機構＝DD。REQ.02/06 が既に「同時刻タイブレークは DD で確定」と委任しており、その方針の追認。
- 実現性: A が低摩擦。
- 申し送り: DD のタイブレーカーは **REQ.02 の label 解決（taken_at 最新）と REQ.06/list の降順安定出力の両方に同一規則**を適用すること（例: snapshot_id の全順序による安定タイブレーク）。Q1-b で形式を SPEC 凍結しないことと矛盾しない（DD が自ら選んだ形式を前提に機構を決めるだけ）。

**Q1-d: 選択肢 A を採用**（上記導出を REQ.07 に明記）。本設問は v3 が DB を自動作成していたため対応状況が存在せず、新規確定。

- 整合性: REQ.07 自身が「観測可能な差は副作用ファイルの有無のみ」と宣言している。「空ストア相当」を忠実に展開すれば delete はそのまま REQ.02 ロジックが走り、`label:<L>` は解決 0 件で **ERROR（exit 1）**、`<snapshot_id>` は該当 0 行で **WARNING（exit 0）**。設問 A の導出はこれと一致。B（一律 exit 0）は同じ「該当ラベル無し」を DB の物理存在の有無だけで exit 1/0 に分岐させ、**DB 存在を exit code に露出**させて REQ.07 自身の趣旨に反する。
- 堅牢性: `label:<L>` は名前参照であり、名前解決失敗を黙って成功扱い（B）にすると project-root 誤りや DB 欠落を WARNING で覆い隠す。A は大声で失敗。
- 実現性: A は空ストア経路をそのまま流すのみ、B は特例分岐の追加。
- 申し送り: REQ.07 に「DB 不在 ≡ 空ストア。delete の挙動は REQ.02 から導出（label 解決失敗=exit 1、snapshot_id 該当 0=exit 0）。新たな特例ではない」と明記する。

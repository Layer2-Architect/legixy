## Q1: 複数解釈 — サブノード自動 ID の SHA-256 ハッシュ入力（重大）

**質問**: REQ.05 は自動生成サブノード ID を `{親ID}#{SHA-256 の先頭16文字}` と定めますが、**SHA-256 の入力テキストが何か**が明示されていません。以下のどれですか? この確定なしに DD のハッシュ実装と SUBNODE-INV-5（ID 決定性）・SUBNODE-INV-3（ID 一意性）の検証を書けません。

**SPEC 上の該当箇所**: SPEC-LGX-002 §3 REQ.05, REQ.06（見出し正規化）

**選択肢**:

- [ ] 選択肢 A: 正規化後の **当該見出しテキスト単体**（REQ.06 の正規化を適用した文字列）
- [x] 選択肢 B: **heading_path**（h1〜当該見出しまでの祖先見出しを区切り文字で連結した経路）
- [ ] 選択肢 C: **content_range の本文バイト列**（見出し直後から次見出しまで）
- [ ] その他: <自由記述>

**回答**:

**選択肢 B を v3 実測で精密化して採用**（2026-06-07 開発者決定・AI 起草）。正確な定義:

```
subnode_id = {parent_id} + "#" + hex(SHA-256(parent_id + "|" + heading_path.join("|")))[0..16]
```

- **heading_path**: h2 見出しは `[正規化済み自見出し]`、h3 見出しは `[正規化済み h2 コンテキスト, 正規化済み自見出し]`。h1 出現で h2 コンテキストはリセット（h1 は path に含まれない）。
- **正規化（REQ.06）**: (1) Markdown 装飾 `**` `__` `~~` `*` `_` `` ` `` を長いトークン順に除去 → (2) Unicode NFC 正規化 → (3) U+3000（全角スペース）→ ASCII スペース → (4) 前後トリム → (5) 連続スペースを 1 個に統合。
- 根拠: v3 実装（`te-graph/src/subnode/id_gen.rs:6-20`、`normalizer.rs:5-48`、`extractor.rs:99-102`）。**既存プロジェクトの graph.toml に永続化済みの ID と一致させる必要があるため、この式以外の選択は移行互換上不可**。ハッシュ入力に parent_id を含む点が選択肢 B の原文に対する精密化。

## Q2: 例外未定義 / 一意性 — 同名見出しの ID 衝突回避

**質問**: Q1 で選択肢 A（見出し単体ハッシュ）を採る場合、同一ドキュメント内に正規化後同一の h2/h3 見出しが複数存在すると SHA-256 が一致し ID が衝突します。SUBNODE-INV-3（ID 一意性）はどう保証しますか?

**SPEC 上の該当箇所**: SPEC-LGX-002 §3 REQ.05、§4 SUBNODE-INV-3

**選択肢**:

- [ ] 選択肢 A: heading_path（祖先連結）をハッシュ入力に含めることで自然に区別（Q1=B と統合）
- [ ] 選択肢 B: 出現順インデックスをハッシュ入力 or サフィックスに付与
- [ ] 選択肢 C: 衝突検出時に check が Error を返し、人間に見出しのユニーク化を促す
- [x] その他: 下記回答参照（選択肢 C の Warning 変形）

**回答**:

**その他（C の Warning 変形）を採用**（2026-06-07 開発者決定・AI 起草）。

- **ID 生成式は Q1 のまま不変**とし、衝突時の生成挙動も v3 同様「同一 heading_path の同名見出しは同一 ID に縮退（自動抽出は後勝ち）」を維持する。
- **上積み（v3 差分）**: check 第 1 層に **SubnodeIdCollision 検査**を新設し、同一ドキュメント内で正規化後 heading_path が一致する見出しを検出したら **Warning** を報告する。
- severity を Error にしない理由: v3 で check が通っていた既存プロジェクトが移行直後に G1 ゲート（exit 1）で fail する互換リスクを避けるため。
- SUBNODE-INV-3 は「明示ノードと衝突する自動 ID は生成 skip（v3 既存挙動）、自動同士の衝突は Warning で可視化」と精密化する。
- v3 の無言上書きの根拠: `te-graph/src/parser.rs:126-145`（衝突検出機構なし）。
- 補強根拠（2026-06-07 追記、整合性・堅牢性・実現性の 3 軸比較で採用確定）: v3 自身のサブノード仕様 TE-NEXT-EXT-001 は SUBNODE-INV-3（§7.2）に加え「ハッシュ衝突の理論上の可能性は残る（…`check --formal` で検出される）」（line 725）と規定しており、本 Warning 新設は **v3 仕様が約束しながら実装が果たさなかった検出の回復**である。v3 実装維持はその仕様乖離の温存を意味し、3 軸全てで劣後。

## Q3: 責務範囲 / 境界 — refresh-subnodes の責務所在

**質問**: 凍結済みコマンド `refresh-subnodes`（見出しリネーム時のサブノード ID 連鎖反映、`--apply` 時 `.refresh-bak.{epoch}` バックアップ、LGX-COMPAT-001 §4 #9）の要求が本 SPEC にありません。サブノード ID は heading 依存のため、見出しリネームで ID が変わり、graph.toml の既存エッジ `from`/`to` が未解決化します。この検出・書き換え・バックアップの責務は本 SPEC（グラフ基盤）が負いますか?

**SPEC 上の該当箇所**: SPEC-LGX-002 §3 REQ.05、SPEC-LGX-001 QSET Q1 と連動

**回答**:

（2026-06-07 開発者決定・AI 起草）

**はい、本 SPEC（グラフ基盤）が負う**（QSET-LGX-001 Q1 で SPEC-LGX-010 新設を決定したが、`refresh-subnodes` は embedding 非依存のため 010 ではなく本 SPEC）。以下を REQ として追記する:

- 見出しリネームにより未解決化したサブノード ID の検出（dry-run 既定）
- `--apply` 時の graph.toml 書き換え: `[[nodes]]` の `id`・`parent`、`[[edges]]` の `from`・`to` を新 ID へ連鎖反映
- 書き換え前バックアップ: `graph.toml.refresh-bak.{unix epoch 秒}`
- 根拠: v3 実装（`te-cli/src/commands/refresh_subnodes.rs:285-360`）。graph.toml の構造操作でありグラフ基盤の責務。

## Q4: 例外未定義 — 入力耐性のしきい値と超過時挙動

**質問**: REQ.10 は「大きすぎる Markdown ファイル」「深いネスト見出し」でクラッシュしないことを要求しますが、**しきい値（バイト数 / ネスト深さ）と超過時の挙動が未定義**です。後段（DD/TS のファズテスト境界、SPEC-LGX-003 REQ.13 の 500,000 文字エラーとの整合）に必要です。超過時は (a) 切り捨て (b) Error 返却 (c) Warning + 継続 のいずれですか? また数値しきい値はありますか?

**SPEC 上の該当箇所**: SPEC-LGX-002 §3 REQ.10

**回答**:

（2026-06-07 開発者決定・AI 起草）

**数値しきい値は設けない（その他）**。v3 のパーサはサイズ・ネスト上限を持たない 1 行走査の線形処理であり（`te-graph/src/subnode/extractor.rs:56-86`）、新規上限の導入は「v3 で処理できたファイルが legixy でエラーになる」互換破壊となるため。REQ.10 は以下のとおり明文化する:

- 入力サイズ・見出しネスト深さに**明示上限なし**。行単位走査で入力サイズに対し線形に動作する。
- 対象外レベルの見出し（既定 h2/h3 以外、h4 以深等）は**無視**（エラーにしない）。
- メモリ不足等の OS 起因の失敗は通常のエラー経路（exit 1）で報告し、クラッシュ（panic/abort）しない。
- 出力側の上限は SPEC-LGX-003 REQ.13（500,000 文字）が既に規定しており入力側と独立。
- 巨大ファイル・深ネストのファズテスト境界は TS の検証観点として規定し、SPEC のしきい値とはしない。

# core-perspectives.md（決定論層・領域固有観点ベース）

このファイルは TP 生成時に AI が参照する観点ナレッジベース（決定論層用）。
プロジェクト固有の観点を蓄積していく。汎用 QA 観点 + ドメイン固有観点の合流点。

> **このファイルは雛形である。** プロジェクトに導入後、領域固有観点を追記して育てること。
> AT で発見された観点は必ずここに昇格させる（→ `07-at-and-nfr.md` §2）。

---

## 使い方

- TP 生成時、AI はこのファイルを参照して観点を生成する
- 新しい観点を発見したら（特に AT で発見されたものは）ここに追記する
- 観点の出所（既存ツール、業界慣行、過去のインシデント等）を可能なら明示

## 汎用観点（チェックリスト）

### 境界値

- [ ] 0 / 1 / 上限 / 上限+1
- [ ] 負の値 / 浮動小数点の特殊値（NaN, ±Inf, -0.0）
- [ ] 空入力 / 単一要素 / 最大要素数
- [ ] 文字列の長さ境界（0, 1, max, max+1）, Unicode 境界（surrogate pair, normalization, BiDi, ZWJ）
- [ ] 時刻の境界（epoch, 閏秒, 閏年, タイムゾーン境界, DST 境界）
- [ ] サイズ・容量の境界（メモリ・ディスク・ネットワークペイロード）

### エラーハンドリング

- [ ] 想定エラー全てに型がついているか
- [ ] 部分成功の扱い（途中でエラーが起きたとき、何が確定し何が破棄されるか）
- [ ] エラー伝播時の情報損失がないか（cause チェーンの保持）
- [ ] panic / unwrap / unhandled exception が本番コードに残っていないか
- [ ] エラー時の副作用解放（ファイル / ロック / 外部接続）保証
- [ ] リトライ可否の判定（一過性 vs 永続的）

### 状態遷移

- [ ] 各状態への入口・出口が定義されているか
- [ ] 遷移不可能な状態組合せが型で禁止されているか
- [ ] 中断・再開のセマンティクスが定義されているか
- [ ] 終端状態（success / cancelled / failed）の保証
- [ ] 並行操作下での状態遷移の atomicity

### 並行性

- [ ] 共有可能な値と共有不可能な値が型で区別されているか
- [ ] critical section の境界が明確か
- [ ] deadlock になりうる lock 順序がないか
- [ ] 複数操作の競合時の優先順位
- [ ] cancellation 伝播の保証
- [ ] backpressure の扱い

### バージョニング・互換性

- [ ] データ形式の前方互換性
- [ ] データ形式の後方互換性
- [ ] migration path の存在
- [ ] 旧バージョンデータの読込時挙動
- [ ] 公開 API のバージョン互換性ポリシー
- [ ] 依存ライブラリのバージョン制約

### 永続化

- [ ] 保存中の電源断・プロセス kill
- [ ] ディスクフル / 権限エラー
- [ ] 部分書き込みからの回復
- [ ] 複数プロセスからの同時アクセス
- [ ] トランザクション境界の明確化
- [ ] バックアップ・復旧時の整合性

### 入力検証

- [ ] 検証タイミング（境界 / Boundary / Control / Entity のどこで）
- [ ] 検証段階（構文 / 形式 / 意味 / 業務ルール）
- [ ] 検証失敗時のメッセージ品質
- [ ] サニタイゼーション漏れによる injection 系脆弱性

### ライフサイクル

- [ ] 初期化失敗時のエラー詳細
- [ ] 起動所要時間
- [ ] shutdown 順序保証（進行中処理のキャンセル待ち、flush、外部切断）
- [ ] リソース解放の完全性
- [ ] シングルインスタンス制約（必要な場合）

### ロギング・観測性

- [ ] 失敗時に問題を特定できるだけの情報がログに出ているか
- [ ] 機密情報がログに混入していないか（PII, secret, token）
- [ ] ログレベルの一貫性
- [ ] 構造化ログのスキーマ安定性

## UC レベル観点（TP[UC] 生成時に必ず網羅）

### 基本フロー

- [ ] 各ステップの前提条件が先行ステップの事後条件を満たすか
- [ ] アクター入力の検証タイミング（段階 1 構文 / 段階 2 形式 / 段階 3 意味）
- [ ] 成功時の事後条件が観察可能か（次の UC の前提として参照可能か）

### 代替フロー

- [ ] 分岐条件が網羅されているか（境界値・列挙の全 case）
- [ ] 各代替フローも基本フローと同じ事後条件に収束するか、または別途定義された事後条件を持つか
- [ ] 代替フローへの遷移条件が明示されているか

### 例外フロー

- [ ] 各ステップでの失敗パスが定義されているか
- [ ] エラー時の状態は不変条件を満たすか（中間状態でアグリゲートが壊れていないか）
- [ ] エラー時のユーザ通知（メッセージキー、ローカライズ）が定義されているか
- [ ] 部分成功時の扱い（ロールバック / 部分コミット / リトライ）

### アクター遷移と権限

- [ ] アクターの権限・状態変化が一貫しているか
- [ ] 責任境界が守られているか
- [ ] ロック中・編集権限なしの状態で操作試行された場合の挙動

### データフロー

- [ ] 入出力データの型・制約・ライフタイム
- [ ] パラメータの所有権移動規約
- [ ] エラー時のデータ解放保証

## FFI / 境界 API 観点（境界 API を持つプロジェクト用）

### Marshalling

- [ ] アグリゲート: opaque handle 経由か / 値渡しか
- [ ] 値型: POD struct か（言語横断レイアウトの安定性）
- [ ] バルクデータ: 一括転送 API があるか
- [ ] 文字列: エンコーディング、null 終端、所有権境界の明示
- [ ] 列挙体: 固定表現、値の安定性

### Panic / エラー

- [ ] 全 FFI 公開関数が panic catch でラップされているか
- [ ] panic / unhandled exception 発生時のエラー code 変換
- [ ] コールバック内の panic も同様にラップ必須

### 性能予算

- [ ] 単発呼出の性能予算（マイクロ秒オーダー）
- [ ] 高頻度呼出シナリオでバッチ API があるか
- [ ] バルク描画・大量データ転送の予算

### ABI 互換性

- [ ] 公開関数シグネチャの変更検出
- [ ] 構造体レイアウト変更
- [ ] enum 値変更
- [ ] スキーマ非互換変更
- [ ] コールバック契約変更（呼出規約・スレッド契約）
- [ ] ABI バージョンのインクリメント漏れ

### 非同期境界

- [ ] コールバック発火スレッドの仕様化
- [ ] callback 用 user_data の寿命管理
- [ ] キャンセル伝播
- [ ] 非同期タスク abort 時のリソース解放

## ローカライズ観点（多言語対応プロジェクト用）

- [ ] core から返されるメッセージはキーとパラメータのペアか（文字列直埋め込みなし）
- [ ] キー命名規則の遵守
- [ ] パラメータ型の宣言と実呼出の一致
- [ ] フォールバック順序の定義
- [ ] CI 整合検査の対象（各ロケールリソース ↔ キーリスト）

## 領域固有観点（ここから下はプロジェクトに応じて追記）

### 例: タイムライン / イベント操作系

- [ ] 空タイムラインへの最初の操作の挙動
- [ ] 全イベント削除後の挙動
- [ ] 同時刻イベントの順序
- [ ] イベント間の因果整合性
- [ ] 過去イベントの編集が未来イベントに与える影響
- [ ] ロック済みイベントへの編集試行

### 例: バージョン管理系（Git 由来）

- [ ] 分岐とマージの結合則
- [ ] 競合解消の semantics
- [ ] 履歴改変の検出
- [ ] ロック中の他者編集の挙動

### 例: 決済 / 取引系

- [ ] 二重課金防止（idempotency key）
- [ ] 通貨換算の rounding ルール
- [ ] チャージバック・返金の整合性
- [ ] 不正検知のしきい値

### 例: 大規模データパイプライン

- [ ] バックフィル時の冪等性
- [ ] スケジュール遅延時の補償処理
- [ ] downstream 障害時の dead letter queue
- [ ] 入力スキーマ変更への耐性

## 4. 並行性（概念レベル）

`impact` は読み取り専用の逐次走査であり、ドメインレベルの並行性はない。グラフ読み込み→起点確認→BFS 走査→結果整形の各処理は順方向探索統括処理の協調下で逐次進む。並行アクセス時の整合性は走査対象外（グラフ状態変更なし: UC-006 事後条件）。具体的な並行機構は DD で扱う。

## 5. 失敗伝搬

- 各操作の戻り値は「結果」概念（成功 / 失敗 + 理由）で表現。具体的なエラー型は DD で確定。
- グラフ定義の読込失敗（グラフ定義境界 → グラフ読み込み処理）は走査結果の失敗記録として記録され、終了状態（失敗）へ伝搬する。
- 起点ノード不在は失敗ではなく「空の走査結果」として扱われ、終了状態（成功）で返却する（UC-LGX-006 代替 2a）。
- 打ち切り情報（最大深度超過）は警告的所見であり、終了状態（成功）を変えない。

## 6. Behavior Allocation（操作のクラス帰属、§6.3）

各操作は一つのクラスに帰属する（複数クラスへの分散なし）。Boundary=境界操作のみ / Control=複数 Entity の協調 / Entity=自身のデータ操作。

| 操作 | 帰属クラス | 役割 | 妥当性 |
|---|---|---|---|
| 順方向探索を受け付ける | 順方向探索コマンド受付窓口 | Boundary（アクター境界） | ✓ 境界操作のみ |
| 探索を統括する | 順方向探索統括処理 | Control（協調） | ✓ グラフ読み込み・起点確認・BFS 走査・結果整形を依頼する中心 |
| グラフを読み込む / ノードとエッジを構築する | グラフ読み込み処理 | Control | ✓ グラフ定義境界→有向グラフへの変換を担う |
| グラフ定義を読み込む / グラフ定義の存在を確認する | グラフ定義境界 | Boundary（外部ファイル境界） | ✓ 境界操作のみ |
| 起点ノードの存在を確認する | 起点確認処理 | Control | ✓ 有向グラフへの照合依頼のみ（走査の越権なし） |
| 起点ノードを照合する / 下流エッジを辿る | 有向グラフ | Entity（自身のデータ） | ✓ ノード・エッジコレクションへの自身データ操作 |
| 有向グラフを順方向に走査する / 打ち切り情報を生成する / 空の走査結果を生成する | BFS走査処理 | Control | ✓ 有向グラフ・走査結果・打ち切り情報を協調利用 |
| 到達ノードと深度を記録する | 走査結果 | Entity（自身のデータ） | ✓ 自身属性（到達ノード・深度情報・失敗記録）への書き込み |
| 走査結果を整形する | 走査結果整形処理 | Control | ✓ 走査結果・打ち切り情報を読み出力窓口へ渡す |
| 走査結果を出力する / 打ち切り情報を出力する | 走査結果出力窓口 | Boundary | ✓ 境界操作のみ |

割り当てに迷う操作なし。各操作が UC ステップ / SEQA メッセージに対応（余剰操作なし）。

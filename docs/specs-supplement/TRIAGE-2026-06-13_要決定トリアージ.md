SPEC 系=`.legixy/engine.db`、v3 実装/互換契約=`.trace-engine/`、実在 v0.1.0=`feedback.db` の三系統不一致。
- **関連項目**: SUPP-007 §2-4, SUPP-010 C-4, SUPP-006 §3, SUPP-008 2.1, SUPP-003 S2-22。
- **決定（1つ）**: 正準 DB パスと旧名フォールバック方針 + v0.1.0 移行元ファイル名。

### クラスタ C — env 変数・モデル配布（env-config）
- **関連項目**: SUPP-009 §2.3（MCP バイナリ解決 `TRACEABILITY_ENGINE_BIN`/`LEXIGY_BIN`/`LEGIXY_BIN` 三系統）, SUPP-010 D-2（`LGX_MODELS_DIR`/`TE_MODELS_DIR` 解決順）, SUPP-006 2.1-c（モデル配布）, 2.6-b（LLM API キー名）。

### クラスタ D — v3↔SPEC 乖離の統治方針（v3-divergence）※ 最重要
SPEC を字義実装すると v3 観測挙動が変わる項目が多数。各個別裁定の前に**統治原則**を 1 つ決めると残りが従う。
- **決定（1つ）**: 既定方針 = 「**SPEC 準拠を原則とし、v3 観測挙動を変える乖離は ADR + 人間承認の加算的拡張として個別に格上げ**」か、「**v3 観測挙動を既定維持し、SPEC 側を【v3差分】注記で追従**」か。
- **推奨**: **SPEC 準拠原則 + 個別 ADR**（DevProc の加算的拡張規律に整合）。これにより §4 の個別 RBA 阻害項目は「SPEC 寄りデフォルト + 例外のみ議論」に縮約。

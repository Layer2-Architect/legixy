# TS-LGX-003: サブノード自動抽出のテスト仕様

> TS は **上流 TP の翻訳**。ゼロから観点を考えない。各 TP 観点を DD-LGX-003 で確定した型・関数シグネチャに即した具体的な入力・期待出力・前提条件へ展開する。

**親 DD**: DD-LGX-003
**継承 TP**: TP-LGX-002（TP[SPEC] グラフ基盤、41 観点）, TP-LGX-013（TP[UC] UC-003 サブノード自動抽出フロー、24 観点）。DD-LGX-003 §8 の TP 引用（TP-LGX-004 Unit/Integration、NFR-LGX-001 Fuzz/Bench）も確認のうえ、本 TS 直接ケース化 or 委譲を §3 に明記する。

委譲（本 TS 対象外）:
- **SUBNODE-INV 違反の検出・SubnodeIdCollision Warning の発行・ID 一意性強制**: 生成段階では行わない（DD-LGX-003 §6、SPEC-LGX-002.REQ.12）。検出・severity 付与は **TS-LGX-001 / SPEC-LGX-004.REQ.14** が所有 → 委譲。本 TS は「生成段階で静粛に縮退・スキップする」挙動のみ検証する。

- **refresh-subnodes（dry-run/--apply・バックアップ・atomicity・命名衝突・並行アクセス）**: 別 UC（Admin Surface）の所有（TP-LGX-002 G-PS-1/2/3, G-CC-1, G-FA-1/2, G-VC-3）→ 該当 UC の TS へ委譲。本 TS の射程外。
- **性能予算（PERF.04/05）・ファズ堅牢性（HARDEN.03）**: → **NFR-LGX-001 / bench / fuzz** へ委譲（DD-LGX-003 §8 Bench/Fuzz 行）。本 TS は決定論・正確性を検証し、性能数値・任意バイト列の網羅クラッシュ試験は持たない。
- **BOM/改行コード正規化（COMPAT.07/08）**: ファイル IO 層が所有（TP-LGX-002 G-VC-4、NFR-LGX-001）→ NFR へ委譲。本 TS は正規化済み入力に対する走査を検証する。

本 TS は「`legixy-graph` の抽出・ID 生成・正規化・判定が SPEC-LGX-002.REQ.05/06/10/12 を DD-LGX-003 の型で正しく具体化し、v3 生成 ID と一致するか」を検証する。

## 2. ケース一覧

### ケース 1: 既知 (parent_id, heading_path) → v3 一致 ID（凍結契約・最重要）

- **観点出典**: TP-LGX-002 §2.5 G-VC-2（ID 生成式 v3 互換）, §2.11 G-DM-1（アドレッシング決定論）, TP-LGX-013 §2.6 R2/R5（凍結式委譲・SUBNODE-INV-5）
- **分類**: Contract
- **前提**: v3 `id_gen.rs:7-20` の生成式 `{parent_id}#{hex(SHA-256(parent_id + "|" + heading_path.join("|")))[0..16]}`。v3 ユニットテスト fixture / 既存 graph.toml に永続化済みの (parent_id, heading_path)→ID ペアを底本に持ち込む
- **入力**: 既知ペア例 `generate_subnode_id("SPEC-LGX-002", &["概要".to_string()])`、および h3 例 `generate_subnode_id("DD-LGX-003", &["公開 API surface".to_string(), "凍結".to_string()])`
- **期待**: 戻り値が v3 が生成する文字列とバイト単位で一致。形式は `{parent_id}#{16文字小文字hex}`。hex 部は `[0..16]`（hex 文字列先頭 16 文字、= digest 先頭 8 バイト）
- **境界条件**: 生成式・ハッシュ入力連結順（`parent_id` + `"|"` + `join("|")`）・切り詰め位置（hex16 文字）はいずれも凍結。1 ビットでもズレれば永続化済み ID と不一致になる破壊的変更

### ケース 2: ID 生成の決定論（property）

- **観点出典**: TP-LGX-002 §2.11 G-DM-1（同一 parent_id + heading_path → 常に同一 ID）, §2.4 G-CC-2（並行下の決定論）
- **分類**: Property-based（proptest）
- **生成器**: 任意の `parent_id: String` と `heading_path: Vec<String>`（要素数 0〜3、各要素任意 Unicode）
- **不変条件**: `generate_subnode_id(p, &hp)` を複数回呼んでも常に同一文字列。並行スレッドから呼んでも結果一致（共有状態なし、read-only）。出力 hex 部は常に 16 文字・全て `0-9a-f`
- **反例ハンドリング**: shrink して最小の不一致例を記録（理論上発生しないが回帰検知）

### ケース 3: heading_path 構築 — h2 は自見出しのみ

- **観点出典**: TP-LGX-002 §2.11 G-DM-2（heading_path 構築規則）, TP-LGX-013 §2.5 DF2
- **分類**: Unit
- **前提**: h2 見出し 1 つのみの Markdown（v3 `extractor.rs:99-102` の `(2,_) => vec![normalized]`）
- **入力**: `extract_subnodes("UC-LGX-003", "## 概要\n本文\n")`
- **期待**: `Vec` 長 1。`[0].heading_path == vec!["概要"]`（1 要素）。`[0].id == generate_subnode_id("UC-LGX-003", &["概要".to_string()])`。`[0].parent_id == "UC-LGX-003"`
- **境界条件**: h2 の heading_path は自見出しのみ（h2 コンテキストを自身に含めない）

### ケース 4: heading_path 構築 — h3 は直上 h2 コンテキスト + 自見出し

- **観点出典**: TP-LGX-002 §2.11 G-DM-2, TP-LGX-013 §2.5 DF2, §2.5 DF3（ハッシュ対象文字列の `|` 区切り整合）
- **分類**: Unit
- **前提**: h2 の下に h3 が続く Markdown（v3 `extractor.rs:99-102` の `(3, Some(h2)) => vec![h2, normalized]`）
- **入力**: `extract_subnodes("UC-LGX-003", "## 基本フロー\n## 概要\n### 詳細\n")`
- **期待**: h3 ノードの `heading_path == vec!["概要", "詳細"]`（2 要素、直上 h2 = `概要`）。ハッシュ対象は `"UC-LGX-003|概要|詳細"`。`id == generate_subnode_id("UC-LGX-003", &["概要".to_string(),"詳細".to_string()])`
- **境界条件**: h3 の直上 h2 は「直近に確定した h2 コンテキスト」（複数 h2 がある場合は最後のもの）

### ケース 5: h1 出現で h2 コンテキストがリセットされる

- **観点出典**: TP-LGX-002 §2.11 G-DM-2（h1 でリセット）, §2.1 G-BV-5（対象外見出しレベル）, TP-LGX-013 §2.6 R6（h2 コンテキストなしの h3）
- **分類**: Unit
- **前提**: `## A` → `# H1` → `### B` の順（v3 `extractor.rs:57` の `level==1 => current_h2 = None`）。h1 自体は heading_path に含まれず抽出対象外
- **入力**: `extract_subnodes("DOC", "## A\n# H1\n### B\n")`
- **期待**: 抽出ノードは `A`(h2) と `B`(h3) の 2 件。`B` の `heading_path == vec!["B"]`（h2 コンテキストが None にリセット済 → `(3, None) => vec![normalized]`）。h1（`H1`）由来ノードは生成されない
- **境界条件**: h1 は heading_path 非寄与かつリセット作用のみ。コンテキストなし h3 は 1 要素パス

### ケース 6: 対象外見出しレベル h4 以深は無視（クラッシュなし）

- **観点出典**: TP-LGX-002 §2.1 G-BV-5（h4 以深対象外）, §2.2 G-EH-1（クラッシュ禁止）, TP-LGX-013 §2.2 AF4
- **分類**: Unit
- **前提**: `extract_subnodes` の既定 levels = `[2,3]`（v3 `extractor.rs:30-32`）。`####` 以降は `target(level)` false で無視
- **入力**: `extract_subnodes("DOC", "#### H4\n##### H5\n###### H6\n")`
- **期待**: 空 `Vec`（長 0）。panic / Err なし
- **境界条件**: h4(level=4)/h5/h6 は抽出対象外。h7 相当（`#######`）は `atx_level` が `count>6 => None` で非見出し扱い（v3 `extractor.rs:122-124`）

### ケース 7: ATX 見出し判定 — `#` の後に空白必須

- **観点出典**: TP-LGX-002 §2.1 G-BV-3（見出し検出）, §2.7 G-IV-1 周辺（抽出規則）, TP-LGX-013 §2.1 BF2（2b 見出し抽出）
- **分類**: Unit
- **前提**: `atx_level`（v3 `extractor.rs:117-138`）は `#` 連続の直後が space/tab/EOL のときのみ見出し。`##foo`（空白なし）は非見出し
- **入力**: (a) `extract_subnodes("DOC", "##no space\n")` / (b) `extract_subnodes("DOC", "## ok\n")` / (c) 行末 `##`（rest = None）`extract_subnodes("DOC", "##\n")`
- **期待**: (a) 空 Vec（`##no space` は見出しでない）。(b) 1 件（`ok`）。(c) `##` のみは `atx_level=Some(2)` だが正規化後 anchor 空 → 後述ケース 10 によりスキップ（空 Vec）
- **境界条件**: `#`直後の文字種（space/tab/EOL=見出し、それ以外=非見出し）が判定境界

### ケース 8: closing `#` の除去

- **観点出典**: TP-LGX-002 §2.7 G-IV-1（見出し抽出の anchor 切り出し）, TP-LGX-013 §2.1 BF2
- **分類**: Unit
- **前提**: v3 `extractor.rs:60-64` は見出しマーカー除去後に `.trim_end().trim_end_matches('#').trim_end()` で closing `#` を除去
- **入力**: `extract_subnodes("DOC", "## 概要 ##\n")`
- **期待**: 抽出 1 件。`heading_path == vec!["概要"]`（closing ` ##` 除去後 trim）。`anchor`（生見出し）も closing `#` 除去済の `概要`
- **境界条件**: 末尾連続 `#` の除去とその前後 trim

### ケース 9: content_range のバイト半開区間 [body_start, end_byte)

- **観点出典**: TP-LGX-002 §2.7 周辺（AutoSubnode 構造の正確性）, TP-LGX-013 §2.5 DF1（出力データの型・制約）。DD-LGX-003 §8「content_range バイト範囲の正確性」
- **分類**: Unit
- **前提**: content_range は「見出し行の次行頭バイト（body_start）」から「同レベル以浅の次見出しの行頭バイト、なければ EOF（end_byte）」（v3 `extractor.rs:71, 92-98, 109`）
- **入力**: `extract_subnodes("DOC", "## A\nbodyA\n## B\nbodyB\n")`（バイト境界既知の ASCII fixture）
- **期待**: `A` の `content_range == (body_start_A, line_start_B)`（`## B` の行頭で終端）。`B` の `content_range == (body_start_B, EOF=content.len())`。両区間は半開・非負・`start <= end`
- **境界条件**: 同レベル(h2)以浅で終端。h3 の content_range は次の同レベル(h3)以浅見出しまで含む（h3 の下の h4 は跨ぐ）。最終見出しは EOF 終端

### ケース 10: 空見出し・装飾のみ見出し → 正規化後空文字列でスキップ

- **観点出典**: TP-LGX-002 §2.1 G-BV-6（正規化後に空文字列となる見出し）, TP-LGX-013 §2.1 BF2
- **分類**: Unit
- **前提**: v3 `extractor.rs:66-69`：`normalize_heading(raw).is_empty()` のときサブノード非生成。さらに `level==2` のとき空 h2 は `current_h2 = None`（後続 h3 のコンテキストにしない）
- **入力**: (a) `extract_subnodes("DOC", "## \n")`（空白のみ）/ (b) `extract_subnodes("DOC", "## **__**\n### C\n")`（装飾のみ h2 + h3）
- **期待**: (a) 空 Vec。(b) 空 h2 はスキップかつ `current_h2=None`、`C`(h3) は `heading_path == vec!["C"]`（コンテキストなし）。装飾のみ h2 ノードは生成されない
- **境界条件**: 正規化後 0 文字 = スキップ。空 h2 のコンテキストリセット作用

### ケース 11: 正規化 5 段階の適用順序（装飾→NFC→全角空白→trim→連続空白統合）

- **観点出典**: TP-LGX-002 §2.7 G-IV-1（正規化規則）, TP-LGX-013 §2.6 R1。DD-LGX-003 §5.1 適用順序・§8「順序依存テスト」
- **分類**: Unit
- **前提**: v3 `normalizer.rs:14-48` / DD-LGX-003 §5.1。順序: ①装飾除去（`**`/`__`/`~~`→ 次に `*`/`_`/`` ` ``、長いマーカー優先）②NFC ③U+3000→ASCII 空白 ④trim ⑤連続 whitespace→単一 ASCII 空白
- **入力**: (a) `normalize_heading("  **太字**  リンク  ")` / (b) `normalize_heading("A\u{3000}B")`（全角空白）/ (c) `normalize_heading("a\tb  c")`（タブ + 連続空白）
- **期待**: (a) `"太字 リンク"`（装飾除去 + 前後 trim + 連続空白統合）。(b) `"A B"`（全角→半角 1 つ）。(c) `"a b c"`（タブ等 whitespace 全種別を単一 ASCII 空白へ）
- **境界条件**: 長いマーカー優先（`**` を `*` より先に除去しないと装飾が残る）。順序を変えると ID が変わるため凍結

### ケース 12: 正規化の NFC（合成済み vs 分解の同一化）

- **観点出典**: TP-LGX-002 §2.11 G-DM-3（Unicode 正規化エッジ: 合成済み vs 分解、互換等価、ZWJ）, §2.5 G-VC-4 関連
- **分類**: Unit
- **前提**: v3 `normalizer.rs:25` の `.nfc()`。NFD（分解）入力を NFC（合成）へ正規化
- **入力**: 分解形 `normalize_heading("é")`（`e` + U+0301 結合アクセント）と合成形 `normalize_heading("é")`（U+00E9）
- **期待**: 両者の戻り値がバイト単位で一致（ともに NFC 合成形 U+00E9）。したがって両入力から生成される ID も一致
- **境界条件**: 合成済み/分解の差異が ID 不安定性を生まないこと（Prettier 等フォーマッタ耐性、REQ.06 根拠）

### ケース 13: 正規化の冪等性（property）

- **観点出典**: TP-LGX-002 §2.7 G-IV-1（決定的適用）, §2.11 G-DM-1。DD-LGX-003 §8 Property「`normalize_heading` の冪等性」
- **分類**: Property-based（proptest）
- **生成器**: 任意 `text: String`（Unicode 任意、装飾文字・全角空白・結合文字を含む分布）
- **不変条件**: `normalize_heading(normalize_heading(t)) == normalize_heading(t)`（正規化済み文字列を再正規化しても不変）。出力に U+3000・連続空白・前後空白・装飾マーカーを含まない
- **反例ハンドリング**: shrink して最小の非冪等例を記録

### ケース 14: `is_auto_generated_fragment` — ちょうど 16 文字小文字 hex

- **観点出典**: TP-LGX-002 §2.3 G-ST-2（自動/明示の形式的区別）, §1.4 SUBNODE-INV-6, TP-LGX-013 §2.6 R3
- **分類**: Unit
- **前提**: v3 `id_gen.rs:64-71`：`len()==16` かつ全文字 `0-9a-f` のとき true
- **入力**: (a) `is_auto_generated_fragment("0123456789abcdef")` / (b) `"0123456789ABCDEF"`（大文字）/ (c) `"0123456789abcde"`（15 文字）/ (d) `"0123456789abcdef0"`（17 文字）/ (e) `"s:overview"`（明示）/ (f) `"0123456789abcdeg"`（`g` は hex 外）
- **期待**: (a) true。(b) false（大文字 hex は対象外）。(c) false（15）。(d) false（17）。(e) false。(f) false
- **境界条件**: 長さ 15/16/17（下限-1/一致/上限+1）と文字種（小文字 hex のみ）の両境界

### ケース 15: `validate_explicit_name` — 文字制約・長さ・両端英数

- **観点出典**: TP-LGX-002 §2.7 G-IV-2（明示 ID `s:` の文字制約）, TP-LGX-013 §2.4 AT2
- **分類**: Unit
- **前提**: v3 `id_gen.rs:27-59`：非空・`len()<=63`・各文字が英数/`-`/`_`・先頭末尾が英数
- **入力**: (a) `validate_explicit_name("overview")` / (b) `""`（空）/ (c) 64 文字の英数列 / (d) `"-abc"`（先頭ハイフン）/ (e) `"abc-"`（末尾ハイフン）/ (f) `"ab.c"`（不正文字 `.`）/ (g) `"a_b-c1"`（中間 `_`/`-` は可）
- **期待**: (a) `Ok(())`。(b) `Err`（空）。(c) `Err`（長さ 64 > 63）。(d) `Err`（先頭非英数）。(e) `Err`（末尾非英数）。(f) `Err`（不正文字 `.`）。(g) `Ok(())`
- **境界条件**: 長さ 0/1/63/64（空/下限/上限/上限+1）。両端英数 vs 中間記号許容の境界

### ケース 16: `validate_explicit_name` 長さ境界（63/64、property 補強）

- **分類**: Property-based（proptest）
- **生成器**: 英数のみから成る長さ 1〜70 の `String`
- **不変条件**: 長さ 1〜63 → `Ok`、長さ 64 以上 → `Err`（先頭末尾英数・中間英数のため文字制約は常に満たす入力空間に限定）

### ケース 17: パース統合 — ドキュメント + 自動生成サブノード + ParentChild エッジ

- **観点出典**: TP-LGX-002 §2.1 G-BV-1/G-BV-2, TP-LGX-013 §2.1 BF1/BF2, §2.5 DF1/DF4。DD-LGX-003 §8 Integration「パース統合」
- **分類**: Integration
- **前提**: `parse_graph_from_str` に 1 ドキュメントノード（`.md`、実体は project_root 配下に存在）を含む graph.toml。当該 .md に h2 が 2 件
- **入力**: `parse_graph_from_str(toml_str, project_root)`（fixture: ドキュメントノード 1 + 見出し 2 件）
- **期待**: `Ok(graph)`。graph に Document ノード 1 + AutoGenerated サブノード 2。各サブノードは `SubnodeKind::AutoGenerated`、`parent_id == Some(ドキュメントID)`、`path` が親と一致（SUBNODE-INV-2）。各サブノードへ `EdgeKind::ParentChild` エッジが親から 1 本ずつ生成される（v3 `parser.rs:147-157`）
- **境界条件**: サブノードゼロ運用（見出しなし .md）でも `Ok` かつ ParentChild エッジ 0（後方互換、G-BV-2）

### ケース 18: 縮退 — 同一 heading_path の複数見出しは同一 ID に後勝ち統合

- **観点出典**: TP-LGX-002 §2.8 G-LC-2（縮退）, §2.1 G-BV-7（切り詰め衝突は INV-3/SPEC-004 へ）, TP-LGX-013 §2.2 AF2。DD-LGX-003 §8 Integration「衝突・縮退」
- **分類**: Integration
- **前提**: 同一 .md に正規化後 heading_path が一致する h2 が 2 つ（→ REQ.05 生成式により同一 ID）。v3 は `IndexMap::insert` の後勝ちで 1 ノードに縮退（v3 `parser.rs:143-145`）。生成段階で Warning/Err なし

- **期待**: graph 上 `概要` サブノードは 1 ノードのみ（縮退）。エラー・Warning は発しない（`Ok`）。INV-3 違反としての検出は本 TS 対象外（→ TS-LGX-001 / SPEC-004.REQ.14 へ委譲）

### ケース 19: 明示優先 — 明示サブノードと衝突する自動生成はスキップ

- **入力**: `parse_graph_from_str` に「明示ノード（衝突 ID）」+「同 ID を生む見出しを持つ .md」を含む graph.toml

- **境界条件**: 明示宣言 > 自動生成の優先順位。スキップは静粛（生成段階で通知しない）

### ケース 20: 部分失敗継続 — 一部ドキュメントファイル不在/読込失敗をスキップして残りを処理

- **観点出典**: TP-LGX-002 §2.2 G-EH-3 周辺（部分失敗トレランス）, TP-LGX-013 §2.2 AF1/AF3, §2.3 EF1。SPEC-LGX-002.REQ.10、NFR REL.02。DD-LGX-003 §6

### ケース 21: 不正 edge kind / パス逸脱 → GraphParseError::ValidationError（exit 1 経路）

- **期待**: いずれも `Err(GraphParseError::ValidationError(_))`。呼出側（`legixy-cli`/`legixy-check`）が exit 1 へ変換（終了コード変換は本 crate の責務外、LGX-COMPAT-001 §3）
- **境界条件**: 「自動抽出の部分失敗＝スキップ継続」と「構造/セキュリティ検証失敗＝即 Err」の経路分離

### ケース 22: コードフェンス内 `#` 行も見出しとして抽出（v3 非認識、凍結 — M-1）

- **観点出典**: TP-LGX-002 §2.5 G-VC-2（v3 互換凍結）, §2.7 G-IV-1（抽出規則）。SPEC-LGX-002.REQ.05「コードフェンス内 `#` 行の扱い（v3 実測凍結）」（v0.4.3 / M-1）

- **境界条件**: フェンス認識化はサブノード ID を変える破壊的変更（次版 SPEC 改訂事項）。本ケースは「認識しない」現状を凍結回帰として固定

### ケース 23: `extract_subnodes` は `extract_subnodes_with_levels(.., &[2,3])` の糖衣

- **観点出典**: TP-LGX-002 §2.5 G-VC-2（v3 互換）, §2.3 G-ST-1。DD-LGX-003 §3 不変条件（糖衣・拡張経路 S2-08）, §11

- **入力**: 同一 content C に対し `extract_subnodes("P", C)` と `extract_subnodes_with_levels("P", C, &[2,3])`

### ケース 24: heading_levels 内部属性 — 指定レベルのみ抽出（M-2、ID 生成に無影響）

- **観点出典**: TP-LGX-002 §2.5 G-VC-2（v3 踏襲内部属性）, §2.1 G-BV-5。SPEC-LGX-002.REQ.05「heading_levels 内部属性」（v0.4.3 / M-2）。DD-LGX-003 §3 / §11（S2-08 拡張経路）
- **分類**: Unit
- **前提**: `extract_subnodes_with_levels` の `levels` で抽出対象レベルを切り替え可能（v3 `parser.rs:121-126` が親ノードの heading_levels を消費）。本属性は ID 生成式に影響しない（生成式は heading_path のみ依存）
- **入力**: 同一 content `"## A\n### B\n"` に対し (a) `levels=&[2]` / (b) `levels=&[3]` / (c) `levels=&[2,3]`
- **期待**: (a) `A`(h2) のみ 1 件。(b) `B`(h3) のみ 1 件（ただし h3 の h2 コンテキストは current_h2 追跡に依存：`[2]` 不在でも `[3]` 単独指定時 current_h2 は更新されないため `B.heading_path==vec!["B"]`）。(c) 2 件。いずれのケースでも各見出しの ID は「その見出しの heading_path から決まる値」であり levels 指定自体は ID 値を変えない
- **境界条件**: levels 内部属性は抽出対象の絞り込みのみ。同一 heading_path → 同一 ID（生成式不変、M-2 凍結）

### ケース 25: `extract_subnodes_with_levels` の levels 空 → 空 Vec（クラッシュなし）

- **観点出典**: TP-LGX-002 §2.1 G-BV-1 周辺（境界）, §2.2 G-EH-1（クラッシュ禁止）。DD-LGX-003 §3 不変条件「`levels` 空の場合は空 Vec（クラッシュなし、REQ.10）」
- **分類**: Unit
- **前提**: `target(level)` が常に false（v3 `extractor.rs:48`）
- **入力**: `extract_subnodes_with_levels("DOC", "## A\n### B\n", &[])`
- **期待**: 空 `Vec`。panic / Err なし
- **境界条件**: levels 空集合 = 抽出対象ゼロの境界。REQ.10 クラッシュ禁止

### ケース 26: 任意バイト列・巨大/深ネスト入力でも panic しない（property、堅牢性の決定論部）

- **観点出典**: TP-LGX-002 §2.1 G-BV-4（巨大/深ネスト）, §2.2 G-EH-1（クラッシュ禁止）, §2.11 G-DM-3（Unicode エッジ）
- **分類**: Property-based（proptest）
- **生成器**: 任意 `content: String`（不正 UTF-8 境界に近い Unicode・大量 `#`・深い見出しネスト・空行を含む分布）

- **反例ハンドリング**: shrink して最小のクラッシュ/不変違反入力を記録
- **注**: 任意「バイト列」（非 UTF-8）に対するクラッシュ網羅試験と性能は **NFR-LGX-001 Fuzz/Bench へ委譲**（DD-LGX-003 §8）。本ケースは UTF-8 文字列空間での構造不変条件を property 化したもの

## 4. テスト技法選択

- **同値分割**: 見出しレベル（h1=リセット / h2,h3=抽出対象 / h4以深=無視）、fragment 判定（16文字小文字hex / それ以外）、明示名（妥当 / 文字制約違反 / 長さ違反 / 両端違反）。
- **境界値分析**: hex fragment 長 15/16/17（ケース 14）、明示名長 0/1/63/64（ケース 15/16）、見出しレベル境界 h1/h2/h3/h4 と `#######`（count>6）（ケース 5/6）、content_range の body_start/EOF 端（ケース 9）、levels 空集合（ケース 25）。
- **Property-based（proptest）**: ID 生成決定論（ケース 2）、正規化冪等性（ケース 13）、明示名長境界（ケース 16）、抽出の構造不変条件と非 panic（ケース 26）。
- **Contract**: v3 一致 ID 凍結（ケース 1）。永続化済み graph.toml との互換境界。
- **状態遷移**: h2 コンテキストの確定/リセット（h1 出現・空 h2）の状態機械（ケース 5/10）、縮退（後勝ち）・明示優先スキップの統合状態（ケース 18/19）。

## 5. テスト基盤

- 言語: Rust（`legixy-graph` crate）
- フレームワーク: cargo test（Unit/Integration）
- Property-based: proptest（ID 決定論・正規化冪等性・明示名長境界・抽出非 panic）
- Fuzz: cargo-fuzz は NFR-LGX-001 HARDEN.03 が所有（本 TS のケース 26 は proptest による決定論部のみ）
- モック: なし。`parse_graph_from_str` を用いてファイル IO を持ち込まずに統合経路を検証（DD-LGX-003 §3）。ファイル IO 経路（不在/read 失敗）は `parse_graph` + 一時ディレクトリ fixture を用いる（ケース 20）
- v3 互換 fixture: v3 `crates/lx-graph/src/subnode/`（id_gen.rs / normalizer.rs / extractor.rs）の挙動を底本とし、既知 (parent_id, heading_path)→ID ペアを移植（ケース 1）

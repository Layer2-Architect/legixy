# TP-LGX-016: UC-LGX-006「順方向探索（impact）」観点（UC レベル）

> TP は **テストケース** ではなく **観点リスト**。UC レベル TP は「ユースケースのフロー記述に問いかける質問のリスト」として書く。SPEC レベル TP（TP-LGX-005）が「仕様が答えるか」を問うのに対し、UC レベル TP は「フローが先行成果物（親 SPEC）を観察可能なステップへ忠実かつ完全に具体化しているか」を問う。

**親**: UC-LGX-006
**ステータス**: green
**最終更新**: 2026-06-13

## 1. 対象スコープ

この TP は UC-LGX-006「順方向探索（impact）」の全フロー（基本フロー Step 1〜4、代替フロー 1a/2a、事後条件）に UC レベル観点をぶつける。

- 対象: UC-LGX-006 全節（概要・アクター・事前条件・基本フロー・代替フロー・事後条件・関連仕様）
- 親 SPEC: SPEC-LGX-005（グラフ走査）REQ.01〜REQ.10
- 関連 SPEC §: LEGIXY-SPEC-001 §4（impact エンジン機能）/ §5（双方向探索）、LGX-COMPAT-001 §3/§4（`impact` の引数・終了コード凍結契約）、SPEC-LGX-002（グラフ基盤・エッジ種別・DAG）
- 委譲方針: 走査セマンティクスそのもの（BFS アルゴリズム・決定論性・visited セット・depth_map 定義・REQ.05 起点不在=空・REQ.06 循環防止）は TP-LGX-005（green 確定済）が所有する。本 TP はそれらを再検証せず、「UC-006 のフロー記述が SPEC-005 の規定を観察可能なステップとして正しく具体化しているか」のみを問う。UC-005（逆方向）と対称の構造をもつ。

## 2. 観点リスト

### 2.1 基本フロー（ステップ連鎖の整合）

- [ ] 観点 BF1: 各ステップの事後条件が後続ステップの前提を満たすか（Step1 コマンド受理 → Step2 BFS 走査開始 → Step3 深度打ち切り → Step4 結果返却 の連鎖整合。Step4「返却する」と事後条件「標準出力に表示される」の橋渡しが UC フローで観察可能か）
- [ ] 観点 BF2: アクター入力の検証タイミングの段階区分が観察可能か（段階 1 構文=`<start_ids>` パース / 段階 2 形式=graph.toml 存在 / 段階 3 意味=ノード存在チェック）。CLI 引数パース失敗（exit 2）と実行時ノード不在（2a）の差が UC フロー上で区別できるか
- [ ] 観点 BF3: 成功時事後条件の観察可能性。「走査結果が標準出力に表示される」は外部観察可能だが、**exit コードが事後条件に明示されていない**（UC-005 も同様の不記載）。EXIT 0 は SPEC-LGX-005.REQ.05 + NFR OBS.05 / USE.04 から導出可能だが、UC フロー記述として独立して確認できるか
- [ ] 観点 BF4: `start_ids`（複数形）と基本フロー記述の整合性。Step1 は `<start_ids>` と複数形だが、Step2 は「起点ノードから」と単数形で記述している。複数起点を指定した場合の BFS の起動セマンティクス（各起点から個別に走査するか、全起点を同時にキューに入れるか）が UC フロー上で観察可能か

### 2.2 代替フロー（分岐網羅）

- [ ] 観点 AF1: 分岐の網羅性。1a（`--max-depth` 未指定=全深度）/ 2a（起点ノード不在=ERROR）の 2 分岐が列挙されているが、**`--max-depth` に不正値**（負値・非数値・usize 超え）を与えた場合の失敗パスが代替フローに列挙されていない
- [ ] 観点 AF2: 2a の事後条件収束。「ERROR を報告する」と記述されているが、**exit コード・終了動作（即時終了か継続か）が明示されていない**。SPEC-LGX-005.REQ.05（起点不在=空結果・exit 0）との関係が UC フローで観察可能か。「ERROR を報告する」は exit 1 系か exit 0 か
- [ ] 観点 AF3: 代替フロー 1a（`--max-depth` 未指定）への遷移条件が「未指定」と明示されている点は問題ないが、2a の発火条件「起点ノードが graph.toml に存在しない場合」が UC-LGX-005 と比較して対称的に記述されているか確認

### 2.3 例外フロー（失敗パス）

- [ ] 観点 EF1: graph.toml が存在しない / 破損・パース不能の場合の失敗パスが代替フロー / 例外フローに列挙されているか。事前条件「graph.toml が存在する」が満たされない場合の分岐が UC の代替フローに現れていない（UC-001 の GAP-LGX-189 と同型）
- [ ] 観点 EF2: エラー時の状態が不変条件を満たすか。impact は読み取り専用（事後条件「グラフの状態は変更されない（読み取り専用操作）」と明示）で、中間状態破壊が原理的に発生しない。走査中の副作用なし前提が UC 前提と整合するか
- [ ] 観点 EF3: エラー時のユーザ通知の定義。2a「ERROR を報告する」の報告先（stdout か stderr か）・フォーマット（finding 形式か plain text か）が UC フローで観察可能か

### 2.4 アクター遷移と権限

- [ ] 観点 AT1: アクター（開発者 / Linear Agent コンテナ）の権限が一貫しているか。impact は読み取り専用で両アクターが同一権限で実行可能であることが UC で一貫しているか
- [ ] 観点 AT2: 責任境界。システムは探索と結果提示を行い、影響を受ける成果物への実際の対応はアクター責務であることが UC で明示されているか（または暗示されているか）
- [ ] 観点 AT3: impact 実行中に graph.toml が外部更新（並行アクセス）された場合の整合性前提が UC レベルで成立しているか

### 2.5 データフロー

- [ ] 観点 DF1: 入出力データの型・制約。入力（graph.toml + `<start_ids>` + `--max-depth`）→ 出力（visited 一覧 + depth_map = stdout）の分離が UC で観察可能か。**stderr への出力（打ち切り Info / エラー通知）が UC フロー上で言及されていない**
- [ ] 観点 DF2: 出力フォーマットの多態性。事後条件「走査結果が標準出力に表示される」はフォーマット（Text 既定 / `--json` 切替）を指定していない。グローバル `--json` フラグとの関係が UC フローで観察可能か
- [ ] 観点 DF3: エラー時のデータ解放保証。読み取り専用・メモリ上グラフのみで永続リソース確保解放を伴わないことが UC 前提と整合するか

### 2.6 領域固有観点（グラフ走査・順方向 impact）

- [ ] 観点 R1: impact（順方向）は engine.db（embedding・ドリフトスコア）を**使用しない**ことが UC-006 の事前条件「graph.toml が存在する」（engine.db 不要）と SPEC-LGX-005.REQ.01（chain/custom/parent_child エッジを辿るのみ）から読み取れるか。UC-005（逆方向）が engine.db を事前条件に含めるのと非対称であることが観察可能か
- [ ] 観点 R2: 終了コード契約の観察可能性。UC 事後条件に exit コードが記載されておらず、LGX-COMPAT-001 §3/§4 の凍結契約（impact: exit 0 = 正常 / exit 1 = ERROR 検出 / exit 2 = 構文誤り）が UC フローから独立して参照できない
- [ ] 観点 R3: Step3 の深度制御（`--max-depth` 指定時の打ち切り）と Alt 1a（未指定=全深度）が SPEC-LGX-005.REQ.04 の規定（打ち切り時 stderr Info 出力）を UC フローに反映しているか。REQ.04 が「`--max-depth` 指定かつ超過ノードあり → stderr Info」を要求するが、UC の代替フロー / 基本フロー Step3 にその反映がない
- [ ] 観点 R4: Step4 の出力形式（visited: 走査順・depth_map: 距離）が SPEC-LGX-005.REQ.09 の必須フィールド（ID 走査順 / タイプ / パス / 深度 / 使用エッジ）と整合しているか。「使用したエッジの情報」が UC Step4 の出力定義に欠けている
- [ ] 観点 R5: グローバル `--project-root` / `--json` / `--models-dir` を `impact` が受理することが LGX-COMPAT-001 §3 と整合するか。UC フローはグローバルオプションを言及していないが、これは UC の記述粒度として問題ないか

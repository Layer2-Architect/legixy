# TP-LGX-013: UC-LGX-003「サブノード自動抽出」観点（UC レベル）

> TP は **テストケース** ではなく **観点リスト**。UC レベル TP は「ユースケースのフロー記述に問いかける質問のリスト」として書く。SPEC レベル TP（TP-LGX-002）が「仕様が答えるか」を問うのに対し、UC レベル TP は「フローが先行成果物（親 SPEC）を観察可能なステップへ忠実かつ完全に具体化しているか」を問う。

**親**: UC-LGX-003
**ステータス**: green
**最終更新**: 2026-06-13

## 1. 対象スコープ

この TP は UC-LGX-003「サブノード自動抽出」の全フロー（基本フロー Step 1〜4、代替フロー 2a/2b/3a、事後条件）に UC レベル観点をぶつける。

- 対象: UC-LGX-003 全節（概要・アクター・事前条件・基本フロー・代替フロー・事後条件・関連不変条件）
- 親 SPEC: SPEC-LGX-002（グラフ基盤）REQ.01〜REQ.13 + LGX-EXT-001 §7.1 SUBNODE-INV-1〜6
- 関連 SPEC §: LGX-EXT-001 §3.3, §3.6, §4.5（ID 生成方式）、SPEC-LGX-004.REQ.14（SubnodeIdCollision Warning）
- 委譲方針: サブノード ID 生成式の正確性・SUBNODE-INV の検証セマンティクス・衝突検出の**規定そのもの**は TP-LGX-002（green 確定済）が所有する。本 TP はそれらを再検証せず、「UC-003 のフロー記述が SPEC-LGX-002 の規定を観察可能なステップとして正しく具体化しているか」のみを問う。

## 2. 観点リスト

### 2.1 基本フロー（ステップ連鎖の整合）

- [ ] 観点 BF1: 各ステップの事後条件が後続ステップの前提を満たすか（Step1 graph.toml 読み込み → Step2 各ノードのサブノード抽出ループ → Step3 明示サブノードのマージ → Step4 ID 一意性検証 の連鎖整合）
- [ ] 観点 BF2: Step2 のサブステップ（2a〜2h）の実行順序が観察可能か。特に 2b 見出し抽出 → 2c 正規化 → 2d 階層パス構築 → 2e ハッシュ文字列構築 → 2f ID 生成 → 2g 追加 → 2h エッジ生成 の依存チェーンが UC フローで把握できるか
- [ ] 観点 BF3: Step3 の「明示サブノードのマージ」が Step2 の自動生成処理との順序関係・結合の仕方として観察可能か。`#s:` 接頭辞サブノードが自動生成サブノードと同一グラフ構造に収容されることが事後条件に反映されているか
- [ ] 観点 BF4: Step4 の「全サブノードの ID 一意性検証（SUBNODE-INV-3）」がどの主体（システム / check コマンド）によって実施されるかが UC フローで明示されているか

### 2.2 代替フロー（分岐網羅）

- [ ] 観点 AF1: 代替フロー 2a（ファイル不在→スキップ）と 2b（h2/h3 見出しゼロ→サブノードなし）の遷移条件が明示されているか（何が真であれば分岐するか）
- [ ] 観点 AF2: 代替フロー 3a（明示 ID と自動生成 ID の衝突→明示優先）の収束先が観察可能か。SPEC-LGX-002.REQ.12「生成スキップ」の事後条件（グラフ上のノード数、生成されなかった自動生成エントリの扱い）が UC で観察可能か
- [ ] 観点 AF3: 代替フロー 2a/2b は基本フローの他ノードへの処理継続を明示しているか（ドキュメントノード 1 件のスキップが残りのループを停止させないことが分かるか）
- [ ] 観点 AF4: 「h4 以深の見出し」や「h1 見出し」（h2 コンテキストリセット）のケース——対象外見出しレベルが抽出されないことが UC の代替フローまたはフロー記述に表れているか

### 2.3 例外フロー（失敗パス）

- [ ] 観点 EF1: Step2a（ファイル読み込み）が「不在」ではなく「破損・パース不能」だった場合の失敗パスが UC フローに定義されているか（代替フロー 2a はファイル不在のみを扱い、破損・IO エラー時の失敗パスを列挙していない）【GENUINE/WEAK 暫定: 親 SPEC-LGX-002.REQ.10 はクラッシュ禁止のみ規定、UC フロー記述への明示反映が必要か否かが判断点】
- [ ] 観点 EF2: Step4 の ID 一意性検証失敗（SUBNODE-INV-3 違反）が検出された場合の失敗パスが UC フローに定義されているか。SPEC-LGX-002.REQ.12 は「生成段階ではエラーを発しない」と規定しており、Step4 での違反検出の主体と出力先が UC フローと一致するか
- [ ] 観点 EF3: エラー時の状態が不変条件を満たすか（サブノード自動抽出はグラフ構築の一部であり、途中失敗時にグラフが中途状態で残らないことが UC 前提として担保されているか）

### 2.4 アクター遷移と権限

- [ ] 観点 AT1: アクターが「システム（graph.toml 読み込み時に自動実行）」のみである前提が基本フロー全体で一貫しているか。人間が直接呼び出すサブコマンドではなく、graph.toml 読み込み処理の一部として埋め込まれていることが観察可能か
- [ ] 観点 AT2: 責任境界。サブノード抽出（自動）と明示サブノード定義（人間が graph.toml に記述）の分担が UC フローで明確か。Step3 の「マージ」が「上書き」でなく「明示優先」である権限関係が UC で観察可能か

### 2.5 データフロー

- [ ] 観点 DF1: 入出力データの流れ。入力（graph.toml 上のドキュメントノード + 対応 .md ファイル本文）→ 出力（自動生成サブノード群 + ParentChild エッジ群）の型・制約・ライフタイムが UC で観察可能か

- [ ] 観点 DF3: ハッシュ対象文字列の構築（Step2e「親ID|見出し1|見出し2」）が SPEC-LGX-002.REQ.05 の生成式と整合しているか。UC に記述された形式（`|` 区切り）がソースとして正確か

### 2.6 領域固有観点（サブノード自動抽出）

- [ ] 観点 R1: Step2c「見出しテキストの正規化」の具体内容（トリム・連続空白統合に加えて Markdown 装飾除去・全角スペース正規化・NFC が SPEC-LGX-002.REQ.06 で必須規定）が UC フローに反映されているか、またはStep2c から REQ.06 への委譲が明示されているか

- BF3（GAP-LGX-201）・AF3（GAP-LGX-203）・EF1（GAP-LGX-204）・R1（GAP-LGX-205）は WEAK 候補（親 SPEC が既に答えており、UC フロー記述への明示反映が任意か必須かの裁定が必要）。

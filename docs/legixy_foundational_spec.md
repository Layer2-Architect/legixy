## 1. 本文書の位置づけ

本文書は legixy の最上位仕様書である。legixy が扱う成果物モデル、エンジン機能、設計原則、不変条件を定義する。

本文書は legixy 開発における Single Source of Truth であり、個別の設計文書（SPEC-LGX-*、NFR-LGX-*、UC-LGX-* 等）、実装、テストは本文書と整合している必要がある。

個別機能の詳細要求は SPEC-LGX-001〜009 に、非機能要件は NFR-LGX-001 に、機能拡張（サブノード化・キャッシュ最適化）は LGX-EXT-001 / LGX-EXT-002 に分割される。

---

## 2. legixy の概要

### 2.1 位置づけ

legixy は、有向グラフを主体とするトレーサビリティエンジンである。Rust CLI として実装され、MCP サーバーとしても動作する。ステートレスに設計され、同じ入力に対して同じ出力を返す。

legixy の役割は以下である。

- 有向グラフによるトレーサビリティ管理
- 形式的検証（ID 形式、ファイル存在、グラフ整合性）および意味的整合性検証（セマンティックドリフト検出）
- 順方向・逆方向のグラフ探索
- 変更の影響度判断とコンテキスト合成

### 2.2 有向グラフ主体の設計

legixy は、有向グラフを Source of Truth とする。マトリクス（RTM, Requirements Traceability Matrix）は人間可読なビューとして有向グラフから生成される派生物である。

成果物間の関係を有向グラフで保持することにより、順方向（影響範囲特定）・逆方向（原因探索）の双方向探索が単一の構造の上で成立する。

## 3. 成果物モデル

### 3.1 成果物の種類

legixy は以下の成果物タイプを扱う。プロジェクトの性質に応じて増減可能だが、標準セットは以下である。

| ID | 成果物 | 説明 |
|----|--------|------|
| SPEC | 要求仕様書 | プロダクトの要求定義 |
| UC | ユースケース | ユーザーの操作フロー |
| RB | ロバストネス図 | UC の分析 |
| SEQ | シーケンス図 | オブジェクト間の相互作用 |
| DD | 詳細設計書 | 実装の設計図 |
| TS | テスト仕様書 | 期待動作の定義 |
| TC | テストコード | 実行可能なテスト |
| SRC | ソースコード | 実装本体 |

### 3.2 成果物間の連鎖

標準的な連鎖順序は以下である。プロジェクトにより調整可能。

```
SPEC → UC → RB → SEQ → DD → TS → TC → SRC
```

### 3.3 双方向の視点

各成果物は、開発視点と調査（不具合分析・問合せ対応）視点の両方から参照される。

| 成果物 | 開発視点 | 調査視点 |
|--------|---------|------------|
| UC | ユーザー操作の定義 | 操作問合せへの回答根拠 |
| DD | 実装設計 | 不具合原因の分析根拠 |
| TS | 正解の定義 | 期待動作の参照先 |
| SRC | 実装 | 不具合箇所の特定対象 |

同じ成果物を異なる視点から参照できることが、開発と調査を単一構造で繋ぐ。

### 3.4 有向グラフ表現

成果物間の関係は、有向グラフとして `docs/traceability/graph.toml` に記述される。

```toml
[[nodes]]
id = "UC-SP-001"
type = "UC"
path = "docs/usecases/UC-SP-001.md"

[[nodes]]
id = "DD-SP-001"
type = "DD"
path = "docs/detailed-design/DD-SP-001.md"

[[edges]]
from = "UC-SP-001"
to = "DD-SP-001"
kind = "chain"
```

このファイルが Git 管理の Source of Truth となる。マトリクス（matrix.md）は派生物として生成される。

## 4. エンジン機能

legixy が提供する主要機能は以下である。

| 機能 | 説明 | 利用場面 |
|------|------|---------|
| `check` | ID 形式、ファイル存在、グラフ整合性の検証 | CI、ローカル開発 |
| `compile_context` | ファイルパスから上流成果物を解決 | 編集前のコンテキスト取得 |
| `investigate` | 不具合から逆方向探索 + ドリフトスコアで枝刈り | 不具合調査時 |
| `impact` | 変更の下流への影響範囲特定 | 変更前の影響評価 |
| `semantic_check` | embedding による意味的整合性検証 | CI、定期検証 |
| `observe` | 観測（Observation）の記録、フィードバックループへの入力 | 編集・検証時 |

---

## 5. 双方向探索

有向グラフは順方向・逆方向の両方で探索可能である。

| 方向 | ユースケース | 主な機能 |
|------|------------|---------|
| 順方向 | 変更の影響範囲特定 | `impact`, `semantic_check` |
| 逆方向 | 不具合の原因探索 | `investigate`, `compile_context` |

---

## 6. ドリフトスコア

各エッジには意味的類似度（ドリフトスコア）が計算・保存される。

- 順方向探索時: ドリフトが閾値以下のエッジを「古くなった可能性」としてフラグ
- 逆方向探索時: ドリフトスコアで探索を枝刈りし、怪しいノードを優先

ドリフトスコアは embedding モデルによる類似度計算で算出される。

## 7. データ構造

| 場所 | 内容 | 管理 |
|------|------|------|
| `docs/traceability/graph.toml` | グラフ構造 | Git |
| `.legixy/engine.db` | embedding、ドリフトスコア、Observation | gitignore |

グラフ構造は Git 管理により変更履歴が追跡可能。計算結果とメタデータは再生成可能なキャッシュとして DB で管理される。

## 8. 実装方針

legixy は Rust で実装される。理由は以下。

- MCP 経由の頻繁な呼出しで起動時間が決定的に重要（数十 ms 以内）
- ONNX による embedding 計算で FFI オーバーヘッドが小さい
- SQLite アクセスのゼロコピー効率
- 単一バイナリでの配布（Python 依存なし）
- 型システムによる不変条件の表明

---

## 9. 設計原則

legixy の設計は以下の原則に従う。これらは個別の実装判断の基準となる。

**P-01: 判断は人間に委ねる**

legixy は候補の提示と分析を行うが、最終的な判断（承認、却下、修正内容の決定）は人間が行う。成果物の自動修正は許可されない。

**P-02: 成果物の自動修正は行わない**

legixy は成果物の修正案を提示するが、実際の修正は人間の承認を経て行われる。ドキュメントやコードがエンジンの独自判断で書き換わることはない。

**P-03: RTM はプロトコルである**

トレーサビリティマトリクス（RTM）は開発の副産物ではなく、開発と調査を繋ぐプロトコルである。両者が同じ有向グラフ／RTM を参照することで、知識の二重管理が排除される。

**P-04: 構造は一つ、視点は複数**

同じ成果物・同じグラフを、開発と調査は異なる方向から参照する。構造の統一が一気通貫を実現する。

**P-05: 構造に従って動作する**

自由な推論に依存せず、有向グラフという構造に沿って動作する。エンジンの出力は構造上の根拠を持つ。

**I-01: 単純さを優先する**

機能追加時は、複雑さを増す実装より、既存の単純な構造を活用する実装を優先する。

**I-02: 性能は起動時間を重視する**

MCP 経由での頻繁な呼び出しが発生するため、起動時間を最優先で最適化する。

## 10. 不変条件

legixy が満たすべき不変条件を以下に列挙する。これらは実装とテストにより検証される。各下位 SPEC は本節の不変条件 ID を参照する。

### 10.1 コンテキスト解決の不変条件

| ID | 内容 |
|----|------|
| CTX-INV-1 | 決定論保証: 同じ入力に対して常に同じコンテキスト結果を返す |
| CTX-INV-2 | グラフ整合性: 返される成果物はグラフ定義と矛盾しない |
| CTX-INV-3 | カスタムエッジ独立性: カスタムエッジはチェーン上流に影響しない |
| CTX-INV-4 | DAG 制約: グラフにサイクルが存在しない |
| CTX-INV-5 | 未解決エッジの許容性: エッジが存在しないノード ID を参照する場合、当該エッジを除外した部分グラフを構築し、未解決エッジを `unresolved_edges` として記録する（check で Warning 報告）。クラッシュ・例外伝播を行わず、DAG 検証から独立（循環判定に影響しない）。動的グラフで必然的に発生するダングリング参照（見出しの削除・リネーム）がシステム全体を停止させないことを保証する（VAL Finding E-01 対処、SPEC-LGX-002.REQ.11 実装） |

### 10.2 フィードバックループの不変条件

| ID | 内容 |
|----|------|
| FB-INV-1 | Observation 冪等性: 同じ内容の Observation は重複生成されない |
| FB-INV-2 | Proposal 承認原子性: 承認は 1 トランザクションで完了する |
| FB-INV-3 | 承認前不変性: pending の Proposal は context 結果に影響しない |
| FB-INV-4 | DB 不在時安全性: DB がなくてもグラフ上流は正常に返される |
| FB-INV-5 | Proposal 重複排除: 同じ semantic_key で pending の Proposal は最大 1 つ |

### 10.3 スコア管理の不変条件

| ID | 内容 |
|----|------|
| SCORE-INV-1 | ハッシュ一致保証: ノードのハッシュが一致するスコアのみ fresh とする |
| SCORE-INV-2 | モデルバージョン一致: 現在のモデルバージョンと一致するスコアのみ有効 |

### 10.4 MCP サーバーの不変条件

| ID | 内容 |
|----|------|
| MCP-INV-1 | Agent Surface 限定: MCP は compile_context, observe, get_compile_audit の 3 ツールのみ |
| MCP-INV-2 | 忠実な転送: Rust CLI 出力のフィルタリング・省略なし |
| MCP-INV-3 | Observation 重複排除: MCP 経由でも重複排除が機能する |
| MCP-INV-4 | 監査ログ完全性: compile_context の全呼び出しが記録される |

### 10.5 状態管理の不変条件

| ID | 内容 |
|----|------|
| STATE-INV-1 | ステートレス性: legixy は永続的な独自状態を持たず、graph.toml（Git 管理）と engine.db（再生成可能なキャッシュ）のみを扱う |
| STATE-INV-2 | グラフ不変性: graph.toml の変更は全て Git commit を経由する |

## 11. 制約と前提

### 11.1 前提条件

- **ドキュメントの存在**: legixy はドキュメント（成果物）を前提とする。新規開発では成果物を順次作成する。既存コードへの適用では先にドキュメント整備が必要。
- **Git の利用**: コードとドキュメント、および graph.toml が Git リポジトリで管理されている必要がある。

### 11.2 スコープ外

- **ドキュメントの新規作成**: legixy は既存のドキュメントと有向グラフを管理するが、ゼロからのドキュメント作成は行わない。
- **コードの自動修正**: legixy は修正提案を行うが、実際の修正は人間が承認して実行する。

### 11.3 既知の限界

- **AI の精度**: 意味的ドリフト検出、原因分析は完璧ではない。人間のレビューが前提である。
- **既存ドキュメントの質に依存**: 分析の品質は、成果物の質に依存する。成果物が不完全であれば分析も不完全になる。
- **スケールの限界**: 現在の設計は中規模までのプロジェクトを想定している。超大規模システム（数十万ノード）での動作は未検証。

### 11.4 変更に対する姿勢

本仕様書は Draft であり、開発の進行に応じて更新される。

- 不変条件（§10）は、よほどの理由がない限り変更しない。
- 設計原則（§9）は、長期的な一貫性のため保守的に扱う。
- 機能（§4）は、実運用のフィードバックに応じて柔軟に進化させる。

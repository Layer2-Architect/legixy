## 3. 要求事項

### SPEC-LGX-001.REQ.01: 中核エンジンとしての位置づけ

**内容:** legixy は有向グラフ主体のトレーサビリティエンジンとして、有向グラフに基づくトレーサビリティ管理・コンテキスト合成・形式/意味的検証を提供する。
**根拠:** LEGIXY-SPEC-001 §4
**検証方法:** 全 UC が本 SPEC 配下の下位 SPEC に対応すること

### SPEC-LGX-001.REQ.02: 機能カテゴリ

**内容:** legixy は以下 9 カテゴリの機能を提供する:
1. グラフ基盤（SPEC-LGX-002）
2. コンテキスト解決（SPEC-LGX-003）
3. 検証（SPEC-LGX-004）
4. グラフ走査（SPEC-LGX-005）
5. embedding とドリフト検出（SPEC-LGX-006）
6. フィードバックループ（SPEC-LGX-007）
7. マイグレーション（SPEC-LGX-008）
8. MCP サーバ（SPEC-LGX-009）
9. embedding 運用・監査（SPEC-LGX-010）

**根拠:** UC-LGX-001〜013 の網羅
**検証方法:** 各 UC が上記いずれかのカテゴリに対応すること

### SPEC-LGX-001.REQ.03: アーキテクチャ

**内容:** legixy は以下の構成を採る:
- Rust CLI（複数の機能パッケージから成るワークスペース構成。本 SPEC および下位 SPEC 中の `legixy-core`, `legixy-graph`, `legixy-db`, `legixy-cli` 等の crate 名は**例示**であり、crate 分割の正準リストは DD 段階で凍結する。初期候補は LGX-COMPAT-001 §2 の 10 crate 写像）
- TypeScript MCP サーバ（引数転送のみ、ビジネスロジックは Rust 側）

**根拠:** LEGIXY-SPEC-001 §4, LGX-EXT-001 §10、QSET-LGX-001 Q3 回答（2026-06-07）
**検証方法:** プロジェクト構造が「Rust CLI ワークスペース + TS MCP サーバ」の分離に従っていること（crate 単位の構造検証は DD 凍結後に行う）

### SPEC-LGX-001.REQ.04: データ構造の中心

**内容:** legixy の一次データ構造は**有向グラフ（graph.toml）** である。v0.1.0 のマトリクス（matrix.md）は graph.toml から生成される人間可読ビューに降格する。
**根拠:** LGX-EXT-001 §4.1, §4.4
**検証方法:** SPEC-LGX-002.REQ.01, SPEC-LGX-002.REQ.02 の充足

### SPEC-LGX-001.REQ.05: サブノード化の導入

**内容:** legixy は**サブノード**（ドキュメント内の部分単位）をグラフノードとして扱える。サブノードは自動生成（Markdown 見出しから）と明示（graph.toml で宣言）の 2 方式。
**根拠:** LGX-EXT-001 §3
**検証方法:** SPEC-LGX-002 の不変条件 SUBNODE-INV-1〜6

### SPEC-LGX-001.REQ.06: 不変条件の継承と追加

**内容:** legixy は LEGIXY-SPEC-001 §10 で定義される以下の不変条件を維持する:
- CTX-INV-1〜4（コンテキスト合成の不変条件）
- MCP-INV-1〜4（MCP サーバの不変条件）

加えて、legixy で以下を新規追加する:
- SUBNODE-INV-1〜6（サブノード化に伴う不変条件、LGX-EXT-001 §7）

**根拠:** LEGIXY-SPEC-001 §10, LGX-EXT-001 §7
**検証方法:** 各 SPEC-LGX-002〜010 の「不変条件との関係」節

### SPEC-LGX-001.REQ.07: 後方互換性

**内容:** legixy は v0.1.0 のプロジェクトデータ（engine.db、matrix 形式 .legixy.toml）を自動移行する。移行は非破壊的でなければならない。
**根拠:** LGX-EXT-001 §4.4, §8.2
**検証方法:** SPEC-LGX-008（マイグレーション）

### SPEC-LGX-001.REQ.08: Admin Surface と Agent Surface の分離

**内容:** legixy は以下 2 種の Surface を明確に分離する:
- **Admin Surface**: CLI コマンド（check, feedback, analyze, proposals, approve, reject, embed, init, migrate, snapshot, drift, report, calibrate, refresh-subnodes 等）
- **Agent Surface**: MCP ツール（compile_context, observe, get_compile_audit の 3 つのみ）

承認操作（approve, reject, analyze, feedback）は Admin Surface にのみ存在し、Agent（Claude Code）から呼び出してはならない。

**根拠:** CLAUDE.md 絶対ルール5, LEGIXY-SPEC-001 §2, LGX-EXT-001 §6
**検証方法:** MCP サーバが 3 ツールのみ公開していること（SPEC-LGX-009）

## 4. 不変条件との関係

### 4.1 不変条件 × SPEC 責任マトリクス

全 28 件の不変条件について、各下位 SPEC における責任の種類を示す。LEGIXY-SPEC-001 §10 の 17 件、LGX-EXT-001 §7.2 の 6 件（SUBNODE-INV-1〜6）、LGX-EXT-002 §5.2 の 4 件（CACHE-INV-1〜4）、プロジェクト独自の 1 件（CTX-INV-5）で構成される。

**凡例:**
- **実装**: 当該 SPEC の REQ が不変条件を実現する具体的機構を規定する
- **検証**: 当該 SPEC の REQ が不変条件違反の検出手段を規定する
- **関連**: 直接の実装・検証はないが、REQ が当該不変条件に関与する（継承宣言・前提・補強等）
- **-**: 関与しない

> **検証要否分類（GAP-LGX-003 対応）:** 本表で「検証」owner を持たない不変条件は、§4.4 の検証 owner 要否ポリシーにより **C（construction-time guarantee、検出不要）** または **D（needs-detection、検出は TS/TC 層に委譲）** のいずれかに分類される。

### 4.2 カテゴリ別の責任所在

- **CTX-INV-1〜5（コンテキスト合成の不変条件）**: 主に SPEC-LGX-003（コンテキスト解決）と SPEC-LGX-002（グラフ基盤）で実装。CTX-INV-1 の決定論性は走査（SPEC-LGX-005）でも実装される。CTX-INV-2/4/5 の違反検出は SPEC-LGX-004（検証）が担う。CTX-INV-5（未解決エッジの許容性）は legixy で新規追加された「グラフが一時的に不整合な状態での動作継続性」に関する不変条件。
- **FB-INV-1〜5（フィードバックループの不変条件）**: 主に SPEC-LGX-007（フィードバックループ）で実装。FB-INV-3（承認前不変性）は SPEC-LGX-003（コンテキスト解決）でも「pending proposal が context 結果に影響しない」という形で実装される。FB-INV-4（DB 不在時安全性）は SPEC-LGX-003 主導で、グラフ情報のみでの動作継続を保証する。
- **SCORE-INV-1〜2（スコア管理の不変条件）**: SPEC-LGX-006（embedding とドリフト検出）で実装。content_hash / model_version による fresh 判定ロジックに対応。SPEC-LGX-010（embedding 運用・監査）は bulk similarity API の consumer として SCORE-INV-1/2 に関連し、`drift` コマンドの次元不一致 Error が SCORE-INV-2 違反状態の検出を担う。
- **MCP-INV-1〜4（MCP サーバーの不変条件）**: Surface 面は SPEC-LGX-009（MCP サーバ）、観測・監査の実装は SPEC-LGX-007（フィードバックループ）。監査ログの記録自体は SPEC-LGX-003 が担う。
- **STATE-INV-1〜2（状態管理の不変条件）**: STATE-INV-1（ステートレス性）は SPEC-LGX-009（MCP サーバの短命プロセス）で実装。STATE-INV-2（graph.toml 変更は Git 経由）は SPEC-LGX-008（マイグレーション）で運用ガイダンスとして規定される。
- **SUBNODE-INV-1〜6（サブノード化の不変条件）**: 実装は SPEC-LGX-002（グラフ基盤）、検出は SPEC-LGX-004（検証）。INV-5（ID 決定性）のみ生成ロジック側で保証されるため検証対象外。
- **CACHE-INV-1〜4（Prompt Caching と MCP Result Persistence の不変条件、LGX-EXT-002）**: CACHE-INV-1（バイト単位決定論）、CACHE-INV-2（セクション配置順序）、CACHE-INV-3（大規模返却エラー）は SPEC-LGX-003 で実装。CACHE-INV-4（メタデータ付与忠実性）は SPEC-LGX-009 で実装され、MCP-INV-2（忠実な転送）を拡張する形で `_meta` フィールドの付与が本文改変とならないことを保証する。
- **MCP-INV-1〜4（MCP サーバの不変条件）**: Surface 面は SPEC-LGX-009（MCP サーバ）、観測・監査の実装は SPEC-LGX-007（フィードバックループ）。監査ログの記録自体は SPEC-LGX-003 が担う。
- **SUBNODE-INV-1〜6（サブノード化の不変条件）**: 実装は SPEC-LGX-002（グラフ基盤）、検出は SPEC-LGX-004（検証）。INV-5（ID 決定性）のみ生成ロジック側で保証されるため検証対象外。

### 4.3 各下位 SPEC との関係

各下位 SPEC（002〜010）の §4 にも、本 SPEC で担当する不変条件のサブセット表を掲載する。本表と下位 SPEC 表の整合性は人間査読で維持する（自動検証ツールは将来課題）。正準ソース宣言と同一コミット更新規律は §7.3 が規定する。

> **査読確認記録（GAP-LGX-002 対応）:** 2026-06-10 の査読により、§4.1 マトリクスと下位 SPEC（002〜010）§4 サブセット表の整合を確認済み。以降の整合維持は §7.3 の同一コミット更新規律に従う。

### 4.4 検証 owner 要否ポリシー（GAP-LGX-003 対応）

§4.1 で「検証」owner を持たない不変条件について、その不在が**設計判断**であることを以下の 2 分類で明文化する。これにより検証網羅性の主張と RPC（責務保存率検査）の母数の根拠を与える。

**分類:**
- **C（construction-time guarantee）**: 構成的に保証され、違反状態が実行時に出現し得ないため、専用の検出機構を要しない。
- **D（needs-detection）**: 違反は出現し得るが、検出は SPEC レベルの検証カテゴリ（SPEC-LGX-004）ではなく TS/TC 層（テスト仕様・テストコード）に委譲する。

| 不変条件 | 分類 | 根拠 |
|---------|:---:|------|
| CTX-INV-1（決定論保証） | D | 出力順序の決定性はテストで検出（走査順・出力のスナップショット比較） |
| CTX-INV-3（カスタムエッジ独立性） | C | chain 解決と custom 解決が別経路として構成され、混入経路が存在しない |
| FB-INV-1〜5（フィードバックループ） | D | DB 状態遷移の検証は TS/TC 層（状態遷移テスト・並行性テスト） |
| SCORE-INV-1（ハッシュ一致保証） | C | fresh 判定が content_hash 照合そのものとして構成される |
| MCP-INV-1（Agent Surface 限定） | C | MCP サーバが 3 ツールのみを公開する構成で保証（SPEC-LGX-009） |
| MCP-INV-2/3/4（転送忠実性・重複排除・監査） | D | 転送・記録の挙動検証は TS/TC 層 |
| STATE-INV-1/2（ステートレス性・Git 経由） | C | 短命プロセス構成・graph.toml 書込経路の構成で保証 |
| SUBNODE-INV-5（ID 決定性） | C | 生成ロジックが決定論的関数として構成される（§4.2 記載どおり検証対象外） |
| CACHE-INV-1〜4（キャッシュ決定論ほか） | D | バイト単位決定論・配置順序はテストで検出 |

## 5. v0.1.0 からの主要差分

| 項目 | v0.1.0 | legixy |
|------|--------|----|
| 一次データ構造 | マトリクス（matrix.md） | 有向グラフ（graph.toml） |
| 粒度 | ドキュメント単位 | サブノード単位（任意） |
| サブノード生成 | なし | h2/h3 見出しから自動 or 明示宣言 |
| DAG 検証 | なし（マトリクスのため） | Kahn's algorithm |
| compile_context 粒度制御 | なし | granularity オプション |

## 7. 変更ポリシー（GAP-LGX-004 対応）

### 7.1 変更主体と承認

本 SPEC を含む全 SPEC-LGX-NNN の変更は**人間承認を必要とする**（CLAUDE.md ハードルール 1）。AI は変更を提案し（`docs/spec-change-proposals/`）、人間が決定する。承認なき SPEC 本文の変更は無効である。

### 7.2 下位 SPEC の partition 拡張に伴う同期

下位 SPEC（002〜010）の機能 partition が拡張される場合（新カテゴリ追加・UC 写像変更等）:

- **同期トリガ**: 当該下位 SPEC の前段ループが FCR.frontend_status = ACCEPTED に到達した時点で、本 SPEC（REQ.02 カテゴリ一覧・ヘッダ表対応 UC・§4.1 マトリクス）への同期要否を判定する。
- **同期期限**: 遅くとも UC フェーズ着手時（SPP-LGX-001 次反復）までに反映する。即時反映できない場合は REQ.02 予約注記の形式で**既知の予約状態**として明示する。

### 7.3 不変条件マトリクスの正準ソース宣言

- 本 SPEC §4.1 の不変条件 × SPEC 責任マトリクスを**正準（canonical source）**とする。
- 下位 SPEC §4 のサブセット表は §4.1 の射影であり、両者が矛盾する場合は §4.1 が優先する。
- §4.1 を変更するコミットは、影響を受ける下位 SPEC §4 のサブセット表を**同一コミットで更新しなければならない**（旧 GAP-LGX-002 の同期規律を本項が吸収）。

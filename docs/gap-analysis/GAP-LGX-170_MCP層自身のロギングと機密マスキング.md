# GAP-LGX-170: MCP サーバ自身のロギング水準と機密マスキング方針が未定義

**親 TP**: TP-LGX-009
**観点出典**: TP-LGX-009 §2.9 観点 44（ロギング・観測性）
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08
**敵対的精査（2026-06-09）**: minor (verification: low-value, 人間判断で drop 可)。NFR-LGX-001 SEC.05 のマスキング義務は Contextual Retrieval 用 API キー（Rust CLI 側経路）が対象で、MCP→Agent 転送経路とは別。MCP-INV-2（忠実転送）は stderr 本文の改変禁止を含意するため、MCP 層での stderr マスキングはむしろ INV 違反になり、方針は実質「マスキングせず忠実転送」と決まる。MCP サーバ自身の診断ログ有無は thin forwarder の DD 実装事項。残る明文化は一行クラリフィケーションで決定相関性は低い。

## 1. 観点

MCP-INV-4（監査ログ完全性）の実装本体は SPEC-LGX-003/007 に委譲されるが、**MCP サーバプロセス自身の診断ログ**（呼出しトレース、子プロセス起動失敗、タイムアウト等の運用ログ）の有無・水準・出力先と、ログに混入しうる機密情報（stderr 経由のファイルパス、トークン、環境変数値）のマスキング方針が未定義。

## 2. 現状の SPEC

SPEC-LGX-009 §4 は **MCP-INV-4 を SPEC-LGX-003 REQ.07 / SPEC-LGX-007 REQ.06 に委譲** し、REQ.03 で CLI 出力を改変しないことで監査情報が保全されると述べるが、これは **Rust CLI 側の監査ログ（context_log 等）** の話。**MCP サーバプロセス自身が自身の運用ログを出すか、出すなら機密マスキング（NFR-LGX-001 SEC.05 のクレデンシャルマスキング義務に相当）を行うか** は未定義。core-perspectives §ロギング・観測性「機密情報がログに混入していないか（PII, secret, token）」に対応する観点が空白。

## 3. 期待される情報

SPEC または委譲先に追加されるべき記述:

- MCP サーバ自身の運用ログの有無・水準・出力先（stderr / ファイル）
- ログ・エラー応答に CLI stderr を含める際の機密マスキング方針（NFR-LGX-001 SEC.05 との整合）、または「マスキングは行わず stderr を忠実転送する（MCP-INV-2 優先）」という確定
- 環境変数（バイナリパス等）の値がログに出ないことの保証

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- 全 UC（MCP 経由）: 障害解析時のログ可用性と機密保護のトレードオフが未定義
- 下流の DD/TS: ロギング層の有無と機密マスキングのテストが書けない
- NFR-LGX-001 SEC.05（クレデンシャルマスキング義務）との整合

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-009 v0.6.0（人間裁定 fix・承認 2026-06-10）: REQ.03 にロギング・マスキング方針を確定 — stderr はマスキングせず忠実転送（MCP-INV-2 優先）、SEC.05 マスキングは Rust CLI 側責務、MCP 自身の診断ログは stderr 最小限（詳細 DD）、環境変数値の非ログ出力を保証。

# GAP-LGX-195: 大規模返却エラー（500,000 文字超過）の例外フローが UC に存在しない

**親 TP**: TP-LGX-012
**観点**: §2.3 EF1
**ステータス**: closed (2026-06-13)
**起票日**: 2026-06-13
**分類（暫定）**: GENUINE

## 1. 観点

SPEC-LGX-003.REQ.13 は返却本文が 500,000 文字を超えた場合に「切り捨て・要約をせず明示的エラーと粒度切替提案を返す」ことを規定する（CACHE-INV-3）。この失敗パスは Step6（ContextResult 返却）で発生しうる重要な分岐であるが、UC-LGX-002 の代替フロー・例外フローにこのケースが一切記述されていない。Step6 が正常終了するか・エラーを返すかの分岐が UC フローから観察不能。

## 2. 現状の UC / SPEC

- **UC-LGX-002**: 代替フロー 2a/3a/4-A/4-B/4-C のみ。サイズ超過エラーへの言及なし
- **SPEC-LGX-003.REQ.13**: 「500,000 文字を超える場合、compile_context はエラーを返却する。本文の切り捨てや自動要約は行わない。エラー返却形式: `Error: compile_context result exceeds 500,000 characters. ...`」
- **TP-LGX-003 E-01（GREEN 確定済）**: SPEC レベルでは REQ.13 が答えている

## 3. 推奨対応（人間裁定）

**(A) UC-LGX-002 代替フロー（または例外フロー節を新設して）に追記する**
例: 「6-A. Step6 での返却本文が 500,000 文字を超える場合、システムはエラーを返却し、粒度変更（`--granularity subnode` など）を提案する。切り捨て・要約は行わない（SPEC-LGX-003.REQ.13）」

**(B) drop（委譲容認）**
サイズ超過エラーの詳細（閾値・エラー形式・提案文）は SPEC-LGX-003.REQ.13 が規定しており、UC フロー記述に反映せずとも SPEC 委譲で設計上の空白はない。UC の役割はフロー「骨格」の記述であり、エラーケースの詳細は SPEC 所管とみなす。

## 4. 影響範囲

- UC-LGX-002 代替フロー（追記の場合）
- TP-LGX-012 EF1（解消後 GREEN 化）

## 5. 解消（2026-06-13）

敵対的精査裁定: **方針B（委譲容認）**。本観点は親 SPEC（対応 TP[SPEC] green 確定済）が回答しており、UC フローへの明示列挙は任意。GREEN-by-delegation として close。詳細: docs/spec-change-proposals/2026-06-13_uc-gap-adversarial-review.md §F。

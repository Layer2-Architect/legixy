# GAP-LGX-157: migrate の `--from`/`--to` の意味が SPEC と互換契約で矛盾

**親 TP**: TP-LGX-008
**観点出典**: TP-LGX-008 §2.6 観点 V-9
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

`legixy migrate` の `--from` / `--to` 引数の意味が、境界契約 LGX-COMPAT-001 と SPEC-LGX-008 で食い違っている。LGX-COMPAT-001 §4 行2 は **PATH（ファイル/ディレクトリパス）** とし `--from` を必須・`--to` 既定を `--project-root` とする。一方 SPEC-LGX-008 REQ.06 は **バージョン文字列**（`--from v0.1.0 --to legixy`）の例を示す。CLI 境界契約は凍結済（ハードルール 7）であるため、この矛盾は SPEC 側の修正が必要な可能性が高い。

## 2. 現状の SPEC / UC

SPEC-LGX-008 §3 REQ.06 は **`--from v0.1.0 --to legixy`: バージョン明示** と記述。LGX-COMPAT-001 §4 行2 は **`--from <PATH>`（必須）, `[--to <PATH>]`** と記述し、備考に「`--to` 既定は `--project-root`」。両者が同じフラグに異なる意味（バージョン文字列 vs パス）を与えており、**どちらが正準か** が未確定。

## 3. 期待される情報

SPEC の修正（人間承認が必要・ハードルール 1, 7）として確定すべき事項:

- `--from` / `--to` は PATH を取る（LGX-COMPAT-001 凍結契約を正準とする）ことの確認
- バージョンの指定が別途必要なら別フラグ（例 `--from-version`）を立てるか、自動検出（REQ.09）に委ねることの明示
- REQ.06 の記述例を契約に合わせて修正（バージョン文字列例の除去または別フラグ化）

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- 下流 DD / TS / TC: migrate の CLI シグネチャが確定せず実装・テストが書けない
- LGX-COMPAT-001（凍結境界契約）との整合性違反のまま下流に伝播する
- 他の GAP との依存: GAP-LGX-154（バージョン判定は自動検出か引数指定か）

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-008 v0.6.0（人間裁定 2026-06-10・案A）: REQ.06 を凍結契約どおり --from <PATH> 必須 / --to <PATH> 既定 --project-root / --dry-run / --format に確定、バージョンは REQ.09 自動検出へ移管。凍結契約無変更（ハードルール 7 維持）。v3 実測で from: PathBuf を確認（main.rs:66-73）— 契約・実装の両事実と整合。ADR-LGX-001。

## 6. 関連 ADR

ADR 候補: 「migrate の --from/--to を PATH 意味に確定し、バージョンは自動検出に委ねる」境界契約整合判断。

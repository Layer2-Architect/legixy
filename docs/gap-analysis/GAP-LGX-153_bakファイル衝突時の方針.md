# GAP-LGX-153: `.bak` バックアップが既に存在する場合の方針が未定義

**親 TP**: TP-LGX-008
**観点出典**: TP-LGX-008 §2.5 観点 P-5
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

REQ.02 / REQ.13 は失敗時/移行時に原本を `.bak` で退避するが、**同名の `.bak` が既に存在する場合（連続失敗、過去の migrate、再実行）に、上書きするのか・連番を振るのか・拒否するのか** が未定義。素朴に上書きすると、最初の失敗時に取った原本のバックアップを 2 回目の失敗時に消してしまい、原本保護が破れる。

## 2. 現状の SPEC / UC

SPEC-LGX-008 §3 REQ.02 で **「失敗時は元ファイルを保持（`.bak` 等で原本を保存）」**、REQ.13 で **「旧ファイルは `.bak` 退避」** に触れているが、**`.bak` 名の衝突解決（上書き/連番/拒否）** は未定義。LGX-COMPAT-001 §4 行1 / 行9 は `.bak` / `.refresh-bak.{epoch}` の命名例を示すが、migrate の `.bak` 衝突は規定されていない。

## 3. 期待される情報

SPEC または UC に追加されるべき記述:

- `.bak` 既存時の解決方針（タイムスタンプ/epoch 付き連番が望ましい、`refresh-subnodes` の `.refresh-bak.{epoch}` 慣行と整合）
- 上書きする場合でも「最古の原本」を失わない保証、または上書き拒否
- 複数回失敗・再実行時にバックアップが累積/ローテーションする方針

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- 下流 TS / TC: 連続失敗・再実行時のバックアップ保全テストの期待値が書けない
- REQ.02（非破壊性）の中核保証が崩れるリスク
- 他の GAP との依存: GAP-LGX-143（init `--force` 退避）, GAP-LGX-156（ロールバック）

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-008 v0.6.0（人間承認 2026-06-10）: REQ.02a 新設 — 退避名 <元名>.bak.{epoch}（固定 .bak 上書き禁止・同一秒は連番・累積保持・機械削除なし）。refresh-subnodes 慣行と統一し init --force 退避にも適用。

# GAP-LGX-045: `--sections` の不正形式入力（空トークン・空白のみ・重複 ID）の扱い

**親 TP**: TP-LGX-003
**観点出典**: TP-LGX-003 §2.7 観点 I-04
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08
**重大度**: minor (verification: low-value, 人間判断で drop 可)

> **敵対的精査（2026-06-09）所見**: 挙動は既存 REQ から概ね導出可能。trim 後に空となるトークン（空トークン・空白のみ）は REQ.16「存在しない ID は単に除外」と同一帰結（マッチする subnode ID が無い）。重複 ID は REQ.11（subnode = 親 ID 辞書順 + アンカー出現順）+ CACHE-INV-1（決定論）が事実上 set セマンティクスを強制する（同一 subnode は graph 上の固定位置に 1 度しか出現し得ない）。残る「空トークンを明示エラーにするか黙殺か」は瑣末な設計選択であり、人間判断で drop してよい。維持は否定しないが意思決定価値は低い。

## 1. 観点

`--sections` はコンマ区切りのサブノード ID を受け付けるが、空トークンを含む文字列（`"a,,b"`）、空白のみのトークン（`"a, ,b"`）、重複 ID（`"a,a"`）といった不正形式・縮退入力の扱いが定義されていない。これらが黙殺されるのか、エラーになるのか、重複が重複返却を生むのかが不明。

## 2. 現状の SPEC / UC

- SPEC-LGX-003 REQ.16 は次を規定する: コンマ区切り文字列、各 ID は graph.toml 上の subnode ノード ID と**完全一致**、前後空白は許容、存在しない ID は単に除外（エラーにしない）、親ドキュメント ID 指定は除外 + Info 診断。
- 規定されていないのは:
  - 空トークン（コンマ連続 `,,` や先頭/末尾コンマ）の扱い。
  - 空白のみのトークン（trim 後に空文字列になるもの）の扱い。
  - 同一 ID の重複指定が upstream に重複出現するのか 1 回に dedup されるのか。重複は決定論的整列（REQ.11 の subnode = 親 ID 辞書順 + アンカー出現順）との整合に影響する。
- SPEC-LGX-009 REQ.15 は MCP 側で `sections` を `string`(min1) と zod 制約するが、これは**文字列全体**の最小長であり、コンマ分割後の個々のトークンの空判定は担保しない。

## 3. 期待される情報

SPEC-LGX-003 REQ.16 に以下を追加:

- 空トークン / 空白のみトークンの扱い: trim 後に空となるトークンは単に無視する（存在しない ID と同様にエラーにしない）のか、入力不正としてエラーにするのかを明記。
- 重複 ID の扱い: dedup して 1 回のみ返すのか、指定回数だけ返すのか。決定論性（REQ.11 / CACHE-INV-1）を保つため dedup 後の整列規則を明記。
- 全トークンが無効/空になった場合に結果が空 upstream で正常終了する旨（REQ.16 末尾「結果が空になりうる」との整合）。

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- UC-LGX-004（粒度別テスト拡充）: `--sections` の異常系/縮退系フローが書けない。
- 下流 DD: sections パース処理（split → trim → 空除去 → dedup）の仕様が確定しない。
- TS/TC: TS-LGX-002 §15 T-CC-SECTIONS-001 の縮退入力に対する期待値が定まらない。
- CACHE-INV-1: 重複指定が重複返却を生むと、入力の表記揺れで出力バイト列が変わり決定論が崩れる。

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-003 v0.7.1（人間裁定 fix・承認 2026-06-10）: REQ.16 に縮退入力規則を追記 — trim 後空トークンは無視（存在しない ID と同経路）、重複 ID は dedup（set セマンティクス）し REQ.11 整列に従う（CACHE-INV-1 保全）、全トークン無効は空 upstream で exit 0。

## 6. 関連 ADR

該当なし（パース仕様の明確化レベル）。

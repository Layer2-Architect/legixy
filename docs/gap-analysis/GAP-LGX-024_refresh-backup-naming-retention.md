# GAP-LGX-024: バックアップファイルの命名衝突・保持/掃除方針が未定義

**親 TP**: TP-LGX-002
**観点出典**: TP-LGX-002 §2.6 観点 G-PS-2
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08
**severity**: minor (verification: low-value, 人間判断で drop 可)

> **敵対的精査パス（2026-06-09）**: WEAK_OR_PADDED と判定。技術的には未回答だが decision-relevant 性が低い。(a) 同一秒 2 回 --apply 衝突は単独開発者環境（NFR-LGX-001.SEC.08 / §11 非目標）では非現実的で、衝突時もバックアップ対象は同一ファイルの直近状態。(b) 保持・掃除・.gitignore はバックアップ運用ガイダンスであり SPEC 要求事項ではない。本 GAP 自身が「v3 実測どおり上書き許容と明示するだけでも可」とする通り軽微。人間判断で drop 可。維持する場合の severity は minor。

## 1. 観点

バックアップ名は `graph.toml.refresh-bak.{unix epoch 秒}` で秒単位の粒度を持つ。同一秒内に refresh-subnodes --apply を 2 回実行した場合（スクリプト連続実行・冪等再試行等）、同名バックアップへの上書きで直前のバックアップが失われうる。また、バックアップが蓄積し続けた場合の保持・掃除方針が未定義で、リポジトリ汚染や .gitignore 対象かどうかも不明。

## 2. 現状の SPEC / UC

SPEC-LGX-002 REQ.13 は「書き換え前に `graph.toml.refresh-bak.{unix epoch 秒}` を作成する」とのみ規定する。以下が未定義:

- **命名衝突**: 同一秒内の複数回実行で同名となった場合（上書き / 連番付与 / 中止のいずれか）
- **保持・掃除方針**: 古いバックアップの自動削除・保持件数の上限・手動掃除前提か
- **配置とバージョン管理**: バックアップを project_root のどこに置くか、.gitignore 対象か（VCS 汚染防止）

## 3. 期待される情報

SPEC-LGX-002 REQ.13 に追加されるべき記述:

- 同一秒内の命名衝突回避（例: 既存同名があれば連番/ナノ秒を付与、または既存を上書きしない）。または v3 実測どおり「上書きを許容」と明示し責任分担を確定
- バックアップの保持・掃除方針（自動削除しない＝手動掃除前提を明示するだけでも可）
- バックアップの配置場所と VCS 追跡可否のガイダンス

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- UC-LGX-001 / UC-LGX-003: バックアップからの復旧手順・掃除運用を記述できない
- 下流 TS: バックアップ生成 E2E テストで「同一秒 2 回実行」の期待挙動を定義できない
- 関連: GAP-LGX-023（書き換え atomicity）と同一 REQ.13 上の隣接論点

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-002 v0.4.2（人間裁定 fix・承認 2026-06-10）: REQ.13 に命名衝突回避（同一秒は連番・既存上書き禁止、SPEC-LGX-008.REQ.02a と統一規約）・累積保持（機械削除なし・手動掃除前提）・同一ディレクトリ配置・VCS 追跡対象外推奨（.gitignore ガイダンス）を追記。

## 6. 関連 ADR

該当なし（v3 実測挙動の明文化、または軽微な命名規則追加で解決可能と想定）。

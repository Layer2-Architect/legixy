# GAP-LGX-190: UC-LGX-001 が Step4 の部分失敗継続をフローに反映していない

**親 TP**: TP-LGX-011
**観点**: 2.3 EF4（部分成功時の扱い: ロールバック / 部分コミット / リトライ）
**ステータス**: closed (2026-06-13)
**起票日**: 2026-06-13
**分類（暫定）**: WEAK（親 SPEC で解決済。UC への明示反映は任意の可能性）

## 1. 観点

UC レベル観点「部分成功時の扱い」を UC-LGX-001 基本フロー Step4（形式検証 a〜i）にぶつけた。複数ファイル・複数検査を跨ぐ Step4 で、1 件の読込失敗が全体を中断させるか継続するかは事後条件（出力される CheckReport の範囲）を左右する。

## 2. 現状の UC / SPEC

- UC-LGX-001 Step4 は a〜i の各形式検証を列挙し、Step5 で結果を出力すると記述する。
- しかし「一部ファイルの読込失敗時に**他の検査を継続**し、失敗を ERROR finding として CheckReport に含める」挙動（部分失敗継続）がフロー記述に現れていない。
- SPEC-LGX-004.REQ.05「一部ファイル読込失敗時に他チェックを継続（読込失敗は Error として CheckReport に記録）」で**規定済**（TP-LGX-004 観点 E1/E2 で GREEN）。

振る舞いは親 SPEC で確定済だが、UC-001 のフロー記述（部分成功の事後条件）には反映されていない。

## 3. 推奨対応（人間裁定）

以下のいずれか:

- **(A) UC へ追記**: 基本フロー Step4 に注記「いずれかのファイル読込が失敗しても他の検査を継続し、失敗を ERROR finding として CheckReport に記録する（SPEC-LGX-004.REQ.05）」を追加。または例外フロー `4-E`「ファイル読込失敗時: 当該ファイルを ERROR finding に記録し、残りの検査を継続。Step5 で部分 CheckReport を出力、exit 1」を追加。→ 部分成功の事後条件が観察可能化。
- **(B) drop（委譲容認）**: Step4 の検査実行モデル（部分失敗継続）は SPEC-LGX-004.REQ.05 へ委譲する設計と認め、UC は変更しない。

WEAK 候補。フロー妥当性は人間レビュー領域。

## 4. 影響範囲

- close されないと TP-LGX-011 が green にならず、UC-001 起点の下流に進めない。
- 振る舞い自体は SPEC-LGX-004.REQ.05 で確定済のため、下流実装の正しさには影響しない（記述完全性の問題）。GAP-LGX-189 と同種（UC フロー記述の粒度方針）であり、一括裁定が可能。

## 5. 解消（2026-06-13）

敵対的精査裁定: **方針B（委譲容認）**。本観点は親 SPEC（対応 TP[SPEC] green 確定済）が回答しており、UC フローへの明示列挙は任意。GREEN-by-delegation として close。詳細: docs/spec-change-proposals/2026-06-13_uc-gap-adversarial-review.md §F。

# GAP-LGX-142: 大規模 v0.1.0 入力に対する所要時間・メモリ・サイズ境界が未定義

**親 TP**: TP-LGX-008
**観点出典**: TP-LGX-008 §2.1 観点 B-3
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

数万ノード規模の matrix / 大容量 engine.db を migrate する際の所要時間予算・メモリ上限・許容サイズが定義されておらず、大規模プロジェクトでの実用性・タイムアウト・OOM の境界が判定できない。

## 2. 現状の SPEC / UC

SPEC-LGX-008 §3 REQ.03 は変換ルールを規定するが **入力規模に対する性能予算・上限** に触れていない。NFR-LGX-001 の PERF.* は check 等の通常操作の予算を規定するが、migrate（初回一時操作）の予算は対象に含まれていない（PERF.07 は WAL モード義務のみ）。

## 3. 期待される情報

SPEC または NFR に追加されるべき記述:

- migrate の所要時間予算（または「初回一時操作のため予算対象外」とする明示）
- 想定最大ノード数 / engine.db サイズと、それを超えた場合の挙動（ストリーミング処理か全ロードか）
- メモリ消費の上限方針

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- 下流 TS / TC: 大規模入力の性能・スケール試験の合否基準が書けない
- NFR との整合: migrate を PERF 予算の対象外とするなら NFR 側にも明記が必要
- 他の GAP との依存: 並行性 GAP-LGX-150（migrate 中の長時間ロック）と関連

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-008 v0.7.0（人間裁定 fix・承認 2026-06-10）: REQ.03 に性能予算方針を確定 — migrate は初回一時操作のため PERF 予算対象外（NFR 側明記は NFR 改訂イベントへ申し送り）、サイズ上限なし・全ロード許容（SEC.04 の OOM 防御は別途有効）。

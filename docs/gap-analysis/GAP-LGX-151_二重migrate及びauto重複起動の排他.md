# GAP-LGX-151: 二重 migrate / auto 重複起動時の排他制御が未定義

**親 TP**: TP-LGX-008
**観点出典**: TP-LGX-008 §2.4 観点 C-2, C-3
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

(1) ユーザが 2 つの `legixy migrate` を同時起動した場合、(2) auto=true で複数の legixy コマンドが同時に初回起動した場合、それぞれ変換処理が二重に走る可能性がある。冪等性（REQ.02）は逐次再実行の安全を保証するが、**同時並行起動時の排他（ロックファイル・DB ロック）** は別問題であり未定義。

## 2. 現状の SPEC / UC

SPEC-LGX-008 §3 REQ.02 は **「再実行可能（冪等）」** に触れるが、これは逐次再実行を指し、**同時並行の二重実行に対する排他** は未定義。REQ.01 の auto 経路でも複数コマンドの同時初回起動の競合は未定義。

## 3. 期待される情報

SPEC または UC に追加されるべき記述:

- migrate / auto 変換の排他機構（ロックファイル、SQLite の排他ロック、PID ロック等）
- 二重起動を検出した側の挙動（待機 / 即時 Error / 既に完了として no-op）
- 排他取得失敗時のユーザ向けメッセージ

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- 下流 TS / TC: 同時起動テストの期待値が書けない
- 他の GAP との依存: GAP-LGX-149（auto 初回識別）, GAP-LGX-150（同時アクセス競合）

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-008 v0.7.0（人間裁定 fix・承認 2026-06-10）: REQ.02 に二重実行排他を確定 — 専用ロック機構なし、SQLite 書込ロックが事実上の排他（敗者は busy_timeout 超過で Error exit 1 + 並行実行示唆メッセージ）。auto 重複起動も同様。ADR-LGX-011。

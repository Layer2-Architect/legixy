# GAP-LGX-211: UC-LGX-004 事前条件「サブノード存在」の検証タイミングがフローに観察可能でない

**親 TP**: TP-LGX-014
**観点**: §2.1 BF1（事前条件の検証タイミング）
**ステータス**: closed (2026-06-13)
**起票日**: 2026-06-13
**分類（暫定）**: GENUINE

## 1. 観点

UC-LGX-004 の事前条件「上流成果物にサブノードが存在する（自動抽出 or 明示定義）」が、基本フロー内でどのタイミングに検証されるかが観察可能でない。SPEC-LGX-003.REQ.08 および代替フロー 3a（fallback）の設計からは、サブノード有無の判定は Step 3 の実行時に動的に行われることが読み取れるが、UC のフロー記述（基本フロー・代替フロー・例外フロー）にその判定タイミングが現れていない。

## 2. 現状の UC / SPEC

- **UC-LGX-004 事前条件**: 「上流成果物にサブノードが存在する（自動抽出 or 明示定義）」と記載。
- **UC-LGX-004 代替フロー 3a**: 「サブノードが存在しない上流成果物は、ドキュメント全体として返却する（fallback、Phase 2 Block B 後も維持）」と記載。
- **SPEC-LGX-003.REQ.08**: サブノードが target_files に含まれる場合の親解決を規定。サブノード有無に応じた動的挙動は Phase 2 Block B の UC-004 記述で対応。
- **問題点**: 事前条件がアクターに「サブノードが存在することを確認してから呼び出せ」と要求するのか、それとも「サブノード不在でも 3a fallback が発動する（事前条件は理想状態の記述）」なのか、フロー記述から判断できない。

## 3. 推奨対応（人間裁定）

**(A) UC への追記案**

事前条件の記述を変更し「上流成果物の一部がサブノードを持つことが期待される（ない場合は 3a フォールバック）」に緩める。基本フロー Step 3 の前に「上流成果物ごとにサブノード有無を確認し、存在する場合は 3a〜3c を実行、存在しない場合は代替フロー 3a を適用する」を示す条件分岐ステップを追記する。

**(B) drop（委譲容認）案**

事前条件の「サブノードが存在する」は利用場面の典型前提を示す説明的記述とみなし、検証タイミングの詳細は SPEC-LGX-003 のサブノード展開規定（REQ.08 / Phase 2 Block B）に委譲して UC フローの修正は不要と裁定する。代替フロー 3a がすでに fallback を示しているため実装上の曖昧さはない。

## 4. 影響範囲

- UC-LGX-004 §事前条件・§基本フロー Step 3（追記案の場合）
- 下流への影響: RBA / SEQA で「サブノード有無の条件分岐」がどのドメインオブジェクトに帰属するかの設計に影響する可能性がある

## 5. 解消（2026-06-13）

敵対的精査裁定: **方針B（委譲容認）**。本観点は親 SPEC（対応 TP[SPEC] green 確定済）が回答しており、UC フローへの明示列挙は任意。GREEN-by-delegation として close。詳細: docs/spec-change-proposals/2026-06-13_uc-gap-adversarial-review.md §F。

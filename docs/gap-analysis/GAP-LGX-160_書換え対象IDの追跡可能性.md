# GAP-LGX-160: 書き換え対象 ID・バックアップ場所の追跡可能性（通常 run 時の記録）が未定義

**親 TP**: TP-LGX-008
**観点出典**: TP-LGX-008 §2.9 観点 O-3
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

REQ.11 は `--dry-run` で書き換え対象を「事前確認」できるとするが、**実際に migrate を実行した（dry-run でない）後に、何が書き換えられ・どこにバックアップが取られたかの記録（ログ/サマリ出力/id-map 以外の監査痕跡）** が追跡可能かが定義されていない。REQ.08 は失敗時のバックアップ場所提示に触れるが、成功時の変更サマリは別。

## 2. 現状の SPEC / UC

SPEC-LGX-008 §3 REQ.11 は **dry-run での事前確認** に、REQ.08 は **失敗時のバックアップ場所・リカバリ手順提示** に触れているが、**成功した通常 migrate 実行後の「何を書き換え、どこに何をバックアップしたか」のサマリ/ログ出力** は未定義。STATE-INV-2（Git commit が完了）に向け、ユーザが変更内容を確認して commit する運用には、変更サマリが必要。

## 3. 期待される情報

SPEC または UC に追加されるべき記述:

- 通常 migrate 成功時の変更サマリ出力（生成ファイル、書き換えた ID 数/一覧、バックアップ場所）
- `--json` グローバルオプション使用時の構造化出力での記録（LGX-COMPAT-001 §7）
- ログレベルと出力先（stdout サマリ vs ログ）の方針

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- UC-LGX-009: 移行完了後にユーザが内容確認 → Git commit する運用ステップ（STATE-INV-2）の事後条件が定義できない
- 下流 TS / TC: 成功時出力の検証テストの期待値が書けない
- 他の GAP との依存: GAP-LGX-156（ロールバック判断に変更サマリが必要）

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-008 v0.7.0（人間裁定 fix・承認 2026-06-10）: REQ.08 に成功時変更サマリ（stdout: 生成/更新ファイル・書換 ID 件数・バックアップ場所、--format json は構造化〔スキーマ DD〕、診断は stderr）を確定。STATE-INV-2 の commit 運用を支える事後条件。

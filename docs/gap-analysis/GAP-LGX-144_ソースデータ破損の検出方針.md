# GAP-LGX-144: ソースデータ（engine.db / matrix）破損時の検出・拒否方針が未定義

**親 TP**: TP-LGX-008
**観点出典**: TP-LGX-008 §2.2 観点 E-2 / §2.11 観点 D-4
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

移行元データが破損している場合（壊れた v0.1.0 engine.db、TOML として不正な matrix 形式 `.legixy.toml`、matrix.md と engine.db が整合しない等）に、migrate がそれを検出して拒否するのか、黙って壊れた graph.toml を生成するのかが定義されていない。「破損を黙って引き継がない」保証が必要。

## 2. 現状の SPEC / UC

SPEC-LGX-008 §3 REQ.02 は **失敗時の原本保持・中断耐性** に触れ、REQ.09 は **バージョン不整合の検出** に触れているが、**ソースデータ自体の破損（バージョンは判別できるが内容が壊れている）の検出と拒否方針** は未定義。REQ.08 は失敗時メッセージを規定するが、何を「失敗」と判定するかの破損検知基準が無い。

## 3. 期待される情報

SPEC または UC に追加されるべき記述:

- 破損検出の対象と方法（engine.db の整合性検査 / matrix の構文・参照整合検査）
- 破損検出時の挙動（Error で中断し原本温存、部分移行しない）
- 「対象が無い（空）」（GAP-LGX-141）と「対象が壊れている」の区別
- 移行後 graph.toml の妥当性検証（壊れた入力から壊れた出力を生成しない保証）

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- UC-LGX-009: 例外フロー（破損入力）が定義できない
- 下流 TS / TC: 破損シナリオテストの期待値が書けない
- CTX-INV-2（グラフ整合性）: 破損入力からの graph 生成が不変条件を侵す恐れ

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-008 v0.7.0（人間裁定 fix・承認 2026-06-10）: REQ.03a 新設 — 破損検出（engine.db open/必須テーブル、TOML パース/必須構造）、検出時 Error exit 1・原本温存・部分移行なし、空入力との診断区別、生成 graph.toml の確定前妥当性検証（パース可能性 + ID 一意性）。

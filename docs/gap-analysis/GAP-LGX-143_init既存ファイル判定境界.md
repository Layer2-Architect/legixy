# GAP-LGX-143: init の「既存ファイルがある場合エラー」の判定境界が曖昧

**親 TP**: TP-LGX-008
**観点出典**: TP-LGX-008 §2.1 観点 B-5
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

`init` が生成する複数の成果物（`.legixy.toml`, 空 graph.toml, 初期 engine.db, ICONIX 8 ディレクトリ + `.gitkeep`）の一部だけが既に存在する場合、「既存ファイルがある」のエラー判定がどのファイル集合を見て下されるかが曖昧。例えば `docs/specs/` だけ存在する、`.legixy.toml` だけ存在する等の部分状態での挙動が未定義。

## 2. 現状の SPEC / UC

SPEC-LGX-008 §3 REQ.07 で **既存ファイルがある場合はエラー（`--force` で上書き）** と触れているが、**「既存ファイル」の判定対象（どれか 1 つでもか、特定ファイルのみか）と、部分的に存在する場合の挙動（既存分のみ温存か全消去か）** は未定義。

## 3. 期待される情報

SPEC または UC に追加されるべき記述:

- エラー判定の対象ファイル集合の明示（`.legixy.toml` の存在のみか、生成物のいずれかが存在すれば、か）
- 部分存在時に `--force` が「既存を上書き」するのか「不足分のみ補完」するのかの方針
- 既存の空でないディレクトリ（ユーザの既存ドキュメント）を `.gitkeep` 配置で破壊しない保証

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- UC-LGX-009: init フローの代替フロー（部分既存）が定義できない
- 下流 TS / TC: TS-LGX-007（init テスト）の境界ケースの期待値が書けない
- 他の GAP との依存: GAP-LGX-153（`.bak` 衝突）と上書き方針で関連

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-008 v0.6.0（人間承認 2026-06-10）: REQ.07 の「既存」判定対象を legixy 管理生成物（設定ファイル・graph.toml・engine.db）に限定、--force の破壊範囲も legixy 生成物のみ（REQ.02a 退避 + .gitkeep 不足分補完・ユーザファイル不変）。init [--force] シグネチャ不変。

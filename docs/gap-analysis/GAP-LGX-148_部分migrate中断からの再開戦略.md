# GAP-LGX-148: 部分マイグレーション/中断からの再開戦略が未定義

**親 TP**: TP-LGX-008
**観点出典**: TP-LGX-008 §2.3 観点 S-3 / §2.11 観点 D-2
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

migrate が複数段階（engine.db 変換、graph.toml 生成、id-map 生成、config 移行）からなるとき、ある段階で中断し再実行された場合に、**完了済段階を検出してスキップする（resume）のか、全段階を最初からやり直すのか、何が再計算されるか** の収束戦略が定義されていない。REQ.02 は「再実行可能（冪等）」と述べるが、中間状態からの再開戦略の具体が無い。

## 2. 現状の SPEC / UC

SPEC-LGX-008 §3 REQ.02 で **「途中中断されても DB が壊れない」「再実行可能（冪等）」** に触れているが、**部分完了状態（DB は変換済だが graph.toml 未生成 等）からの再実行時の振る舞い（resume / 全やり直し / 検出継続）** は未定義。

## 3. 期待される情報

SPEC または UC に追加されるべき記述:

- 中間状態の表現（どこまで完了したかの記録、または各段階の冪等性で全やり直し可能とする方針）
- 再実行時の戦略（resume か全やり直しか）と、それが原本・既生成物を破壊しない保証
- 各段階が独立に冪等であることの明示（GAP-LGX-147 の状態判定と連動）

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- UC-LGX-009: 中断→再開の代替/例外フローが定義できない
- 下流 TS / TC: 中断シミュレーション後の再実行テストの期待値が書けない
- 他の GAP との依存: GAP-LGX-147（migrate 済判定）, GAP-LGX-152（atomic 書込）, GAP-LGX-153（`.bak` 衝突）

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-008 v0.7.0（人間裁定 fix・承認 2026-06-10）: REQ.02 に再開戦略を確定 — resume なし・全やり直し方式、中間状態の進捗記録なし（STATE-INV-1 整合）、各段階の冪等性で再実行が収束、DB コミット先行 + atomic 確定により中間状態は「DB のみ新」に限定。

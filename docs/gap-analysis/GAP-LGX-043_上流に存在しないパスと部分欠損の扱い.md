Document ID: GAP-LGX-043

# GAP-LGX-043: target_files が graph.toml 上にノードを持たない / 上流が一部欠損するときの扱い

**親 TP**: TP-LGX-003
**観点出典**: TP-LGX-003 §2.7 観点 I-01, §2.2 観点 E-05
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

target_files として渡されたパスが graph.toml 上に対応ノードを持たない場合、および上流連鎖の途中でファイルが読めない / ノードが欠損している場合（部分成功）に、compile_context がエラーを返すのか、空の上流で正常終了するのか、欠損分を黙殺して残りを返すのかが定義されていない。

## 2. 現状の SPEC / UC

- SPEC-LGX-003 REQ.01 は target_files を「これから作成・編集するファイルのパス配列」とし、CLI 互換契約（LGX-COMPAT-001）で `target_files: string[]`(min1) と規定する。よって**空配列**は MCP/CLI 入力層で reject される（B-05 と同様にスキーマで担保）。
- しかし「文字列としては有効だが graph.toml にノードが無いパス」を渡したときの挙動は未定義。REQ.02 は「target_files に対応するノードを起点として」上流を辿るとするが、**対応ノードが存在しない場合の起点不在**を扱っていない。
- REQ.15 が「これから作成するファイル」を対象に含意する一方、未作成ファイルはノード未登録のことが多く、この経路の正準挙動が不明。
- 上流連鎖の途中でファイル本文が読めない / ノードが欠損する**部分成功**の扱い（エラーで止める / 欠損を飛ばして残りを返す / 欠損を Metadata に記す）も未定義。

## 3. 期待される情報

SPEC-LGX-003 REQ.01 / REQ.02 に以下を追加:

- target_files の各パスが graph.toml 上にノードを持たない場合の挙動: (a) エラー、(b) 当該パスを起点なしとして無視し残りで上流解決、(c) 空 upstream で正常終了、のいずれを正準とするか。複数パスのうち一部のみ未登録の場合の扱いも明記。
- 「これから作成するファイル」（ノード未登録パス）を意図的に許容するなら、その場合の上流解決方法（パスの命名規約から chain 位置を推定する等）を規定。
- 上流連鎖途中のファイル読取失敗 / ノード欠損時の部分成功方針（CTX-INV-1 のバイト決定論を壊さない形で）。

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- UC-LGX-002 / UC-LGX-004: 「未登録パスを渡す」「上流が壊れている」代替/例外フローが書けない。
- 下流 DD: UpstreamWalker の起点解決とエラー型（起点不在 / 部分欠損）の設計が確定しない。
- TS/TC: 未登録パス・部分欠損の期待値が定まらない。
- CTX-INV-1（決定論）: 部分欠損時に黙殺すると、欠損の有無で結果が変わり決定論検証が複雑化する。

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-003 v0.7.0（人間承認 2026-06-10）: REQ.20 新設 — 起点不在は無視して残りで解決 exit 0（推測排除・診断記録）、上流部分欠損は部分成功 exit 0 + 欠損の決定論的記録（CTX-INV-1 保全）。

## 6. 関連 ADR

該当なし（挙動定義レベル。ただし「未作成ファイルを起点に許容するか」は設計方針なので必要なら ADR 候補）。

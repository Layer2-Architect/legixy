# GAP-LGX-085: max_depth による打ち切り発生の可観測性が未規定

**親 TP**: TP-LGX-005
**観点出典**: TP-LGX-005 §2.1 観点 5
**ステータス**: closed (2026-06-10)
**重要度**: minor (verification: low-value, 人間判断で drop 可)
**起票日**: 2026-06-08

> **敵対的精査メモ（2026-06-09）**: REQ.04 で省略時 = 無制限が既定であり、`--max-depth N` を明示した利用者は打ち切りの可能性を認識済みとみなせる。打ち切り通知の有無は決定論的な到達集合・depth_map を一切変えず、`--json` スキーマ詳細（truncated フラグの有無）は REQ.09 が DD 凍結対象に正しく委譲済み。SPEC レベルで方針を確定しても下流の挙動契約は不変であり、観測性 UX の好みの問題に留まる low-value。人間判断で drop 可。

## 1. 観点

`--max-depth N` を指定した走査で、深度 N を超える到達可能ノードが存在し打ち切られた場合、その「打ち切りが発生した（結果が部分集合である）」事実が利用者に観測可能か（例: stderr への Info、または `--json` 出力中のフラグ／件数）が規定されていない。打ち切りが無言だと、利用者は impact / investigate の結果を「到達可能全集合」と誤認する恐れがある。

## 2. 現状の SPEC / UC

SPEC-LGX-005 REQ.04 で **「`max_depth` を超えるノードは返さない」** に触れているが、**打ち切りが起きたことを示す通知・フラグ・件数の有無** は未定義。REQ.09 の出力情報（ID 走査順 / type / path / 使用エッジ / 深度）にも打ち切り境界の表現は含まれていない。

NFR-LGX-001 OBS.02（結果は stdout、ログは stderr）/ USE.02（エラー時の実用的出力）は方針を与えるが、打ち切りという「正常だが部分的な結果」の可視化方法は SPEC-LGX-005 が確定すべき。

## 3. 期待される情報

SPEC-LGX-005 に追加されるべき記述（いずれかの方針の確定）:

- 打ち切り発生時に通知を出さない（無制限が既定であり、明示的に `--max-depth` を与えた利用者は打ち切りを認識済みとみなす）とする方針、または
- 打ち切り発生時に stderr へ Info を出す／`--json` 出力に「truncated: true」相当のフラグや打ち切り件数を含める方針
- 採用方針が SPEC-LGX-010 の `--json` 出力一貫性方針および v3 実測（打ち切り通知の有無）と整合すること

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- UC-LGX-005 / UC-LGX-006: 利用者が部分結果を全集合と誤認するリスクが残り、要求追跡・影響分析の判断を誤らせうる
- 下流の DD/TS: `--json` スキーマ（REQ.09 が DD 凍結対象とする）に truncated フラグを含めるか否かが決まらない
- 他の TP / GAP との依存関係: GAP-LGX-088（未解決・到達不能の可視化）と出力の可観測性方針を共有する

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-005 v0.4.0（人間裁定 2026-06-10: Info 追加）: REQ.04 に打ち切り発生時の stderr Info 1 件（発生 + 除外件数）を新設【v3 差分: v3 は無言打ち切り】。stdout の到達集合・depth_map・終了コードは不変。--json truncated フラグは REQ.09 の DD 凍結対象へ申し送り。

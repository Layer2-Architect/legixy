# GAP-LGX-236: UC-LGX-006 が --max-depth 打ち切り時の stderr Info 出力（SPEC-005.REQ.04）を反映していない

**親 TP**: TP-LGX-016
**観点**: 2.6 R3（SPEC-LGX-005.REQ.04 の打ち切り通知が UC フローに反映されているか）
**ステータス**: closed (2026-06-13)
**起票日**: 2026-06-13
**分類（暫定）**: WEAK（SPEC-005.REQ.04 委譲で解決可。UC フロー記述への明示は任意の可能性）

## 1. 観点

UC レベル観点「各ステップの事後条件が後続ステップの前提を満たすか」を UC-LGX-006 の基本フロー Step3（`--max-depth` 指定時の深度打ち切り）にぶつけた。打ち切り発生時の観察可能な通知が UC フローに現れているかを問う。

## 2. 現状の UC / SPEC

- UC-LGX-006 基本フロー Step3: 「`--max-depth` が指定されている場合、その深度で走査を打ち切る」
  - 打ち切りの事実（除外されたノードが存在すること）のユーザへの通知が記述されていない。
- SPEC-LGX-005.REQ.04（GAP-LGX-085 対応、人間裁定 2026-06-10）: 「`--max-depth` 指定時に深度超過で除外された到達可能ノードが 1 件以上存在する場合、**stderr に Info 1 件**（打ち切り発生 + 除外ノード件数）を出力する【v3 差分】」と規定済。
- REQ.04 は「stdout の到達集合・depth_map・終了コードは不変（決定論保全）」と明記しており、打ち切り Info は stderr のみに出力される。
- UC の事後条件「走査結果が標準出力に表示される」は stdout の結果のみを言及し、stderr への Info 出力は言及されていない。
- 代替フロー 1a（`--max-depth` 未指定=全深度）にも打ち切り Info の非発生（Info なし）が明示されていない。

振る舞いは SPEC-005.REQ.04 で確定済だが、UC のフロー記述（打ち切り通知の観察可能性）に反映されていない。

## 3. 推奨対応（人間裁定）

以下のいずれか:

- **(A) UC へ追記**: 基本フロー Step3 に「`--max-depth` 指定かつ深度超過で除外ノードが存在する場合、stderr に打ち切り情報（Info）を出力する（SPEC-LGX-005.REQ.04）」を追記。または代替フローとして「3a. `--max-depth` で除外ノードが発生した場合、stderr に Info を出力する（stdout 結果・終了コードは不変）」を追加。
- **(B) drop（委譲容認）**: 打ち切り発生時の stderr Info は SPEC-LGX-005.REQ.04 に委譲する設計と認め、UC はメイン走査フロー（stdout 結果）のみを記述する。→ UC は変更しない。

WEAK 候補（SPEC が既に答えており UC フロー記述への明示は任意の可能性）。

# GAP-LGX-102: 巨大テキストがモデルの最大トークン長を超過した場合の切り捨て/分割/エラー方針が未定義

**親 TP**: TP-LGX-006
**観点出典**: TP-LGX-006 §2.1 観点 B-02
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08
**重要度**: minor (verification: low-value, 人間判断で drop 可) — 敵対的精査パス 2026-06-09: トークナイザ/モデル層の挙動であり DD 凍結事項。REQ.01 は pooling/prefix のみ規定し max-token は model 既定動作。SPEC-decision-relevance 低。WEAK_OR_PADDED として維持。

## 1. 観点

ドキュメントノード本文が ONNX モデルの最大入力トークン長（paraphrase-multilingual-MiniLM-L12-v2 は典型的に 128〜512 トークン）を超過した場合、embedding 生成が (a) 先頭で切り捨てる、(b) 分割して平均する、(c) エラーにする、のいずれを採るかが未定義。切り捨て採用時、ドキュメント末尾の情報が embedding に反映されず drift / 類似度の精度に影響する。

## 2. 現状の SPEC / UC

SPEC-LGX-006 §3 REQ.01/REQ.02 はモデルとプーリングを規定するが、**入力長上限と超過時の方針**を記述していない。REQ.09 はサブノード入力を content_range に限定しテンプレ寄与を排除するが、これは入力範囲の話であり、範囲内テキストがモデル上限を超える場合の扱いとは別問題。NFR-LGX-001.SEC.04 は 100 MB 超ファイルの OOM 防止を要求するが、これはクラッシュ防止であり embedding 品質上の切り捨て方針ではない。

## 3. 期待される情報

- モデルの最大トークン長の取扱い（モデル metadata から動的取得か、設定値か）
- 超過時の方針: 切り捨て（先頭 N トークン）/ チャンク分割 + 平均 / Warning 付き切り捨て / Error
- 切り捨て採用時、ユーザへの通知（Warning）有無
- サブノード分割（SPEC-LGX-002）との関係（サブノード化が事実上の分割になるか）

## 4. 影響範囲

- UC-LGX-007: 長大ドキュメントの embedding 生成フローの事後条件
- 下流 DD / TS: トークナイザ層の上限処理仕様・テスト境界値（上限ちょうど / 上限+1）
- drift 検出（REQ.05）の精度: 切り捨て後テキストの content_hash と embedding の整合

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-006 v0.7.0（人間裁定 fix・承認 2026-06-10）: REQ.01 にトークン上限超過時の方針を確定 — 上限はモデル metadata から取得、先頭切り捨て + 集約 Warning【v3 差分: v3 は無言】、チャンク分割平均は不採用（実用対策はサブノード化）、境界詳細は DD。

## 6. 関連 ADR

切り捨て vs 分割は精度/性能トレードオフを含むため ADR 候補。

# GAP-LGX-233: UC-LGX-006 の start_ids（複数形）と基本フロー「起点ノードから」（単数）が不整合

**親 TP**: TP-LGX-016
**観点**: 2.1 BF4（アクター入力の検証タイミング・複数起点セマンティクスの観察可能性）
**ステータス**: closed (2026-06-13)
**起票日**: 2026-06-13
**分類（暫定）**: GENUINE 寄り（SPEC-LGX-005 にも複数起点の明示記述なし。下流 DD 設計に影響しうる）

## 1. 観点

UC レベル観点「各ステップの前提条件が先行ステップの事後条件を満たすか」および「アクター入力の検証タイミング」を UC-LGX-006 の基本フロー Step1/Step2 の入力定義にぶつけた。

## 2. 現状の UC / SPEC

- UC-LGX-006 Step1 は `legixy impact <start_ids> [--max-depth <n>]` と複数形 `start_ids` で記述している。
- Step2 は「システムが起点ノードから有向グラフを順方向（下流方向）に BFS 走査する」と単数形「起点ノード」で記述している。
- Step4 の返却形式「visited: 影響を受ける全ノード（走査順）」「depth_map: 各ノードの起点からの距離」も単一起点を前提とした記述。
- SPEC-LGX-005.REQ.07 は「`legixy impact <node-id>`」と単数形 `<node-id>` で CLI インターフェースを定義しており、複数起点の指定について明示的な記述がない。
- LGX-COMPAT-001 §4 #11「`impact <start>`」も単数形。
- 複数起点を指定した場合の走査セマンティクス（①各起点から個別に走査して結果を連結するか、②全起点を同時にキューに投入して単一 BFS を走らせるか）が UC・SPEC のいずれにも未定義。

## 3. 推奨対応（人間裁定）

以下のいずれか:

- **(A) UC へ修正・明示化**: Step1 の `<start_ids>` を `<start_id>` に修正し、単一起点のみを受け付ける仕様と明確にする。または `<start_ids>` を維持して Step2 に複数起点時のセマンティクス（例:「複数指定時は全起点を BFS の初期キューに投入する」）を追記し、depth_map の起点からの距離の定義（複数起点時は最小距離か起点ごとの距離マップか）を明示する。
- **(B) SPEC-LGX-005 修正（人間承認必要）**: REQ.07 に複数起点の仕様を追記（または単数と明確化）し、UC を整合させる。
- **(C) drop（UC 側を単数形に統一）**: `start_ids` は慣用表記（CLI の `<args>`）であり単一 ID 指定を意味すると解釈し、フロー記述を `start_id` に統一する旨を ADR または UC 注記に記載する。

GENUINE 寄り：SPEC にも記述がなく、複数起点を受理する実装（v3 の `traverse_multi` 相当）があれば互換性問題になりえる。

## 4. 影響範囲

- close されないと TP-LGX-016 が green にならず、UC-006 起点の下流（RBA 以降）に進めない。
- 複数起点のセマンティクスが未定義のまま RBD/DD に進むと、`depth_map` の型定義（`HashMap<NodeId, usize>` の意味論）が曖昧になるリスクがある。
- v3 の `multi_traverser.rs` が複数起点を受け付ける実装であれば、互換契約（LGX-COMPAT-001）との照合が必要。

## 5. 解消（2026-06-13）

敵対的精査裁定: **GENUINE**（実 SPEC 照合で確定）。UC 修正で解消（A2: UC-LGX-005/006 の起点引数を単数 <node-id> へ凍結互換準拠）。人間承認 2026-06-13（A2/C2/C3 は AskUserQuestion 裁定、推奨案採用）。詳細: docs/spec-change-proposals/2026-06-13_uc-gap-adversarial-review.md §A。

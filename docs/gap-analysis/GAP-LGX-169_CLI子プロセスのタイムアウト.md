# GAP-LGX-169: CLI 子プロセスのハング時タイムアウト・回収セマンティクスが未定義

**親 TP**: TP-LGX-009
**観点出典**: TP-LGX-009 §2.6 観点 33（ライフサイクル）
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08
**敵対的精査（2026-06-09）**: minor (verification: low-value, 人間判断で drop 可)。legixy は v3 と実行時互換が凍結境界（LGX-COMPAT-001）であり、v3 MCP 層に子プロセスタイムアウト機構が存在しない以上、新設は「引数転送ロジックの拡張のみ」（REQ.09）を超える新挙動になる。子プロセス回収・kill は thin forwarder の DD レベル実装事項、ブロッキング上限は NFR-REL 系の所有。SPEC レベルの決定相関性は低い。

## 1. 観点

Rust CLI 子プロセスが応答せずハングした場合（無限ループ、 I/O ブロック、デッドロック等）、MCP サーバがタイムアウトで打ち切るか、いつまでも待つかが未定義。打ち切る場合のシグナル送出（kill）、ゾンビプロセス回収、Agent への応答（タイムアウトエラー）の規定がない。

## 2. 現状の SPEC

SPEC-LGX-009 §3 REQ.06 は **「オーバーヘッドは PERF.03 を満たす必要がある」** に触れ性能予算を示すが、これは**目標値であってタイムアウト上限ではない**。**子プロセスがハングした場合のタイムアウト値・打ち切り手段（SIGTERM/SIGKILL）・回収・タイムアウト時の MCP 応答** は未定義。REQ.07 の `exit -1` フォールバックはシグナル等を含むが、MCP 側からのタイムアウト打ち切りに言及しない。core-perspectives §ライフサイクル「shutdown 順序保証（進行中処理のキャンセル待ち）」、§並行性「cancellation 伝播の保証」に対応する観点が空白。

## 3. 期待される情報

SPEC に追加されるべき記述:

- 子プロセスのタイムアウト上限の有無（あれば値の根拠、PERF.03 との関係）
- タイムアウト時の打ち切り手段（kill シグナル）とプロセス回収（ゾンビ防止）
- タイムアウト時の MCP 応答形式（`isError: true`, メッセージ）と、部分出力の扱い（GAP-LGX-163 と関連）

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- 全 UC（MCP 経由）: 異常時の応答が永久に返らないリスク（Agent のハング）
- 下流の SEQD/DD/TS: 子プロセス管理（spawn → wait → timeout → kill）のシーケンスが書けない
- NFR-LGX-001 PERF.03 / REL 系との整合
- 関連: GAP-LGX-163（部分出力）, GAP-LGX-167（並行制御）

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-009 v0.6.0（人間裁定 2026-06-10: タイムアウト導入【v3 差分】）: REQ.16 新設 — 既定 30 秒・LGX_MCP_TIMEOUT_SEC で変更可・0 で無効（v3 互換動作）・SIGTERM→猶予 5 秒→SIGKILL・必ず回収・応答は isError + `Rust CLI failed (timeout after {N}s):`・部分出力非転送（MCP-INV-2 保全）。正当入力空間の挙動は不変。ADR-LGX-010。

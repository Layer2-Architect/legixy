# GAP-LGX-004: umbrella SPEC に partition 改訂の変更ポリシー（統治手順）が規範要件として無い

**親 TP**: TP-LGX-001
**観点出典**: TP-LGX-001 §2.5 観点 17（分割の網羅性と無重複 — 下位 SPEC のスコープ拡張時に umbrella を再改訂する統治手順）
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08
**severity**: minor
**敵対的精査メモ（2026-06-09）**: GENUINE と確認。SPEC-LGX-001 に「変更ポリシー」節は存在しない（grep 確認済）一方、`docs/DevProc_V4/03-spec-level-tdd.md` line 33 が SPEC 推奨構造として「変更ポリシー（誰の承認で変えられるか）」を挙げており、欠落は事実。GAP-LGX-001（具体的な partition 不一致の実例）とは粒度が異なり DUPLICATE ではない（001 は単一インスタンス、004 は根本の統治手順欠如）。GAP-LGX-002 の同期規律要求は本 GAP の subset であり本 GAP に集約するのが適切。severity は minor（ハードルール 1 と process docs が外部から統治を補っているため、umbrella 自身への明文化は構造的整備であり致命性は低い）。

## 1. 観点

下位 SPEC（002〜010）がスコープを拡張して partition（責務分割）を変更した際に、umbrella（SPEC-LGX-001）の網羅宣言・カテゴリ表・不変条件マトリクスを再改訂する統治手順が、umbrella 自身に規範要件（変更ポリシー）として規定されているか。

## 2. 現状の SPEC / UC

`docs/DevProc_V4/03-spec-level-tdd.md` §2 は SPEC 推奨構造として「変更ポリシー（誰の承認で変えられるか）」節を挙げているが、SPEC-LGX-001 には **変更ポリシー節が存在しない**。§6 変更履歴はあるが、これは過去の改訂記録であって、将来の改訂をどのトリガ・誰の承認・どの同期規律で行うかの規範ではない。

partition 改訂手順は現状 **下位 SPEC 側の散文にのみ存在する**。例えば SPEC-LGX-010 §1.3 が「UC-012/013 生成に伴い SPEC-LGX-001 REQ.01/REQ.02 を再改訂する必要がある。これは UC フェーズ着手時に SPP-LGX-001 次反復として処理する」と述べているが、これは子 SPEC が親の改訂義務を片務的に予告しているにすぎず、umbrella 自身が「下位 SPEC が partition を変えたら umbrella を X の手順で同期する」という統治を規範化していない。結果として GAP-LGX-001 のような umbrella↔下位 SPEC 不一致が、検出も是正も保証されない状態で発生しうる。

## 3. 期待される情報

SPEC に追加されるべき記述（SPEC-LGX-001 に新節「変更ポリシー」として）:

- umbrella の網羅宣言（REQ.01/REQ.02）・カテゴリ表（REQ.02）・不変条件マトリクス（§4.1）を変更できる主体と承認プロセス（ハードルール 1: SPEC 変更は人間承認）。
- 下位 SPEC（002〜010）が新カテゴリ・新 UC・新不変条件を導入し partition を拡張した場合に、umbrella を同期改訂するトリガと期限（例: 当該下位 SPEC の FCR ACCEPTED 後の UC フェーズ着手時に SPP 次反復として umbrella を更新する、を規範化）。
- 同期漏れを検出する規律（GAP-LGX-002 と整合: マトリクス・網羅宣言を同一コミットで更新する等のハードルール化）。

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- GAP-LGX-001 の根本原因: 統治手順が無いため、下位 SPEC のスコープ拡張が umbrella に反映されず網羅宣言が陳腐化する事象が再発する。
- cross-SPEC partition 全体: 新規 SPEC 追加（spec-add イベント）時に umbrella のどこを誰がいつ更新するかが不定で、孤児カテゴリ／孤児不変条件の混入リスクが残る。
- 下流全般: umbrella が partition の正準であるという前提が運用上崩れると、全 SPEC のトレーサビリティの起点が不安定になる。

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-001 v0.7.0（人間承認 2026-06-10）: §7 変更ポリシーを新設（§7.1 人間承認、§7.2 同期トリガ=FCR ACCEPTED・期限=UC フェーズ着手時、§7.3 §4.1 正準宣言+同一コミット更新規律）。旧 GAP-LGX-002 の同期規律 subset を §7.3 が吸収。

## 6. 関連 ADR

- partition 改訂の統治手順（トリガ・承認・同期規律）の確定は architectural 判断であり ADR-LGX-NNN を起票候補。

Document ID: GAP-LGX-108

# GAP-LGX-108: embed --all の部分失敗トレランスと単一トランザクション境界の整合が矛盾しうる

**親 TP**: TP-LGX-006
**観点出典**: TP-LGX-006 §2.2 観点 E-03
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

REQ.08 は「embedding 生成・格納は単一 SQLite トランザクション内で完結する」と規定する一方、REQ.09 はサブノードについて「失敗時は該当サブノードのみ Error 計上、他ノードは継続（部分失敗トレランス）」と規定する。`embed --all` で 1 ノードの生成が失敗したとき、単一トランザクションで全ロールバックするのか、ノード単位でコミットして部分成功を残すのかが矛盾しうる。トランザクション境界が「全 embed = 1 Tx」か「ノード単位 = N Tx」かが未確定。

## 2. 現状の SPEC / UC

SPEC-LGX-006 §3 REQ.08（トランザクション境界、NFR REL.06）と REQ.09（サブノードの部分失敗トレランス、ドキュメントノードの部分失敗継承）が、**`embed --all` 実行単位のトランザクション粒度**について整合的に読めない。「単一トランザクション内で完結」がノード 1 件あたりなのか、`embed --all` 全体なのかが曖昧。ドキュメントノード（サブノードでない）の生成失敗時の確定範囲も明記されていない。

## 3. 期待される情報

- トランザクション粒度の明確化: (a) ノード/サブノード単位 1 Tx（部分成功が永続化される）、(b) embed --all 全体 1 Tx（1 件失敗で全ロールバック）
- REQ.08 の「単一トランザクション」と REQ.09 の「部分失敗トレランス」の両立方法（例: ノード単位 Tx + 失敗ノードを skip して継続）
- ドキュメントノードの生成失敗時に確定するもの/破棄されるものの明示
- 部分失敗後の終了コード（GAP-LGX-109 と連携）

## 4. 影響範囲

- UC-LGX-007: 部分失敗時の事後条件（何が永続化されるか）が定義できない
- 下流 DD / TS: トランザクション境界の実装仕様と中断テスト（REQ.08 検証方法「中断テスト」の期待値）
- NFR-LGX-001.REL.02（部分失敗時の継続）/ REL.06（トランザクション境界）の両立検証

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-006 v0.6.0（人間承認 2026-06-10）: REQ.08 のトランザクション粒度を「ノード/サブノード単位 1 Tx」に確定（REQ.09 部分失敗トレランス優先、--all 全体 1 Tx 禁止）。終了コード詳細は SPEC-LGX-010 に委譲。

## 6. 関連 ADR

トランザクション粒度（性能 vs 原子性）は architectural 判断のため **ADR 候補**。

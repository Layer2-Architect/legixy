# GAP-LGX-232: UC-LGX-006 の事後条件に成功時 exit コードが記載されていない

**親 TP**: TP-LGX-016
**観点**: 2.1 BF3（成功時の事後条件が観察可能か）
**ステータス**: closed (2026-06-13)
**起票日**: 2026-06-13
**分類（暫定）**: WEAK（SPEC-005 + LGX-COMPAT-001 委譲で解決可。UC-005 も同様の不記載で対称的）

## 1. 観点

UC レベル観点「成功時の事後条件が観察可能か」を UC-LGX-006 の事後条件節にぶつけた。特に exit コードが後続 UC や CI の前提として参照可能かを問う。

## 2. 現状の UC / SPEC

- UC-LGX-006 の事後条件は「走査結果が標準出力に表示される」「グラフの状態は変更されない（読み取り専用操作）」の 2 点のみ。
- 成功時の exit コード（exit 0）が事後条件に明示されていない。
- exit 0 は SPEC-LGX-005.REQ.05（起点不在=空結果・非エラー → exit 0）+ NFR-LGX-001 OBS.05/USE.04 + LGX-COMPAT-001 §3/§4（終了コード凍結契約）から導出可能だが、UC フロー記述として独立した根拠がない。
- UC-LGX-001（グラフ読み込みと検証）は事後条件に「ERROR 0 件時 exit 0」「ERROR 有時 exit 1」を明示しており、UC-006 との記述粒度差がある。
- UC-LGX-005（逆方向探索）も同様に exit コードを事後条件に記載していないため、対称的な不記載。

## 3. 推奨対応（人間裁定）

以下のいずれか:

- **(A) UC へ追記**: 事後条件に「exit 0 で終了する（エラーがない場合）」を追加し、2a 代替フロー（起点不在=ERROR）の exit コード（SPEC-LGX-005.REQ.05 参照）とあわせて整理。
- **(B) drop（委譲容認）**: exit コード規約は LGX-COMPAT-001 §3/§4 が所有し、UC は走査結果のユーザ可視部分（stdout 表示・グラフ不変）のみを記述する設計と認める。→ UC は変更しない（UC-005 とも統一）。

WEAK 候補。UC-005 との対称性を保つため B 案が妥当の可能性が高い。一括裁定を推奨。

## 4. 影響範囲

- close されないと TP-LGX-016 が green にならず、UC-006 起点の下流に進めない。
- exit コードの実装値（0/1/2）は LGX-COMPAT-001 で既に凍結済であり、下流実装の正しさには影響しない。
- GAP-LGX-231（BF1）/ GAP-LGX-234（AF2: 2a exit コード）と同種（フロー末尾・exit コード観察可能性）であり、一括裁定が可能。

## 5. 解消（2026-06-13）

敵対的精査裁定: **方針B（委譲容認）**。本観点は親 SPEC（対応 TP[SPEC] green 確定済）が回答しており、UC フローへの明示列挙は任意。GREEN-by-delegation として close。詳細: docs/spec-change-proposals/2026-06-13_uc-gap-adversarial-review.md §F。

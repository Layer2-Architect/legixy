# GAP-LGX-215: UC-LGX-004 Step 3（サブノード抽出処理）の失敗パスがフローに列挙されていない

**親 TP**: TP-LGX-014
**観点**: §2.3 EF1（Step 3 実行中の失敗パス欠落）
**ステータス**: closed (2026-06-13)
**起票日**: 2026-06-13
**分類（暫定）**: GENUINE

## 1. 観点

UC-LGX-004 の基本フロー Step 3（a. エッジを辿り関連サブノードを特定 / b. 各サブノードの本文を抽出 / c. ドリフトスコアを付与）において、各サブステップが失敗した場合の例外パスが代替フロー・例外フロー節に一切列挙されていない。

具体的な失敗ケースとして以下が想定される:
- Step 3a: サブノードへのエッジが graph.toml に記録されているが対応するファイルが不在、またはエッジ定義が不整合
- Step 3b: サブノードファイルの破損・パース不能、または `content_range` で指定された範囲がファイルサイズを超過
- Step 3c: embedding が存在せず drift_score が計算不能（UC-LGX-007 未実行状態）

## 2. 現状の UC / SPEC

- **UC-LGX-004 代替フロー / 例外フロー**: Step 1〜4 に対する失敗パスが一切記述されていない。代替フロー 3a（サブノード不在の fallback）は「サブノードが存在しない」という正常な条件分岐であり、サブノード処理の実行時失敗ではない。
- **SPEC-LGX-003.REQ.20**: 上流連鎖途中の欠損（エッジ先ノードのファイル不在等）は「欠損ノードを飛ばして残りの上流を返す部分成功・exit 0」と規定。この規定は Step 3 の失敗シナリオに適用可能と考えられるが、UC のフロー記述として明示されていない。
- **問題点**: Step 3 固有の失敗（content_range 超過・サブノードファイル破損・エッジ不整合）が上流欠損とは異なる文脈で発生しうるが、UC フローに失敗パスが存在しない。

## 3. 推奨対応（人間裁定）

**(A) UC への追記案**

代替フロー（または例外フロー）に以下を追加する:
- 「3-ERR. Step 3b でサブノード本文の抽出が失敗した場合（ファイル不在・破損・content_range 超過）、当該サブノードを欠損として記録し、SPEC-LGX-003.REQ.20 の部分成功規定に従い残りのサブノードで処理を継続する（exit 0）」
- 「3c-ERR. Step 3c で drift_score 付与が不能の場合（embedding 不在等）、当該サブノードの drift_score を null/absent として UpstreamArtifact を返却する（UC-LGX-007 との関係は条件付き）」

**(B) drop（委譲容認）案**

SPEC-LGX-003.REQ.20（上流部分欠損は部分成功・exit 0・欠損は決定論的に記録）に委譲し、UC フロー記述の修正は不要と裁定する。Step 3 の失敗は「上流連鎖途中の欠損」の一形態として解釈可能。

## 4. 影響範囲

- UC-LGX-004 §代替フロー（追記案の場合）
- 下流への影響: RBA / SEQA でサブノード抽出失敗時の制御フローをどのドメインオブジェクトが処理するかの設計に影響する。GENUINE 確定の場合は RBA 着手前に解消が望ましい

## 5. 解消（2026-06-13）

敵対的精査裁定: **方針B（委譲容認）**。本観点は親 SPEC（対応 TP[SPEC] green 確定済）が回答しており、UC フローへの明示列挙は任意。GREEN-by-delegation として close。詳細: docs/spec-change-proposals/2026-06-13_uc-gap-adversarial-review.md §F。

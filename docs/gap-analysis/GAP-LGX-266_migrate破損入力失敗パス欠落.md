# GAP-LGX-266: migrate の破損入力失敗パスが UC フローに欠落

**親 TP**: TP-LGX-019
**観点**: §2.3 EF1
**ステータス**: closed (2026-06-13)
**起票日**: 2026-06-13
**分類（暫定）**: GENUINE

## 1. 観点

migrate Step 2 での破損入力（engine.db open/クエリ失敗・TOML パース失敗・必須構造欠落）に対する失敗パスが UC の代替フロー/例外フローに列挙されていない。SPEC-LGX-008.REQ.03a（破損検出→Error 中断・原本温存・部分移行なし）が UC フロー記述として観察可能でない。

## 2. 現状の UC / SPEC

UC-LGX-009 の migrate 代替フロー:
- **2b.** migrate で v0.1.0 プロジェクトが**見つからない**場合、ERROR を報告する

SPEC-LGX-008.REQ.03a は「対象が壊れている」（パース失敗・必須構造欠落）を「対象が無い」（パース成功 + 抽出 0 件）と明確に区別し、破損検出時は **Error（exit 1）で中断・原本温存・部分移行なし** を要求する。

破損ケース例:
- engine.db が存在するが open 失敗 / 必須テーブル欠落
- `.legixy.toml` または `.trace-engine.toml` が存在するが TOML パース失敗
- `[id.chain]` の `order` が欠落・不正（REQ.03 により Error）
- matrix.md が存在するが想定構造でない（ただし 0 件は正常終了）

UC の代替フロー 2b は「**見つからない**」（不在）のみを扱い、「**見つかったが壊れている**」（破損）の失敗パスが存在しない。このギャップは REQ.03a が明確に区別を要求している文脈で重要。

## 3. 推奨対応（人間裁定）

**(A) UC へ追記案:**
代替フロー 2b に加えて、破損検出の失敗パスを追加する:
- **2c.** migrate で読み込んだ入力データが破損している場合（TOML パース失敗・`[id.chain]` 欠落・engine.db 必須テーブル欠落等）、ERROR を報告して終了する（原本は保持する）

「見つからない（2b）」と「壊れている（2c）」の区別が UC で観察可能となる。

**(B) drop（委譲容認）案:**
SPEC-LGX-008.REQ.03a + TP-LGX-008 E-2/D-4 へ委譲確定とする。UC は最上位の分岐（不在）を示し、破損の詳細検出は SPEC レベルの規定として委譲する。ただし委譲容認とした場合でも REQ.03a の「対象なし vs 対象破損の明確な区別」が下流 RBA で適切に反映される必要がある。

## 4. 影響範囲

- 下流 RBA/SEQD: migrate のエラーパスシーケンスで破損検出→中断パスが entity として現れるか
- 下流 DD: 破損検出ロジック（open 失敗・TOML パース失敗・必須フィールド検証）の実装
- 下流 TC: 破損 fixture を使った E2E テスト（TP-LGX-008 E-2 の TC レベル実装）
- SPEC-LGX-008.REQ.03a との整合性保証

敵対的精査裁定: **方針B（委譲容認）**。本観点は親 SPEC（対応 TP[SPEC] green 確定済）が回答しており、UC フローへの明示列挙は任意。GREEN-by-delegation として close。詳細: docs/spec-change-proposals/2026-06-13_uc-gap-adversarial-review.md §F。

# GAP-LGX-203: UC-LGX-003 代替フロー 2a/2b でループ継続が未明示

**親 TP**: TP-LGX-013
**観点**: §2.2 AF3
**ステータス**: closed (2026-06-13)
**起票日**: 2026-06-13
**分類（暫定）**: WEAK

## 1. 観点

UC-LGX-003 代替フロー 2a「ドキュメントファイルが存在しない場合、そのノードのサブノード抽出はスキップする」および 2b「h2/h3 見出しが存在しない場合、サブノードは生成されない」は、個別ノードの処理結果を記述しているが、スキップ後に残りのドキュメントノードへのループが継続するかどうかが明示されていない。

基本フロー Step2 は「各ドキュメントノード（.md ファイル）に対して」とループを示唆するが、代替フローの記述はループ文脈での「継続」を明示していない。全ノードがスキップ対象だった場合の事後条件（サブノードゼロのグラフが成立するか）も不明瞭。

## 2. 現状の UC / SPEC

- UC-LGX-003 代替フロー 2a: 「そのノードのサブノード抽出はスキップする」——次のノードへのループ継続については記述なし。
- UC-LGX-003 代替フロー 2b: 「サブノードは生成されない」——同上。
- SPEC-LGX-002.REQ.09: 「プロジェクトはサブノード化を必須としてはならない。全ドキュメントノードのみで運用することも可能（v0.1.0 互換）」——サブノードゼロ運用が正常系として成立することを規定。
- SPEC-LGX-002.REQ.10: 「大きすぎる Markdown ファイル、深いネスト見出し、不正な TOML、project_root 外への参照パス」でクラッシュ禁止——暗黙的にループ継続を前提とするが UC への反映はない。

## 3. 推奨対応（人間裁定）

(A) **UC へ追記案**: 代替フロー 2a/2b に「その後、次のドキュメントノードへの処理を継続する（ループを継続する）」を追記する。また、全ノードスキップ時の事後条件を追記するか、SPEC-LGX-002.REQ.09 への委譲明示を加える。

(B) **drop（委譲容認）案**: 「各ドキュメントノードに対して」というループ構造は基本フロー Step2 に明示されており、代替フローは分岐を記述するのみで継続は自明として drop する。SPEC-LGX-002.REQ.09 が全ノードスキップ後の正常系（サブノードゼロ運用）を担保。

## 4. 影響範囲

- UC-LGX-003 §代替フロー 2a, 2b
- 影響度は低（ループ継続は基本フロー記述から自明）。WEAK 候補として下流への波及は軽微。

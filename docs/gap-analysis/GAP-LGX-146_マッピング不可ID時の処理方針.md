# GAP-LGX-146: マッピング不可 ID 検出時に処理を継続するか中断するかが未定義

**親 TP**: TP-LGX-008
**観点出典**: TP-LGX-008 §2.2 観点 E-6
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

REQ.11 で旧 graph.toml 内の ID を新 ID へ書き換える際、マッピング不可の ID があれば Warning を報告するとあるが、その後 **処理を継続して未マップ ID を残したまま graph.toml を生成するのか、中断するのか**、また **未マップ ID が graph.toml 上でどう扱われるか（旧 ID のまま残置/エッジ削除/dangling 参照）** が定義されていない。

## 2. 現状の SPEC / UC

SPEC-LGX-008 §3 REQ.11 で **「マッピング不可の ID があれば migrate コマンドが Warning を報告し、人間の確認を促す」** に触れているが、**Warning 後の処理継続/中断の選択、未マップ ID の最終的な扱い** は未定義。

## 3. 期待される情報

SPEC または UC に追加されるべき記述:

- マッピング不可 ID 検出時に処理を継続するか中断するか（人間確認を待つインタラクションの有無）
- 未マップ ID を graph.toml にどう書くか（旧 ID 残置 / エッジ除外 / プレースホルダ）と、それが CTX-INV-2（DAG・整合性）を侵さない保証
- `--dry-run` 時に未マップ ID 一覧を提示する出力仕様

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- UC-LGX-009: ID 互換移行フローの例外パスが定義できない
- 下流 TS / TC: 旧 ID 含む graph.toml 入力テストの期待値が書けない
- 他の GAP との依存: GAP-LGX-159（id-map 一意性/重複）

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-008 v0.6.0（人間承認 2026-06-10）: REQ.11 にマッピング不可 ID の既定 abort（graph.toml/id-map 不変・exit 1）と継続 opt-in 時のエッジ除外（dangling 防止）を確定。継続フラグ名は DD で定義。§4 CTX-INV-2 行更新。

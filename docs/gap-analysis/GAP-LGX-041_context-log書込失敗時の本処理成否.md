Document ID: GAP-LGX-041

# GAP-LGX-041: context_log 書込が失敗したとき compile_context の本処理は成功するか失敗するか

**親 TP**: TP-LGX-003
**観点出典**: TP-LGX-003 §2.2 観点 E-03, §2.4 観点 C-03, §2.6 観点 P-02 / P-03
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

`compile_context` の本処理（上流成果物の返却）が成功したあと、監査ログ `context_log` への書き込みが失敗（DB ロック競合で busy_timeout 超過 / ディスクフル / 書込権限欠如）した場合に、呼出し全体が成功扱いになるのか失敗扱いになるのか、また本処理結果の返却と記録の atomicity（トランザクション境界）が定義されていない。

## 2. 現状の SPEC / UC

- SPEC-LGX-003 REQ.07 / §4 MCP-INV-4 は「compile_context の**全呼出し**が context_log に記録される」と要求する。
- SPEC-LGX-003 REQ.09 は並行書込を「WAL + busy_timeout により排他制御する」と規定するが、**busy_timeout を超過してロックが取れなかった場合の挙動**（リトライ上限・エラー返却・ログ欠落の許容可否）には触れていない。
- NFR-LGX-001 REL.07 は busy_timeout の上限を示すが、超過時に compile_context が取るべき終端状態には接続されていない。
- LGX-EXT-002 §5.1 は「サイズ超過時のエラーもログ対象」とするが、**ログ書込そのものが失敗したとき**の本処理の成否は未定義。

結果として「MCP-INV-4: 全呼出し記録」を厳格適用すると記録失敗時は呼出しも失敗させるべきに読めるが、ユーザにとって価値ある本処理（上流返却）は既に成功している。両者のどちらを優先するかが SPEC に無い。

## 3. 期待される情報

SPEC-LGX-003（必要なら SPEC-LGX-007 REQ.06 と協調）に以下を追加:

- context_log 書込失敗時、compile_context は (a) 本処理結果を返したうえで成功扱いとし記録欠落を warn ログに残す、(b) 記録失敗を理由にエラー返却する、のいずれを正準とするか。
- 本処理の返却（stdout / MCP content）と context_log 書込が同一トランザクションか別トランザクションか（atomicity の有無）。
- busy_timeout 超過時のリトライ方針（回数・間隔）と最終的な終端状態。
- MCP-INV-4「全呼出し記録」の「全」の例外条件（記録失敗を許容するなら、その記録漏れの検出可能性）。

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- UC-LGX-002 / UC-LGX-004: 異常系（DB ロック・ディスクフル）のフロー事後条件が書けない。
- 下流の SEQA/SEQD/DD: context_log 書込のエラーパスとトランザクション境界が設計できない（Rust 側の `Result` 型・SQLite トランザクション設計に直結）。
- TS/TC: 「DB ロック時に compile_context がどう振る舞うか」のテスト期待値が確定しない。
- 関連: GAP-LGX-042（DB 完全不在時の記録不能ケース。本 GAP は DB 存在下での書込失敗を扱う点で区別）。

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-003 v0.7.0（人間承認 2026-06-10）: REQ.19 新設 — 本処理優先（別 Tx・stdout 返却 + exit 0・stderr Warning）、MCP-INV-4 をベストエフォートに確定。ADR-LGX-004 に可用性 > 監査完全性の判断を記録。

監査ログ完全性（MCP-INV-4）と可用性のトレードオフを伴うため、判断時は ADR 起票候補:

# GAP-LGX-168: MCP サーバ起動・CLI バイナリ解決失敗時の挙動が未定義

**親 TP**: TP-LGX-009
**観点出典**: TP-LGX-009 §2.6 観点 32（ライフサイクル）
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08
**敵対的精査（2026-06-09）**: minor (verification: low-value, 人間判断で drop 可)。REQ.07 の `exit -1` フォールバックが「プロセス起動不能」（バイナリ解決失敗からの spawn 失敗を含む）を `Rust CLI failed (exit -1):` 形式で報告すると規定済みであり、呼出し時経路は実質回答済み。残る「起動時 fail-fast でブートを止めるか」の選択は thin forwarder の DD レベル実装事項。

## 1. 観点

MCP サーバ起動時、または初回ツール呼出し時に Rust CLI バイナリが解決できない（`TRACEABILITY_ENGINE_BIN` 不正パス・`--engine-binary` 不正・既定名が PATH 上に無い）、作業ディレクトリが存在しない等の初期化失敗時に、MCP サーバがどう振る舞うか（起動拒否 / 各ツール呼出しでエラー返却 / エラーメッセージ形式）が未定義。

## 2. 現状の SPEC

SPEC-LGX-009 §3 REQ.08 は **「起動設定（CLI バイナリパス, 作業ディレクトリ）は環境変数または起動引数で渡す」**、REQ.12 は OS 別バイナリ名に触れ、LGX-COMPAT-001 §5 補足は **バイナリ解決順（env → フラグ → 既定名）** を規定するが、**解決に失敗した場合の挙動**（サーバ起動を fail-fast で止めるか、ツール呼出し時に毎回エラーを返すか、そのエラーが REQ.07 の `Rust CLI failed (exit N):` 形式か別形式か）は未定義。REQ.07 の `exit -1` フォールバックは「起動不能」を含むが、それが起動時 fail-fast と呼出し時返却のどちらかは不明。core-perspectives §ライフサイクル「初期化失敗時のエラー詳細」に対応する観点が空白。

## 3. 期待される情報

SPEC に追加されるべき記述:

- バイナリ解決失敗時に MCP サーバ起動を fail-fast で停止するか、起動後に各呼出しでエラーを返すかの確定
- 当該エラーの応答形式（REQ.07 の `exit -1` フォールバックと同一か、起動前なので別経路か — GAP-LGX-164 とも関連）
- 作業ディレクトリ・`--project-root` 不正時の扱い

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- 全 UC（MCP 経由）: デプロイ/設定ミス時の例外フローが未定義
- 下流の SEQD/DD/TS: 起動シーケンスとバイナリ解決失敗ハンドリングが書けない
- REQ.08 の「検証方法: 起動シナリオテスト」が失敗ケースを含められない
- 関連: GAP-LGX-167（リソース枯渇起因の spawn 失敗）, GAP-LGX-164（起動前エラー経路）

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-009 v0.6.0（人間裁定 fix・承認 2026-06-10）: REQ.08 に初期化失敗時挙動を確定 — 起動時 fail-fast せず、各ツール呼出し時に REQ.07 の `Rust CLI failed (exit -1):` 形式の isError 応答（v3 実質挙動の正準化）。project-root 不正も同経路。

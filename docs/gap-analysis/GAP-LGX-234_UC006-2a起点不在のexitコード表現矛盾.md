# GAP-LGX-234: UC-LGX-006 代替フロー 2a「ERROR を報告する」と SPEC-005.REQ.05「空結果・非エラー」が矛盾

**親 TP**: TP-LGX-016
**観点**: 2.2 AF2（各代替フローも基本フローと同じ事後条件に収束するか）/ 2.6 R2（終了コード契約の観察可能性）
**ステータス**: closed (2026-06-13)
**起票日**: 2026-06-13
**分類（暫定）**: GENUINE 寄り（SPEC との表現矛盾があり、exit コード規約が UC フローから確定できない）

## 1. 観点

UC レベル観点「各代替フローも基本フローと同じ事後条件に収束するか、または別途定義された事後条件を持つか」を UC-LGX-006 代替フロー 2a にぶつけた。

## 2. 現状の UC / SPEC

- UC-LGX-006 代替フロー 2a: 「起点ノードが graph.toml に存在しない場合、ERROR を報告する」
  - 「ERROR を報告する」は exit 1 系（実行時エラー）を示唆する表現。
  - 終了動作（報告後に即時終了するか、空結果を出力して終了するか）が未定義。
  - exit コード（exit 0 か exit 1 か）が明示されていない。
- SPEC-LGX-005.REQ.05: 「起点ノードがグラフに存在しない場合、**空の結果を返す（エラーではない）**」
  - exit 0 を意味する（TP-LGX-005 観点 6 で GREEN 確定、OBS.05/USE.04 と整合）。
- 「ERROR を報告する（2a）」と「空の結果を返す・エラーではない（REQ.05）」は正反対の意味合いを持つ表現であり、UC と SPEC で矛盾が生じている。

この矛盾は SPEC が答えているため下流実装は SPEC.REQ.05 に従えば正しいが、UC フロー記述として不整合が残る。また、「ERROR」という語が「error finding を CheckReport に含める（exit 1）」を意図しているのか「エラーメッセージをユーザに通知する（exit 0）」を意図しているのかが読み取れない。

## 3. 推奨対応（人間裁定）

以下のいずれか:

- **(A) UC を SPEC に整合させる**: 代替フロー 2a を「起点ノードが graph.toml に存在しない場合、空の走査結果を返して exit 0 で終了する（SPEC-LGX-005.REQ.05）」に修正。「ERROR を報告する」という表現を除去し、SPEC との整合を明示する。
- **(B) SPEC を UC に整合させる（人間承認必要）**: 起点不在を「エラー」扱いにする設計判断を行い、SPEC-LGX-005.REQ.05 を「ERROR を報告して exit 1 で終了する」に変更する。ただし LGX-COMPAT-001 §4 #11（impact の終了コード）との整合確認が必要。
- **(C) UC に注記追加**: 2a に「（SPEC-LGX-005.REQ.05 より exit 0 で終了する。「ERROR」はユーザへの情報提供メッセージを指す）」と注記を加える。

GENUINE 寄り：SPEC.REQ.05 と UC の表現が直接矛盾しており、下流 RBA/DD がどちらを参照するかによって実装が変わりうる。

## 4. 影響範囲

- close されないと TP-LGX-016 が green にならず、UC-006 起点の下流（RBA 以降）に進めない。
- exit コード（exit 0 vs exit 1）の矛盾は下流 TS/TC の期待値に直接影響し、RBA 設計の責務定義にも波及する。
- GAP-LGX-232（BF3: exit コード不記載）と同根の問題族であり、一括裁定を推奨。

## 5. 解消（2026-06-13）

敵対的精査裁定: **GENUINE**（実 SPEC 照合で確定）。UC 修正で解消（A1: UC-LGX-006 2a を「空結果 + exit 0」へ修正（SPEC-LGX-005.REQ.05 準拠））。人間承認 2026-06-13（A2/C2/C3 は AskUserQuestion 裁定、推奨案採用）。詳細: docs/spec-change-proposals/2026-06-13_uc-gap-adversarial-review.md §A。

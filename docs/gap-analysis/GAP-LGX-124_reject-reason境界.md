# GAP-LGX-124: reject の --reason の空文字列・長さ境界方針が未定義

**親 TP**: TP-LGX-007
**観点出典**: TP-LGX-007 §2.1 観点 B5
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08
**重大度（2026-06-09 敵対的精査パス）**: minor（verification: low-value, 人間判断で drop 可）。--reason は必須引数として既に COMPAT §4 で保証されており、空文字許容/長さ方針の欠落は監査証跡の品質方針という補助論点。主フロー成立を妨げない。

## 1. 観点

`reject <id> --reason <text>` の `--reason`（必須）について、空文字列を許容するか、最大長があるかが定義されていない。

## 2. 現状の SPEC / UC

SPEC-LGX-007 REQ.05 は `reject ... --reason <text>` を必須引数として記述。LGX-COMPAT-001 §4 も `--reason <S>`（必須）とのみ規定。空文字列 `--reason ""` が「指定あり」とみなされるか、長さ上限があるかは未定義。reject_reason は REQ.09 で proposals テーブルに格納される監査証跡だが、その品質保証（空理由の拒否等）に触れていない。

## 3. 期待される情報

SPEC-LGX-007 REQ.05 に以下を追加すべき:

- `--reason` の空文字列を許容するか（監査証跡として空理由を拒否すべきか）
- 拒否する場合の終了コード（使用法誤り exit 2 か意味的不正 exit 1 か）
- 最大長の有無

## 4. 影響範囲

この GAP がクローズされないと以下に影響:

- UC-LGX-008: reject フローの入力検証ステップが具体化できない
- 下流の TS / TC: --reason 境界値テストの期待値が決まらない
- 監査証跡（reject_reason）の品質保証に関連（監査ログ順序の旧 GAP-LGX-137 は SPEC-LGX-003 主導の OUT_OF_SCOPE として敵対的精査で削除済み）

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-007 v0.5.0（人間裁定 fix・承認 2026-06-10）: REQ.05 に --reason 境界を確定 — 空/空白のみは拒否 exit 1（監査証跡品質: 空理由を指定ありとみなさない）、最大長なし・無加工保存。

# GAP-LGX-103: モデル出力 shape の動的確定における異常 shape の扱いが未定義

**親 TP**: TP-LGX-006
**観点出典**: TP-LGX-006 §2.1 観点 B-03
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08
**重要度**: minor (verification: low-value, 人間判断で drop 可) — 敵対的精査パス 2026-06-09: ONNX ランタイム境界の shape 検証は DD/実装層。誤モデル配置はオペレーション事項。SPEC レベル決定性は低い。WEAK_OR_PADDED として維持。

## 1. 観点

REQ.01 は出力次元をモデルの出力 shape から動的に確定するとしているが、ONNX モデルが (a) 0 次元、(b) 想定外の 3 次元以上の shape、(c) shape 取得自体に失敗、(d) mean pooling に必要な軸構造を持たない出力を返した場合の挙動が未定義。互換性のないモデルを誤って配置した場合の検出手段がない。

## 2. 現状の SPEC / UC

SPEC-LGX-006 §3 REQ.01 は「出力次元はモデルの出力 shape から動的に確定するため、384 次元以外にも対応する」と記述するが、**正常な 2 次元 [batch, hidden] 出力以外**（異常 shape / 取得失敗 / mean pooling 非対応）に対する検証・エラー方針を記述していない。REQ.01 検証方法は「モデル読み込みテスト」とあるが、互換性のないモデルの拒否仕様は不在。

## 3. 期待される情報

- モデル読込時の出力 shape 妥当性検証（期待する rank / 軸構造）
- 異常 shape 検出時の挙動: Error + 終了コード（embed 自体が起動不能か、当該推論のみ失敗か）
- shape 取得失敗・mean pooling 非適用モデルの拒否メッセージ（modeldir 誤配置の示唆）
- SCORE-INV-2（モデルバージョン一致）との関係（互換性のないモデルが model_version 不一致として弾かれるか、shape 検証で弾かれるか）

## 4. 影響範囲

- UC-LGX-007: モデル切替時の初期化失敗パス（例外フロー）が定義できない
- 下流 DD / TS: ONNX ランタイム境界の shape 検証ロジックとテスト
- GAP-LGX-106（モデル不在/読込失敗）と隣接（読込成功だが出力が異常なケース）

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-006 v0.7.0（人間裁定 fix・承認 2026-06-10）: REQ.01 にモデル読込時の出力 shape 検証（mean pooling 可能な軸構造・正の hidden 次元）を確定、非適合は embed 起動不能 Error exit 1 + 誤配置示唆診断。読込時検証と SCORE-INV-2（版照合）は別経路と明記。

## 6. 関連 ADR

該当なし（実装方針判断）。

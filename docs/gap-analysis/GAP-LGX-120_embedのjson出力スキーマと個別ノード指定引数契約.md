# GAP-LGX-120: embed の --json 出力スキーマと個別ノード指定オプションの引数契約が未定義

**親 TP**: TP-LGX-006
**観点出典**: TP-LGX-006 §2.8 観点 F-02（＋ 観点 F-03 を統合）
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

(1) グローバル `--json` は全コマンドで受理される（LGX-COMPAT-001 §3）が、`embed` の `--json` 出力スキーマ（生成件数 / スキップ件数 / 失敗件数 / ノード ID 別結果）が未定義。SPEC-LGX-010 は読取系コマンドの `--json` スキーマを各 REQ で詳細に定めているが、書込系 `embed` の対応物が無い。
(2) REQ.02 は「個別ノード ID 指定オプションも提供」とするが、その**フラグ名・引数形式**（`--node <ID>` か位置引数か、複数指定可否）が SPEC-006 にも LGX-COMPAT-001 §4 #4（`embed [--all]` のみ列挙）にも無い。互換契約の embed 行に個別指定オプションが載っていないため、これを新設するのか v3 実測に既存なのかも不明。

## 2. 現状の SPEC / UC

SPEC-LGX-006 §3 REQ.02 は「個別ノード ID 指定オプションも提供」とするのみで引数形式を規定していない。LGX-COMPAT-001 §4 #4 は `embed | — | [--all]` とフラグを `[--all]` のみ列挙し、個別ノード指定オプションも `--force`（REQ.10 が言及）も記載していない。`--json` は §3 グローバルで受理されるが embed の出力スキーマは未定義。embed コマンドの引数契約と出力契約が SPEC-006 / 互換リファレンス間で完結していない。

## 3. 期待される情報

- `embed --json` の出力スキーマ（例: `{generated, skipped, failed, errors: [{node_id, reason}]}`）
- 個別ノード指定オプションのフラグ名・形式・複数指定可否（LGX-COMPAT-001 §4 #4 への追記要否。互換契約の境界 API 凍結対象、ハードルール 7）
- `--force`（REQ.10）が互換契約の embed フラグ一覧に載るべきかの整合
- これらが v3 実測に既存か、legixy 新設か（既存なら互換維持、新設なら互換チェックリスト §7 への影響評価）

## 4. 影響範囲

- LGX-COMPAT-001 §4 #4 / §7 チェックリスト: embed の凍結引数契約の完全性（境界 API、ハードルール 7）
- MCP 非公開（embed は Admin Surface）だが CLI スクリプト連携の `--json` 消費者
- GAP-LGX-109（終了コード）/ GAP-LGX-111（個別 ID × --all × --force 組合せ）と同根の引数契約
- 下流 DD / TS: embed の引数パーサと --json 出力スキーマテスト

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-006 v0.6.0（人間承認 2026-06-10・凍結契約への加算的拡張）: REQ.02 に --node <ID>（複数可・--all 排他）/--force/--json スキーマ {generated, skipped, failed, errors[]} を確定。v3 実測でフラグ不在（main.rs:91-94、--all のみ）を確認し純加算＝後方互換。ADR-LGX-002 + LGX-COMPAT-001 v1.1.0 §4 #4/§7 追記。MCP 非公開維持。

## 6. 関連 ADR

個別ノード指定オプション・`--force` が LGX-COMPAT-001 の凍結境界契約に影響する場合、互換性判断として ADR 候補（ハードルール 7、境界 API 凍結）。

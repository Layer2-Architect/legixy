# GAP-LGX-243: UC-LGX-007 embed の部分失敗継続挙動が UC フローに未反映

**親 TP**: TP-LGX-017
**観点**: §2.3 EF1
**ステータス**: closed (2026-06-13)
**起票日**: 2026-06-13
**分類（暫定）**: WEAK

## 1. 観点

UC-LGX-007 の embed 基本フロー Step3 は各ノードの処理ステップ（a: ハッシュ計算 → b: スキップ判定 → c: 前処理 → d: 推論 → e: 格納）を列挙するが、「1 ノードの生成・格納に失敗した場合、当該トランザクションのみ rollback して後続ノードの処理を継続する」という部分失敗継続挙動（SPEC-LGX-006.REQ.08）が UC フローに現れない。成功件数・スキップ件数・失敗件数の報告という事後観察可能状態も UC に記述されていない。

## 2. 現状の UC / SPEC

**UC-LGX-007 embed 基本フロー Step3（現行）:**
> 3. 各ノードについて:
>    a. ファイル内容のハッシュ（SHA-256）を計算する
>    b. engine.db の既存ハッシュと比較し、変更がなければスキップする（SCORE-INV-1）
>    c. 変更がある場合、前処理（EmbeddingPreprocessor）を適用する
>    d. ONNX モデルで embedding を生成する（トークン化 → 推論 → Mean Pooling → L2 正規化）
>    e. engine.db の embeddings テーブルに格納する

ノード単位の失敗パス（例: d の推論失敗・e の格納失敗）と「後続継続」の記述が無い。

**SPEC-LGX-006.REQ.08 の規定（既存）:**
> embedding の生成・格納のトランザクション粒度は**ノード/サブノード単位の 1 トランザクション**とする。1 ノード/サブノードの生成・格納が失敗した場合、当該トランザクションのみ rollback して Error に計上し、後続ノードの処理を継続する

SPEC は部分失敗継続を明確に規定している。UC フロー記述への反映が欠落しているが、SPEC が既に答えている。

## 3. 推奨対応（人間裁定）

**(A) UC へ追記案:**
Step3 に例外パスを追加する:
> 3. 各ノードについて（1 ノードの失敗は当該ノードのみに留め後続処理を継続する）:
>    a〜e. （現行と同様）
>    （失敗時）当該ノードを Error 件数に計上し、次のノードへ進む

または事後条件に「生成件数・スキップ件数・失敗件数が報告される」を追加する。

**(B) drop（委譲容認）案:**
SPEC-LGX-006.REQ.08 が「ノード単位 Tx・後続継続」を明示規定しており、UC フローへの再記述は冗長。「各ノードについて」という繰り返し記述で部分継続が暗示されているとみなし、UC フローへの追記は不要と裁定する。

## 4. 影響範囲

- UC-LGX-007 embed 基本フロー Step3 の失敗パス記述（案 A の場合）
- または事後条件への件数報告追記
- 後続 RBA での部分失敗処理設計根拠
- TP-LGX-011（UC-001）GAP-LGX-190（部分失敗継続フロー未反映）と同種のパターン

## 5. 解消（2026-06-13）

敵対的精査裁定: **方針B（委譲容認）**。本観点は親 SPEC（対応 TP[SPEC] green 確定済）が回答しており、UC フローへの明示列挙は任意。GREEN-by-delegation として close。詳細: docs/spec-change-proposals/2026-06-13_uc-gap-adversarial-review.md §F。

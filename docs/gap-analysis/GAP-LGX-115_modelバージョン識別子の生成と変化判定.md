# GAP-LGX-115: model_version 識別子の生成方式と「変化した」の判定基準が未定義

**親 TP**: TP-LGX-006
**観点出典**: TP-LGX-006 §2.5b 観点 V-01
**ステータス**: closed (2026-06-10)
**起票日**: 2026-06-08

## 1. 観点

REQ.10 は「`model_version` が変化した場合、全 embedding を再生成する」とし、REQ.03 は model_version を必須情報に挙げるが、**model_version 識別子が何から生成されるか**（モデルファイルのハッシュ / 設定ファイルのモデル名文字列 / モデル metadata のバージョン）と、**「変化した」をどう判定するか**が未定義。例えば設定のモデル名は同一だがモデルファイルの中身が差し替わった場合に再生成がトリガされるか、逆にファイルは同一だが設定名表記が変わった場合に偽の再生成が起きるかが決まらない。SCORE-INV-2（モデルバージョン一致 = 有効）の判定の核心。

## 2. 現状の SPEC / UC

SPEC-LGX-006 §3 REQ.03 は「使用モデルのバージョン識別子」を格納情報に挙げ、REQ.10 は「model_version が変化した場合 全再生成」とするが、**識別子の生成方式・等価判定の基準**を規定していない。§4 SCORE-INV-2（現在のモデルバージョンと一致するスコアのみ有効）の判定がこの定義に依存する。REQ.01 は代替モデル（768 次元の e5/mpnet 等）への切替を許容しているため、model_version はモデル名だけでなく次元・前処理の差も識別できる必要がある可能性がある。

## 3. 期待される情報

- model_version の生成元: モデルファイルのハッシュ / 設定のモデル名 / metadata バージョン / それらの複合
- 「変化した」の等価判定基準（何が変わると再生成トリガになるか）
- 設定名は同一だがモデルファイルが差し替わった場合の検出可否
- 前処理（e5 系の query:/passage: プレフィックス、REQ.01）の有無が model_version に反映されるか

## 4. 影響範囲

- SCORE-INV-2: モデル一致判定の決定性
- REQ.10（モデル更新時の再計算）のトリガ条件
- UC-LGX-007 / UC-LGX-009（移行）: モデル切替・v0.1.0 旧モデルからの移行（GAP-LGX-116 と連携）
- 下流 DD / TS: model_version 生成ロジックとモデル切替テスト（REQ.10 検証方法）

## 5. 解決経緯（クローズ時に記入）

SPEC-LGX-006 v0.6.0（人間承認 2026-06-10）: REQ.10 に model_version 複合キー（モデル名+ONNX 内容ハッシュ+前処理プロファイル+出力次元）と完全一致判定を確定。hex 桁数等は DD 委譲。ADR-LGX-003。

## 6. 関連 ADR

model_version の生成方式（再現性 vs 検出力）は ADR 候補。GAP-LGX-116（旧 embedding 移行）と連携。

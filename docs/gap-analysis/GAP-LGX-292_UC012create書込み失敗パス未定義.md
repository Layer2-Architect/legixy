# GAP-LGX-292: UC-LGX-012 が snapshot create の書込み失敗パスを定義していない

**親 TP**: TP-LGX-022
**観点**: 2.3 EF2（各ステップでの失敗パスが定義されているか / 部分成功時の扱い）
**ステータス**: closed (2026-06-13)
**起票日**: 2026-06-13
**分類（暫定）**: WEAK（親 SPEC の実行時失敗 = exit 1 + 単一 Tx atomicity へ委譲可。書込み系特有の失敗パスは UC 完全性として明示価値あり）

## 1. 観点

UC レベル観点「各ステップでの失敗パスが定義されているか」「部分成功時の扱い」を UC-LGX-012 基本フロー Step2（embeddings ストアの現行全行を単一トランザクションでスナップショット領域へ複製）にぶつけた。create は本 UC で唯一の書込み系処理（SPEC-LGX-010.REQ.07: engine.db 初期化 + スナップショット領域書込み）であり、書込み失敗時の状態が事後条件を左右する。

## 2. 現状の UC / SPEC

- UC-LGX-012 の create 代替フローは `2a`（空ストア = WARNING + exit 0 + 非永続）のみを定義する。
- 空ストア以外の create 失敗 — 複製トランザクションの失敗、engine.db 初期化失敗（ディスクフル・権限エラー）、書込み I/O 失敗 — の挙動（exit code、部分スナップショットの残存有無）が UC のフロー記述に列挙されていない。
- 一方、SPEC レベルでは:
  - REQ.01 が「受理済みフラグ・引数の値の意味的不正および**実行時失敗** = 実行エラー（exit 1）」を規定。
  - REQ.02 が「複製は**単一トランザクションで行う**」と atomicity を規定（→ 失敗時は部分スナップショットが残らない）。
  - REQ.07 が「`snapshot create` のみ書込み系として engine.db の初期化（不在時の新規作成 + スキーマ作成）とスナップショット領域への書込みを行う」と書込み範囲を規定。
- すなわち「create 失敗 → exit 1、単一 Tx により部分複製なし」は SPEC から**導出可能**だが、UC-012 のフロー記述（書込み失敗パス・部分成功の事後条件）には明示されていない。

## 3. 推奨対応（人間裁定）

以下のいずれか:

- **(A) UC へ追記**: 例外フロー `2-E`「複製トランザクションまたは engine.db 初期化・書込みが失敗した場合: ERROR(stderr) + exit 1。複製は単一トランザクションのため部分スナップショットは永続化されない（SPEC-LGX-010.REQ.01 / REQ.02 / REQ.07）」を追加。→ 唯一の書込み系コマンドの失敗時事後条件（atomicity）が観察可能化。
- **(B) drop（委譲容認）**: create の実行時失敗の終了コードは REQ.01 へ、部分複製非残存は REQ.02 の単一 Tx 規定へ委譲する設計と認め、UC は変更しない。→ UC は変更しない。

WEAK 候補。振る舞いは SPEC から導出可能であり、UC フロー記述の粒度方針に依存する（UC-LGX-001 の GAP-LGX-189 / 190 と同種の「親 SPEC 確定済・UC 記述への明示反映が任意か必須か」の裁定）。ただし create は本 UC で唯一の破壊的・書込み操作であり、失敗時 atomicity の明示は他の読取系 case より価値が高いため、一括 drop ではなく個別判断が望ましい。フロー妥当性は人間レビュー領域。

## 4. 影響範囲

- close されないと TP-LGX-022 が green にならず、UC-012 起点の下流（RBA 以降）に進めない。
- 振る舞い自体は SPEC-LGX-010.REQ.01 / REQ.02 / REQ.07 で確定済のため、下流実装の正しさには影響しない（記述完全性の問題）。GAP-LGX-291（delete 失敗パス）と同じ delete/create 失敗パスの完全性問題であり、一括裁定が可能。
